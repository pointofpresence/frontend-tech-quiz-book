# Какие методы уменьшения объема Docker-образа ты знаешь?

Категория: Процессы

### Краткий ответ

**Методы уменьшения объема Docker-образа**

- минимизация базового образа (alpine, scratch)
- multi-stage build для разделения этапов сборки
- удаление ненужных файлов/кеша в образе
- объединение команд RUN для уменьшения слоев
- использование .dockerignore для исключения файлов
- оптимизация зависимостей (минимальный набор)
- сжатие образа (docker-slim, built-in squash)

### Подробный ответ
#### Основной ответ
Уменьшение объема **Docker-образа** — важная задача для ускорения доставки приложений, снижения использования дискового пространства и повышения безопасности. Существует несколько проверенных методов оптимизации размера образа, которые снижают избыточность и минимизируют слои.

#### Ключевые моменты
- **Использование легковесных базовых образов**: вместо стандартных больших (например, ubuntu, debian) можно применять Alpine Linux (~5 МБ), BusyBox или slim-версии официальных образов. Это существенно снижает стартовый объем.
- **Многоступенчатая сборка (multi-stage build)**: позволяет компилировать или собирать артефакты в одном образе, а в финальный включать только минимальный runtime с необходимыми файлами, исключая инструменты сборки и зависимости разработки, что резко уменьшает размер итогового образа.
- **Минимизация слоев и оптимизация Dockerfile**: сокращение количества RUN-команд и объединение их в одну через && предотвращает создание лишних слоев. Очистка кэшей менеджеров пакетов (apt-get clean, rm -rf /var/lib/apt/lists/*) в рамках той же RUN-команды тоже помогает избежать накопления мусора.
- **Удаление ненужных файлов и зависимостей**: удаление временных файлов, документации, тестов, кэшированных пакетов и dev-зависимостей в конечном образе.
- **Использование .dockerignore**: игнорирование лишних файлов и папок при контексте сборки уменьшает размер передаваемых данных и снизит вероятность попадания ненужного в образ.
- **Сжатие образов и оптимизация слоев**: инструменты типа Docker Slim или BuildKit могут автоматически минимизировать образ и его слои, удаляя неиспользуемые части.

#### Практический контекст
В React-проектах с Node.js вместо полного node-образа часто применяют multi-stage сборку с Alpine для продукта и сборки. В Go-проектах итоговый образ может быть основан на scratch (нулевой образ), поскольку бинарник содержит всё нужное. На production-средах это снижает время деплоя и ускоряет CI/CD pipeline. Это важно для поддержания 99.9% uptime и быстрой масштабируемости.
