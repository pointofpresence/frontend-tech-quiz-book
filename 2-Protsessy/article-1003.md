# В чем отличие между microtask queue и task queue?

Категория: Процессы

### Краткий ответ

**Отличие между microtask queue и task queue**  
- контекст: **асинхронность в Event Loop** JS  
- **microtask queue**: задачи с более высоким приоритетом, выполняются сразу после текущего стека и до рендеринга  
- содержит: Promise callbacks, MutationObservers  
- **task queue**: задачи с меньшим приоритетом, обрабатываются после рендеринга  
- содержит: setTimeout, setInterval, I/O события  
- порядок: microtasks ⬆️ после синхронного кода, затем задачи из task queue ⬇️  
- практическое значение: microtasks позволяют быстрее обрабатывать важные async операции, обеспечивая более отзывчивый UI и правильный порядок выполнения  
---  
Такое понимание показывает глубокое владение Event Loop и оптимизацией асинхронного JS-кода.

### Подробный ответ
#### Основной ответ
**Microtask queue** и **task queue** — это два отдельных механизма планирования выполнения асинхронных операций в JavaScript event loop. Основное отличие в том, что **microtasks** имеют более высокий приоритет и выполняются сразу после текущего выполняющегося скрипта и перед обработкой задач из task queue (микрозадачи обрабатываются до переключения контекста). Task queue используется для более "общих" асинхронных событий, таких как setTimeout, setInterval, или I/O операции.

#### Ключевые моменты
- **Microtask queue** включает промисы (Promise.then/catch/finally), process.nextTick (Node.js), MutationObserver. Они выполняются сразу после завершения текущего синхронного кода и перед тем, как event loop возьмёт следующую задачу из **task queue**.
- **Task queue** содержит коллбэки setTimeout, setInterval, события DOM (клики, таймеры и пр.). После выполнения всех microtasks event loop берёт следующую задачу из task queue и выполняет её.
- Важный нюанс: если в microtask ставится новая microtask, они будут выполняться непрерывно до тех пор, пока очередь microtasks не опустеет, что может привести к блокировке выполнения следующих задач из task queue.

#### Практический контекст
Типично в веб-разработке вы видите различие так: при использовании `Promise.resolve().then(...)` ваш коллбэк попадёт в microtask queue и выполнится быстрее, чем, например, код в `setTimeout(..., 0)`, который попадает в task queue. Это помогает создавать более отзывчивые интерфейсы и правильно управлять асинхронностью в React 18+, Node.js (event loop там аналогичен) и современных браузерах.
