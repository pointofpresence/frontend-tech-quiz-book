# Как исправить код с CountDownLatch и не-атомарным счётчиком (без AtomicInteger)?

Категория: Процессы

### Краткий ответ

**Как исправить CountDownLatch с неатомарным счётчиком?**

- проблема: счётчик неатомарен → **состояния гонки**
- решение 1: заменить счётчик на **AtomicInteger** (если запрещено — нельзя)
- решение 2: **синхронизировать** операции с обычным int
- держать счётчик и операции **под lock (synchronized)**
- все уменьшения и проверки в **одном синхронизированном блоке**
- CountDownLatch уже решает синхронизацию вызовами await/countDown
- если своя логика — использовать **ReentrantLock** + condition
- альтернативно: использовать **Semaphore** или **Phaser** для сложных сценариев
- итог: без AtomicInteger — строго **синхронизировать доступ** к счётчику, чтобы избежать гонок и неконсистентности

---

**Детально:**

CountDownLatch internally handles synchronization for its own counter, но если вместо `countDown()` и `await()` используется внешний неатомарный int-счётчик отдельно (например, свой manual counter), то при многопоточном уменьшении без атомарности или синхронизации возникает race condition — счётчик может быть уменьшен одновременно и привести к неправильному состоянию или deadlock.

Правильный способ — либо заменить счетчик на AtomicInteger, либо обернуть все операции над обычным int в synchronized блок (монитор), обеспечив таким образом **атомарность и видимость изменений** между потоками.

Пример исправления без AtomicInteger:

```java
private final Object lock = new Object();
private int counter;

public void decrement() {
  synchronized (lock) {
    counter--;
    if (counter == 0) {
      lock.notifyAll();
    }
  }
}

public void await() throws InterruptedException {
  synchronized (lock) {
    while (counter > 0) {
      lock.wait();
    }
  }
}
```

Этот подход гарантирует корректную работу с неатомарным счётчиком, но требует тщательно контролировать блокировку, чтобы избежать deadlock или пропуска сигналов.

---

**Резюме:** без AtomicInteger критично **обеспечить синхронизацию всех операций с счётчиком**, чтобы CountDownLatch с неатомарным счётчиком работал корректно и потокобезопасно.

### Подробный ответ
#### Основной ответ
Если у вас есть код с **CountDownLatch** и неатомарным счётчиком (например, обычный `int`), и вы не можете использовать `AtomicInteger`, то для исправления проблемы с потокобезопасностью нужно явно синхронизировать доступ к счётчику. CountDownLatch сам по себе потокобезопасен и корректно работает для ожидания завершения, но инкремент/декремент счётчика должны быть защищены, чтобы избежать условий гонки.

Самый простой и распространенный способ — использовать ключевое слово **synchronized** для методов, которые изменяют счётчик, либо синхронизировать участок кода, где это происходит. Это гарантирует, что только один поток одновременно изменит значение счётчика, избегая некорректных состояний.

#### Ключевые моменты
- **CountDownLatch** отвечает за ожидание завершения заданного количества событий, но не управляет атомарностью внешних переменных.
- Отсутствие `AtomicInteger` требует использования блокировок (например, `synchronized`), чтобы сериализовать доступ к счётчику.
- Использование `volatile` здесь не эффективно, поскольку операция инкремента не атомарна и потребует блокировки.
- Альтернативный подход — использовать другие синхронизированные структуры (например, `Lock` из `java.util.concurrent.locks`), но если задача — исправить быстро без atomic, то `synchronized` — самый прямой способ.

#### Практический контекст
В реальных проектах часто видишь, как для защиты простых счётчиков применяют `synchronized(this)` или выделенный монитор для метода обновления счётчика, при этом CountDownLatch остаётся для ожидания завершения потоков. Такой подход прост и надежен, например, в Java 8+.

Пример:

```java
private int counter = 0;
private final Object lock = new Object();

public void increment() {
    synchronized(lock) {
        counter++;
    }
}
```

И при этом `CountDownLatch` используется без изменений для ожидания завершения. 

Таким образом, сохраняется корректность и атомарность обновления счётчика без использования `AtomicInteger`.
