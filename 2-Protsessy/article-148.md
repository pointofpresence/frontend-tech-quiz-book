# Как можно объединить обновления нескольких стейтов в один ре-рендер в React 17?

Категория: Процессы

### Краткий ответ

**Как объединить обновления нескольких стейтов в один ре-рендер в React 17?**

- управляет асинхронным **бэтчингом** обновлений
- в React 17 бэтчинг есть только внутри **обработчиков событий React**
- **обновления из callback/setTimeout** рендерят отдельно
- для объединения вне React-событий используют **unstable_batchedUpdates** из 'react-dom'
- этот API вручную объединяет несколько setState в один ре-рендер
- **ограничение:** нестабильный API, осторожно в проде
- Альтернатива — объединять стейки логически в один объект стейта (useReducer)

---
##### Подробный ответ для интервью

В React 17 оптимальное объединение (бэтчинг) нескольких обновлений состояния происходит автоматически **только внутри синтетических обработчиков событий React** — все setState вызовы внутри такого обработчика объединяются и вызывают один ре-рендер. Однако, если обновления запускаются вне этого контекста (например, в setTimeout, Promise, нативных событиях), React 17 по умолчанию выполняет отдельные ре-рендеры для каждого setState.

Чтобы объединить обновления в таких случаях, React предоставляет низкоуровневый API — функцию **unstable_batchedUpdates** из пакета 'react-dom'. Она позволяет вручную обернуть несколько вызовов setState и гарантирует один ре-рендер после их выполнения, снижая лишние перерисовки. Пример:

```js
import { unstable_batchedUpdates } from 'react-dom';

unstable_batchedUpdates(() => {
  setState1(val1);
  setState2(val2);
});
```

Эта функция нестабильна и может измениться в будущих версиях, но в React 17 считается основным способом оптимизации вне React-обработчиков.

Кроме того, если часто возникает необходимость обновлять несколько связанных стейтов одновременно, лучше объединить их в один объект состояния и управлять через `useReducer` — это гарантирует атомарные обновления и один ре-рендер без дополнительных ухищрений.

**Итого:** в React 17 автоматический бэтчинг в пределах React-событий, для остальных случаев используется `unstable_batchedUpdates`, либо логическая группировка состояния через `useReducer` для минимизации ререндеров.

### Подробный ответ
#### Основной ответ
В React 17 и ниже обновления стейта в обработчиках событий являются асинхронными, но **по умолчанию каждое setState вызывает отдельный ре-рендер**. Чтобы объединить несколько обновлений стейта в один ре-рендер, нужно использовать **функциональный setState** или методы группировки обновлений вручную, так как автоматическое батчинг обновлений работает преимущественно внутри React событий, но не всегда для асинхронных коллбеков.

#### Ключевые моменты
- В React 17 **батчинг обновлений есть только внутри синхронных обработчиков React**, то есть несколько вызовов setState в обработчике события объединяются в один ре-рендер.
- **Если setState вызывается в асинхронном коде** (например, в setTimeout, промисах, подписках), то батчинг не происходит и каждый setState триггерит отдельный ре-рендер.
- Для объединения обновлений вне React событий можно использовать **unstable_batchedUpdates из ReactDOM**, который позволяет вручную группировать несколько setState в один ре-рендер.
- Кроме того, можно комбинировать весь необходимый стейт в один объект и обновлять одним вызовом, что значительно снижает количество ре-рендеров.
  
#### Практический контекст
Типичный кейс — когда обновления приходят из разных источников (события, таймеры, WebSocket) и нужно минимизировать количество рендеров. В таком случае в React 17 используют **ReactDOM.unstable_batchedUpdates** или объединяют состояние одним вызовом setState, чтобы добиться лучшей производительности и избежать лишних перерисовок. Начиная с React 18 автоматический батчинг распространяется и на асинхронные операции, но в 17 приходится применять эти вручную.
