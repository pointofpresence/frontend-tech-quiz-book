# Как организовать обработку исключений (Exception Handling) в микросервисах?

Категория: Процессы

### Краткий ответ

**Обработка исключений в микросервисах**

- микросервисы — независимые сервисы с изолированной логикой  
- централизованный логгинг ошибок (например, через ELK, Prometheus)  
- локальная обработка с собственными retry и fallback стратегиями  
- используем глобальный middleware/фильтр для консистентного response  
- трассировка ошибок через correlation ID  
- стандартизированные форматы ошибок (JSON API errors или RFC 7807)  
- важен graceful degradation и информативные сообщения клиенту  
  
Такая схема повышает устойчивость, упрощает диагностику и улучшает клиентский опыт.

### Подробный ответ
#### Основной ответ
Обработка исключений в микросервисах — это комплексный процесс, который должен быть продуман на уровне как отдельного сервиса, так и всего распределённого приложения. Важно не просто локально ловить ошибки, но и обеспечить прозрачность, трассируемость и корректное реагирование на отказ системы.

#### Ключевые моменты
- **Локальная обработка ошибок:** в каждом микросервисе стоит использовать try-catch или аналогичные механизмы для перехвата исключений, логирования и возвращения стандартизированных ошибок (например, HTTP-кодов с подробным телом). Важно обеспечить понятные и повторяемые форматы ошибок (например, JSON с `code`, `message`, `details`).
- **Централизованное логирование и трассировка:** распределённые системы сложно отлаживать без корреляции ошибок. Обычно используют системы трейсинга (OpenTelemetry, Jaeger), которые передают traceId через заголовки между сервисами. Это позволяет проанализировать цепочку вызовов и выявить источник ошибки.
- **Управление отказами и fallback:** рекомендации по отказоустойчивости включают circuit breaker (Hystrix, Resilience4j), повторные попытки (retry), таймауты и fallback-логику. Это минимизирует воздействие ошибок из одного сервиса на всю систему.

#### Практический контекст
В реальных проектах, к примеру, в сервисах на Spring Boot с использованием **Spring Cloud Sleuth**+**Zipkin** для трассировки, ошибки перехватываются с помощью глобального `@ControllerAdvice`. Фреймворки API Gateway (Kong, Zuul) часто реализуют уровень fallback и rate limit, минимизируя проблему cascade failure. Также часто используют стандарты, такие как **Problem Details for HTTP APIs (RFC 7807)**, чтобы упростить работу с ошибками на клиенте.
