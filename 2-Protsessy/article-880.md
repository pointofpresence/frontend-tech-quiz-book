# Сколько запросов к БД выполнится с PrefetchRelated в Django?

Категория: Процессы

### Краткий ответ

**Сколько запросов к БД выполнится с PrefetchRelated в Django?**  
- ORM Django: оптимизация **жадной загрузки**  
- PrefetchRelated использует **2+ запроса**  
- 1 запрос для основной модели  
- 1 запрос для связанной модели с фильтрацией  
- данные из связанной модели **собираются в Python**  
- преимущество в **снижении повторных запросов (N+1)**  
- подходит для сложных фильтров и **ManyToMany/ForeignKey**  
- улучшает **производительность** при доступе к связанным данным

### Подробный ответ
#### Основной ответ
В Django использование **`prefetch_related`** оптимизирует количество запросов к базе данных, выполняя при этом ровно **2 SQL-запроса**: первый — основной запрос к основной модели, второй — запрос к связанным объектам. Затем Django объединяет результаты в Python, чтобы избежать N+1 проблемы.

#### Ключевые моменты
- **Первый запрос** загружает главные объекты (например, модели `Author`), тогда как **второй запрос** получает все связанные объекты (например, `Book`), относящиеся к загруженным авторам.
- В отличие от `select_related`, который делает один сложный join-запрос (для **ForeignKey / OneToOne** связей), `prefetch_related` всегда использует отдельные запросы и подходит для **ManyToMany** и обратных связей.
- Механизм выгрузки реализован через **отдельный SQL-запрос с фильтрацией по внешним ключам** и последующее связывание в Python, что увеличивает память, но снижает количество запросов.

#### Практический контекст
Например, чтобы получить авторов с их книгами, без prefetch_related будет выполняться 1 запрос к авторам + N запросов к книгам (N = число авторов). С `prefetch_related('books')` будет ровно 2 запроса, что значительно снижает нагрузку и ускоряет аппликацию при большом количестве данных. Часто встречается в Django 4.x и выше с оптимизациями ORM.
