# Какие есть способы переиспользования методов одного класса в другом (наследование, композиция)?

**Способы переиспользования методов одного класса в другом**

- **Наследование**: подкласс получает все публичные и защищённые методы родителя, расширяет или переопределяет поведение.
- **Композиция**: класс содержит объект другого класса, вызывает его методы для расширения функционала.
- **Делегирование**: методы вызывают аналоги внутри вложенного объекта, обеспечивая интерфейс переиспользования.
- Наследование подходит для "есть класс-тип", композиция — для "имеет" отношения.
- Композиция обеспечивает лучшую гибкость и слабую связанность.
- Множественное наследование (в некоторых языках) расширяет возможности, но усложняет код.
- Применение зависит от задачи и требований к архитектуре: наследование — для иерархий, композиция — для динамического поведения.

# Подробный ответ
## Основной ответ  
Переиспользование методов одного класса в другом — ключевой аспект объектно-ориентированного проектирования. Существуют два классических подхода: **наследование** и **композиция**.  

**Наследование** позволяет производному классу унаследовать методы и свойства базового, расширять или переопределять их. Это удобно, когда классы логически связаны и есть "является" (is-a) отношение.  

**Композиция** — это включение объекта одного класса как поля в другой класс. Здесь класс A использует сервисы класса B через делегирование вызовов, что позволяет создавать более гибкую архитектуру с "имеет" (has-a) отношением и облегчает тестирование и масштабирование.

## Ключевые моменты  
- **Наследование**:  
  - Быстрая переиспользуемость кода и иерархия классов  
  - Может привести к жесткой зависимости и проблемам поддержки (например, «хрупкое наследование»)  
  - В Java и C++ — поддерживает single/multi inheritance (в C++), в других языках — чаще single inheritance с интерфейсами  
- **Композиция**:  
  - Чаще считается более гибкой и предпочтительной в современных системах (принцип "предпочитай композицию наследованию")  
  - Позволяет менять поведение во время выполнения, облегчает внедрение зависимостей (Dependency Injection)  
  - Избегает проблем жесткой иерархии и путаницы с множественным наследованием  
- **Выбор подхода зависит от семантики отношений и требований** — если объект действительно должен быть подтипом, наследование оправдано; если хочет просто использовать функциональность — композиция  

## Практический контекст  
В React 18, например, композиция компонентов через props предпочтительней наследования, чтобы лучше контролировать поведение. В backend на Java 17 при построении доменных моделей часто используются абстрактные базовые классы с наследованием, но для сервисных классов — композиция с помощью Spring DI.  

Понимание правильного применения этих паттернов значительно влияет на масштабируемость и сопровождаемость кода.
