# Какие HTTP-статусы относятся к клиентским (4xx) и серверным (5xx) ошибкам?

Категория: Процессы

### Краткий ответ

**HTTP-статусы клиентских и серверных ошибок (4xx, 5xx)**  
- Клиентские ошибки — HTTP 4xx  
- Основные коды:  
  - 400 Bad Request — неверный запрос  
  - 401 Unauthorized — требуется аутентификация  
  - 403 Forbidden — доступ запрещён  
  - 404 Not Found — ресурс не найден  
  - 429 Too Many Requests — превышен лимит запросов  
- Серверные ошибки — HTTP 5xx  
- Основные коды:  
  - 500 Internal Server Error — внутренняя ошибка сервера  
  - 502 Bad Gateway — некорректный ответ шлюза  
  - 503 Service Unavailable — сервис временно недоступен  
  - 504 Gateway Timeout — таймаут шлюза  
- 4xx показывают ошибки клиента, 5xx — проблемы сервера  
- Важно для отладки, маршрутизации и UX  
- Используются для корректной обработки запросов и ошибок в API

### Подробный ответ
#### Основной ответ  
HTTP-статусы ошибок делятся на две основные категории: **4xx — клиентские ошибки** и **5xx — серверные ошибки**.  
**Клиентские ошибки (4xx)** означают, что запрос, отправленный клиентом, неправильный или не может быть выполнен из-за проблем на стороне клиента. Например, неверный синтаксис, отсутствие авторизации или доступ к несуществующему ресурсу.  
**Серверные ошибки (5xx)** сигнализируют, что запрос был корректным, но сервер не смог его обработать из-за внутренней ошибки или временного сбоя.

#### Ключевые моменты  
- **Клиентские ошибки (4xx):**  
  - 400 Bad Request — неверный формат запроса;  
  - 401 Unauthorized — необходима аутентификация;  
  - 403 Forbidden — доступ запрещён, несмотря на аутентификацию;  
  - 404 Not Found — ресурс не найден;  
  - 429 Too Many Requests — превышение лимитов запросов (rate limiting).  
- **Серверные ошибки (5xx):**  
  - 500 Internal Server Error — общая внутренняя ошибка сервера;  
  - 502 Bad Gateway — неверный ответ от промежуточного сервера;  
  - 503 Service Unavailable — сервис временно недоступен;  
  - 504 Gateway Timeout — таймаут ответа от сервера.  
- Важно понимать, что 4xx ошибки указывают на проблему в клиентском запросе, тогда как 5xx — на проблему на сервере, и это помогает разделять ответственность в логах и мониторинге.

#### Практический контекст  
В реальных проектах правильная классификация ошибок помогает автоматически реагировать на сбои: например, **клиентские 4xx** могут инициировать повтор запроса с исправлениями или отображать понятные пользователю сообщения, а **серверные 5xx** — запускать алерты для команды поддержки и переключать трафик на резервные сервисы. Использование систем мониторинга, таких как Prometheus и Sentry, помогает отслеживать эти ошибки по категориям.
