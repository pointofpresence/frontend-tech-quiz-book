# Что делает команда git add?

Категория: Процессы

### Краткий ответ

**Что делает команда git add?**

- Команда управления **индексом (staging area)** Git  
- Добавляет изменения (новые, модифицированные, удалённые файлы) в **индекс**  
- Готовит селективно файлы к следующему коммиту  
- Не изменяет историю, только формирует содержимое коммита  
- Работает с файлами в **рабочем каталоге**  
- Позволяет разбивать коммит на логичные части  
- Критична для контроля версий и управления изменениями  

Итог: git add формирует промежуточный снимок изменений для последующего коммита.

### Подробный ответ
#### Основной ответ
Команда **git add** переносит изменения файлов из рабочей директории в **индекс (staging area)**, подготавливая их для последующего коммита. Это не означает, что изменения сразу станут частью истории — фактический сохраненный снимок создаётся только после выполнения `git commit`. `git add` служит для явного выбора и формирования набора изменений, которые будут включены в следующий коммит.

#### Ключевые моменты
- **Индекс как промежуточный слой**: git хранит промежуточный набор изменений (staging area), который отделён от рабочей директории и локального репозитория. Это позволяет гибко собирать коммиты из кусочков изменений.
- **Добавление новых, изменённых и удалённых файлов**: `git add` учитывает как создание новых файлов, так и модификации или удаление ранее добавленных, фиксируя это для коммита.
- **Поддержка патчей и выборочных изменений**: с опцией `-p` можно интерактивно добавлять отдельные участки изменений, что упрощает создание чистой и логичной истории коммитов.
- **Не включает удаление при использовании только `git add <file>`** — для оформления удаления нужно явно добавить его (например, `git rm` либо `git add` после удаления файла).

#### Практический контекст
В реальных проектах грамотное использование `git add` помогает создавать аккуратные коммиты с логичной структурой истории — вместо коммита всех изменений сразу можно сформировать несколько меньших коммитов с отдельными логическими изменениями. В командной разработке это облегчает код-ревью и отладку. Например, при работе в React 18 можно сначала сделать отдельный коммит по стилизации компонентов, а потом добавить логику — для прозрачной истории изменений.
