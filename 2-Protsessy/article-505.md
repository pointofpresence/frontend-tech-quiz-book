# У каких сущностей в Java/Kotlin есть собственный стек?

**У каких сущностей в Java/Kotlin есть собственный стек?**  
- многопоточность; область памяти  
- у каждого **потока** есть **собственный стек**  
- стек хранит **локальные переменные**, вызовы функций и параметры  
- **не общий**, изолирован для потокобезопасности  
- стек фиксированного размера, влияет на глубину рекурсии  
- отдельный стек → изоляция контекста вызовов  
- важно для **управления вызовами** и **эффективного исполнения**  
Итог: только у **потоков (threads)** в Java/Kotlin есть собственный стек.

# Подробный ответ
## Основной ответ
В Java и Kotlin **собственный стек (stack)** имеют только **потоки (threads)**. Каждый поток исполнения имеет независимый стек вызовов, который хранит информацию о текущих методах, локальных переменных и состоянии выполнения. Другие сущности — объекты, классы или процессы — собственного стека не имеют.

## Ключевые моменты
- **Стек потока** содержит записи о вызовах методов (stack frames), включая ссылки на локальные переменные, параметры и адрес возврата.
- **Объекты и классы** располагаются в куче (heap), где хранится их состояние; стек для них не выделяется, потому что они не управляют потоком выполнения.
- В Kotlin, как и в Java (JVM), модель памяти совпадает, поэтому и стек есть только у потоков, в том числе корутин у Kotlin нет собственного стека, это лёгкие задачи, работающие на потоках с переключением контекста.

## Практический контекст
При отладке и анализе производительности важно понимать, что стековая память ограничена и локальные переменные в методах живут именно в стеке текущего потока. Ошибки, такие как StackOverflowError, напрямую связаны с переполнением именно этого стека. В отличие от этого, объекты с большой нагрузкой по памяти — на куче. Кроме того, при создании новых потоков в JVM выделяется под них отдельный стек фиксированного размера (например, по умолчанию около 1 Мб, но это настраивается).
