# Какие бывают типы баз данных?

**Типы баз данных**  
- по модели данных: **реляционные, NoSQL**  
- реляционные: таблицы, SQL, ACID  
- NoSQL: колоночные, документные, графовые, ключ-значение  
- реляционные — строгая схема, сложные запросы  
- NoSQL — гибкая схема, масштабируемость, быстрое чтение/запись  
- выбор зависит от задач: аналитика, транзакции, масштабирование  
- популярные СУБД: **PostgreSQL, MySQL, MongoDB, Cassandra, Neo4j**

# Подробный ответ
## Основной ответ
Типы баз данных классифицируются по модели данных, которую они используют для организации, хранения и управления информацией. К основным типам относятся **реляционные**, **документно-ориентированные**, **ключ-значение**, **графовые** и **колонко-ориентированные** базы данных.

## Ключевые моменты
- **Реляционные БД (SQL)** основываются на таблицах с четко заданной схемой, поддерживают сложные JOIN-запросы, транзакции с ACID (напр. PostgreSQL 14+, MySQL 8). Это основной выбор для ERP, финансовых систем.
- **NoSQL базы** делятся на несколько подтипов: 
  - **Документно-ориентированные (MongoDB, Couchbase)** хранят данные в формате JSON-подобных документов — гибко и удобно для быстрых изменений схемы.
  - **Ключ-значение (Redis, DynamoDB)** — максимально простой тип, данные хранятся как пары ключей и значений, идеальны для кэша и session storage.
  - **Графовые (Neo4j, Amazon Neptune)** оптимизированы для отношений и сетевых структур, идеально подходят для социальных сетей, рекомендаций.
  - **Колонко-ориентированные (Apache Cassandra, ClickHouse)** хранят данные по колонкам, что ускоряет аналитические запросы на больших объемах данных.
- **Выбор типа БД зависит от сценария**: реляционные БД — для строгой целостности и структурированных данных, NoSQL — для гибкости, масштабируемости и высоких нагрузок.

## Практический контекст
В реальных проектах часто используют комбинации: например, PostgreSQL 14+ для OLTP (оперативные транзакции) и ClickHouse для OLAP-аналитики. MongoDB — для быстрых agile-проектов с нефиксированной схемой, Redis — для кэширования и rate limiting. Такой подход обеспечивает баланс между производительностью, гибкостью и поддерживаемостью.
