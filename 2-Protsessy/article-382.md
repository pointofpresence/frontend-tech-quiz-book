# Как сделать так, чтобы в Kafka для пользователя хранилось только последнее событие?

**Как хранить в Kafka только последнее событие для пользователя?**

- управление через **ключи сообщений** (key) — ключом укажите идентификатор пользователя  
- включить **compaction** для топика — Kafka оставляет только последнее сообщение с уникальным ключом  
- compaction регулярно удаляет старые дубликаты по ключу, оставляя последнее значение  
- задать параметр топика: `cleanup.policy=compact`  
- полезно, когда нужна актуальная информация, а не полный лог событий  
- ограничение: для удаления записей нужно отправлять сообщения с null-значением (т.н. tombstone)  
- используется для **кеширования состояния** или **репликации исходного состояния**

Таким образом, ключевая реализация — топик с compaction по ключу пользователя, где хранится только актуальное последнее событие.

# Подробный ответ
## Основной ответ
Чтобы в Apache Kafka для пользователя хранилось только последнее событие, необходимо использовать **ключ сообщения (message key)** и настроить топик как **compacted topic**. В режиме компактации Kafka удаляет старые записи с одинаковым ключом, оставляя только последние значения, что эффективно сохраняет для каждого ключа только самое свежее событие.

## Ключевые моменты
- **Использование ключа сообщения**: каждому событию пользователя назначается уникальный ключ, например, userId. Все обновления для данного пользователя публикуются с тем же ключом.
- **Включение логики создания compacted топика**: за счёт параметра `cleanup.policy=compact` Kafka будет держать для каждого ключа только последнюю версию события, удаляя старые версии в фоне.
- **Баланс между latency и storage**: процесс компактации выполняется асинхронно, поэтому старые версии могут на время оставаться, но со временем топик будет очищен и уменьшится до только актуальных данных.
- Дополнительно можно настроить параметры компрессии и retention, чтобы оптимизировать использование диска.

## Практический контекст
Такой подход часто применяется для хранения состояния пользователя или профиля, где важна актуальная информация, а не весь поток событий. Например, в микросервисах обновления пользовательских данных записываются с userId в качестве ключа и топик имеет политику compact. Это позволяет реализовать эффективное event sourcing с минимальным хранилищем без потери актуальных данных.
