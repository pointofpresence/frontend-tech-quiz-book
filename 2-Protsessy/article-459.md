# Как работает получение данных на сервере в Next.js? (Раньше — getServerSideProps, в App Router — async-компоненты)

**Как работает получение данных на сервере в Next.js?**

- Next.js: фреймворк React с SSR и SSG  
- Ранее: **getServerSideProps** — серверный метод для каждой страницы  
- getServerSideProps вызывается **на сервере при каждом запросе**  
- Возвращает props для компонента, рендеринг на сервере  
- В App Router (Next.js 13+) — **async-функции в компонентах**  
- Асинхронный компонент напрямую выполняет вызовы API или БД  
- Позволяет **серверный рендеринг без оберток**, снижая boilerplate  
- Вывод: эволюция от отдельных функций к нативным async React-компонентам для SSR  
- Используется для SEO и **динамического контента** с актуальными данными

# Подробный ответ
## Основной ответ
В **Next.js** получение данных на сервере эволюционировало: в традиционном режиме (Pages Router) для получения данных на сервере использовали функцию **getServerSideProps**, которая запускалась на сервере при каждом запросе, формируя необходимые пропсы для страницы. В **App Router** с релизом Next.js 13+ подход изменился — теперь предпочтительно использовать **асинхронные React-компоненты** и Server Components с встроенной возможностью асинхронного получения данных через `fetch` или другие асинхронные вызовы напрямую внутри компонента.

## Ключевые моменты
- В **getServerSideProps** (Pages Router) код запускается каждый раз на сервере при запросе, позволяет делать запросы к БД, API, и передавать данные в props компонента, что важно для динамического рендеринга.
- В **App Router** в async-компонентах данные получают прямо внутри компонента, благодаря React Server Components, которые выполняются на сервере, поддерживают асинхронность и возвращают уже готовый JSX.
- Такой подход позволяет избавиться от отдельного слоя для загрузки данных, упрощает архитектуру и улучшает производительность за счёт автоматического streaming и оптимизации рендера.
- Fetch в App Router по умолчанию кешируется (используя React Cache) и может быть настроен под ISR (Incremental Static Regeneration) с помощью опций `next: { revalidate: N }`.
- Обновленный подход лучше интегрируется с Suspense, что позволяет показывать loading state на уровне компонентов, не перезагружая страницу полностью.

## Практический контекст
В реальных проектах с Next.js 12-13 часто встречается смешанный мир: legacy проекты продолжают использовать **getServerSideProps** для гибкости, а новые проекты с App Router переходят на async Server Components с встроенным fetch, что снижает количество вспомогательных функций и упрощает работу с данными. Такой переход улучшает developer experience и помогает поддерживать современный, реактивный UI с минимальными задержками и без лишнего client-side JS.
