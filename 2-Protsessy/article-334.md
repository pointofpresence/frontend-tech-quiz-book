# Как вернуть доменную ошибку из сервиса и отдать корректный HTTP-код?

Категория: Процессы

### Краткий ответ

**Как вернуть доменную ошибку из сервиса и отдать корректный HTTP-код?**

- Контекст: обработка ошибок в backend-сервисах, API
- Использую **исключения** (exceptions) или **error objects** для представления доменных ошибок
- В сервисе создаю специфичные ошибки с **описанием причины** (например, NotFoundError, ValidationError)
- В контроллере или middleware **перехватываю ошибки** и сопоставляю с HTTP-кодами:
  - 400 для ошибок валидации
  - 404 для отсутствующих ресурсов
  - 403 для отказа в доступе
  - 500 для непредвиденных ошибок
- Формирую ответ с **корректным статусом** и информативным JSON с сообщением ошибки
- Использую централизованный механизм обработки ошибок для консистентности и поддержки
- Такой подход отделяет доменную логику от протокольной реализации и улучшает UX API клиента

### Подробный ответ
#### Основной ответ
Чтобы вернуть доменную ошибку из сервиса и отдать корректный HTTP-код, нужно реализовать явное разделение бизнес-логики и транспортного уровня. Обычно сервисы выбрасывают или возвращают специальные **доменные ошибки** (exceptions или объекты ошибок), которые содержат контекст проблемы. Контроллер (или слой API) перехватывает эти ошибки, сопоставляет их с HTTP-статусом и формирует ответ клиенту с нужным кодом и сообщением.

#### Ключевые моменты
- **Определение собственных доменных ошибок**, наследуемых от базового класса ошибок, с четкой смысловой нагрузкой (например, `ValidationError`, `NotFoundError`, `BusinessRuleViolationError`).
- **Обработка ошибок на уровне контроллера или middleware** (в зависимости от используемого фреймворка, например, Express в Node.js или ASP.NET middleware), где происходит маппинг каждой доменной ошибки на HTTP-статус (например, 400 для валидации, 404 — для отсутствующих ресурсов, 409 — для конфликтов).
- **Безопасность и аудит**: возвращать клиенту только необходимую информацию, логи полной ошибки сохранять отдельно.
- В комплексных системах часто применяют **паттерн Result / Either**, когда сервис не вызывает исключения, а возвращает результат с флагом успеха/ошибки и метаданными.

#### Практический контекст
В реальных проектах, например в **ASP.NET Core 6**, можно реализовать глобальный фильтр исключений (`ExceptionFilter`), который ловит доменные ошибки и возвращает `ProblemDetails` с нужным HTTP-кодом. В REST API на **Spring Boot (Java)** — использовать `@ControllerAdvice` с методами, обрабатывающими кастомные исключения и передающими статус через `ResponseEntity`. Такой подход улучшает тестируемость, поддерживаемость кода и улучшает UX при взаимодействии с API.
