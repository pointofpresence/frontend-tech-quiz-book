# Напишите запрос для выборки всех кредитов физическим лицам с просроченной задолженностью.

**Запрос всех кредитов с просроченной задолженностью физическим лицам**  
- SQL, запрос к базе кредитов  
- Выбираем кредиты с типом клиента = "физическое лицо"  
- Условие: дата просрочки (due_date) < текущей даты и статус задолженности = "просрочена"  
- Идентификаторы клиента и сумма кредита включены  
- Используем WHERE для фильтрации по типу клиента и просрочке  
- Пример использования функций даты (CURRENT_DATE) для проверки просрочки  
- Результат — список задолженностей для мониторинга и принятия решений  

---

**Пример запроса:**
```sql
SELECT credit_id, client_id, amount, due_date
FROM credits
WHERE client_type = 'individual'
AND due_date < CURRENT_DATE
AND status = 'overdue';
```

# Подробный ответ
## Основной ответ
Для выборки всех кредитов физическим лицам с просроченной задолженностью требуется сформировать SQL-запрос, который фильтрует записи клиентов по типу (физическое лицо) и по признаку просрочки (наличие суммы задолженности по дате платежа). Конкретная реализация зависит от структуры базы данных, но типично в таблице кредитов присутствят поля: `borrower_type` или ссылка на тип клиента, дата последнего платежа, сумма остатка долга и дата платежа.

Пример типового запроса в SQL:
```sql
SELECT *
FROM credits
WHERE borrower_type = 'individual'
  AND due_date < CURRENT_DATE
  AND outstanding_amount > 0;
```

## Ключевые моменты
- Поле `borrower_type` или аналог — определяет категорию клиента (физ. лицо). Иногда используется отдельная таблица клиентов с типом, тогда запрос необходимо дополнить JOIN-ом.
- Условие `due_date < CURRENT_DATE` указывает на просрочку по дате платежа — прошла дата, но задолженность (`outstanding_amount`) ещё не погашена.
- Возможен вариант с подсчётом просрочки через статус кредита или наличие более сложной логики — например, игнорирование кредитов с реструктуризацией или списанием.

## Практический контекст
В реальных системах кредитного учёта таблицы могут называться `loans`, `customers`, и просрочка может фиксироваться в отдельной сущности (например, `delinquencies`). Для высокой производительности используют индексы по типу клиента и дате платежа. В проектах на PostgreSQL 14+ нередко применяют оконные функции и CTE для вычисления длительности просрочки.
