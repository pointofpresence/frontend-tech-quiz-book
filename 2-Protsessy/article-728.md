# Как избежать передеплоя при изменении захардкоженного списка наград? (конфигурация через application.properties vs. внешний сервис).

**Как избежать передеплоя при изменении захардкоженного списка наград?**

- конфигурация через properties — файл жестко связан с билдом  
- изменение требует **рестарт/деплой** приложения  
- внешний сервис (Config Server, Consul, Spring Cloud Config) — конфигурация загружается **динамически**  
- можно реализовать **горячее обновление** без рестарта через refresh scope или watcher  
- использование БД или API для хранения списка наград — актуально и гибко  
- преимущество внешнего сервиса — **гибкость**, **масштабируемость**, оперативные изменения  
- в продакшене лучше держать конфигурацию вне кода для **минимизации downtime** и **оперативности обновлений**

# Подробный ответ
## Основной ответ
Чтобы избежать **передеплоя приложения** при изменении захардкоженного списка наград, лучше отказаться от хранения этого списка прямо в `application.properties` и использовать более динамические подходы: внешний конфигурационный сервис или управляемое хранилище данных. Это позволяет менять значения на лету, без пересборки и релиза артефактов.

## Ключевые моменты
- **Конфигурация через external config сервисы** (например, Spring Cloud Config, Consul, ZooKeeper) даёт возможность менять параметры централизованно и обновлять их в рантайме через механизм refresh (например, actuator /event refresh).
- **Хранение в БД или распределённом кэше** — если список сложный и часто меняется, можно управлять наградами через интерфейс администрирования, данные хранятся в базе, а приложение читает их через DAO. Это решает задачи динамичности и audit.
- **Trade-off**: `application.properties` прост в реализации, хорош для неизменяемых или редко меняемых значений. Но при любых изменениях требуется новый деплой. Внешние сервисы усложняют архитектуру, но дают гибкость и уменьшают downtime.

## Практический контекст
В современных микросервисных архитектурах типично выносить конфигурацию в Spring Cloud Config или использовать Consul/Etcd, что позволяет изменять параметры (например, список наград) без перезапуска сервиса. В проектах с высокой нагрузкой нередко применяют кеширование списка с периодическим обновлением из источника данных, чтобы избежать излишних запросов и обеспечить быструю реакцию на изменение настроек.
