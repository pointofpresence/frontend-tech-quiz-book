# Какие альтернативные брокеры сообщений знаешь (помимо Rabbit)?

Категория: Процессы

### Краткий ответ

**Альтернативные брокеры сообщений (кроме RabbitMQ)**  
- Kafka: **распределённый**, высокая пропускная способность  
- ActiveMQ: классический **JMS-брокер**, поддержка **транзакций**  
- Redis Streams: встраиваемый, **низкая задержка**, **in-memory**  
- Amazon SQS: облачный, **управляемый сервис**, высокая масштабируемость  
- NATS: **легковесный**, для микросервисов, высокая производительность  
- Pulsar: мульти-тенант, поддержка **очередей и потоков**  
- ZeroMQ: **библиотека**, без центрального сервера, гибкая топология  
- MQTT-брокеры (Mosquitto, EMQX): для IoT, **легкий протокол**  
- Google Pub/Sub: облачный, **автоматическое масштабирование**, интеграция с GCP  
- Apache Artemis: современный **JMS-брокер**, форк ActiveMQ  
Эти брокеры различаются по архитектуре, производительности, гарантиям доставки и применению в конкретных сценариях.

### Подробный ответ
#### Основной ответ
Помимо **RabbitMQ**, на рынке существует множество альтернативных брокеров сообщений, каждый из которых подходит для разных сценариев и архитектур. К наиболее популярным относятся **Apache Kafka**, **Amazon SQS**, **Apache ActiveMQ**, **Redis Streams**, а также более легковесные решения вроде **NATS**.

#### Ключевые моменты
- **Apache Kafka** — ориентирован на высокопроизводительную обработку потоковых данных, обеспечивает устойчивое хранение сообщений и обработку с гарантией доставки. Часто используется в больших распределённых системах для аналитики, event sourcing, и микросервисной коммуникации.
- **Amazon SQS** — полностью управляемый сервис от AWS с высокой масштабируемостью, подходит для облачных приложений с асинхронной обработкой заданий, обеспечивает at-least-once доставку, но без гарантий порядка сообщений.
- **Apache ActiveMQ** — классический брокер с поддержкой JMS, подходит для систем с широким спектром протоколов и сложных топологий сообщений, хорошо работает в традиционных enterprise-средах.
- **Redis Streams** — часть Redis 5+, используется для управления журнальными потоками данных, идеален там, где требуется низкая задержка и простая интеграция с кэшом и базой.

#### Практический контекст
В реальных проектах часто выбирают Kafka для больших data pipelines и event-driven архитектур, RabbitMQ для сложной маршрутизации и гарантий доставки, а Redis Streams для небольших систем с требованиями к высокой скорости. AWS SQS часто используется в облачных микросервисах благодаря простоте и управляемости. Выбор брокера зависит от требований к задержкам, гарантиям доставки, сложности маршрутизации и экосистемы.
