# Какие альтернативы Retrofit вы использовали (Ktor)? Их плюсы/минусы.

Категория: Процессы

### Краткий ответ

**Альтернативы Retrofit (Ktor) — плюсы и минусы**  
- Retrofit: **декларативный** HTTP клиент, встроенный **парсинг JSON** (Gson/Moshi)  
- Ktor client: **асинхронный**, **корутины**, гибкая настройка, **мультиплатформенность**  
- OkHttp: **низкоуровневый** HTTP, высокая производительность, часто базис Retrofit  
- Fuel: простой, **легковесный**, быстрый старт, слабее кастомизация  
- Apollo (GraphQL): специализирован для GraphQL API, кодогенерация  
- Плюсы Ktor: хорошая интеграция с Kotlin, **поддержка coroutines**, расширяемость  
- Минусы Ktor: меньше готовых интеграций/библиотек, больший **объем настройки**  
- Retrofit: проще для REST API, широкая экосистема, меньше гибкости  
- Выбор зависит от требований: Retrofit — быстро и удобно, Ktor — кастомизация и мультиплатформенность  

**Итог:** Ktor предпочитаю для корутинного, мультиплатформенного кода и кастомных решений, Retrofit — для стандартных REST с простотой и скоростью разработки.

### Подробный ответ
#### Основной ответ
Помимо **Retrofit**, для работы с HTTP в Kotlin часто используют **Ktor Client** — асинхронный и более гибкий HTTP клиент от JetBrains. Retrofit — это зрелая библиотека с декларативным API и простым интерфейсом, хорошо подходит для синхронного и асинхронного взаимодействия с REST API. Ktor же предоставляет больше контроля и подходит для более сложных, кастомных сценариев сетевого обмена, особенно в мультиплатформенных проектах (KMM).

#### Ключевые моменты
- **Retrofit:**
  - Простота интеграции, использование аннотаций и конвертеров (Gson, Moshi).
  - Встроенная синхронизация и поддержка Callbacks/Coroutines (начиная с версии 2.6+).
  - Хорошо подходит для REST API с четкой структурой.
  - Ограниченная гибкость в настройке запросов и нестандартного поведения.
  
- **Ktor Client:**
  - Полная асинхронность через Kotlin Coroutines, поддержка мультиплатформенности (Android, iOS, JVM, JS).
  - Гибкая настройка pipeline запросов/ответов, interceptors, кастомных плагинов.
  - Требует больше кода для создания моделей запросов и ответов.
  - Lighter, не зависит от Reflection, лучше подходит для Kotlin Native.

#### Практический контекст
В проектах на Android я выбираю **Retrofit** для быстрого старта и чистой интеграции REST API, что снижает boilerplate. Если же нужна мультиплатформенность или продвинутый контроль (например, кастомные WebSocket или сложная обработка ответов), то предпочитаю **Ktor Client** — там больше инструментов для расширения и настройки.

#### Итог
Retrofit — «работает из коробки», удобен для большинства REST-сценариев, Ktor — более современный и гибкий, особенно для мультиплатформенных решений и нестандартных задач, но требует чуть больше усилий на конфигурацию и написание кода.
