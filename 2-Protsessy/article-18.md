# Можно ли создать собственный тип ошибки (custom error type)? Как?

Категория: Процессы

### Краткий ответ

**Можно ли создать собственный тип ошибки (custom error type)? Как?**

- Создание кастомной ошибки — стандартная практика для улучшения обработки ошибок  
- Наследуют от встроенного класса **Error** (в JS, Python, других)  
- Определяют имя через свойство `name` (например, `CustomError`)  
- Добавляют специфичные поля для контекста ошибки  
- В JS пример: класс с `extends Error` + вызов `super(message)`  
- Польза: более точная фильтрация и логирование ошибок  
- В продакшене облегчает отладку и поддержку кода благодаря семантике ошибок

### Подробный ответ
#### Основной ответ
Да, в JavaScript и TypeScript можно создать собственный **custom error type** для более точной обработки ошибок и улучшения отладки. Это достигается через наследование от встроенного класса **Error**. Кастомные ошибки позволяют добавлять специфичные свойства и методы, которые помогают лучше описать контекст ошибки.

#### Ключевые моменты
- **Наследование от Error**: создаётся класс, расширяющий Error, при этом вызывается конструктор базового класса через `super(message)`.
- **Поддержка стека вызовов**: обязательно нужно правильно настроить `this.name` и можно вызвать `Error.captureStackTrace` (в Node.js), чтобы стек соответствовал новому типу.
- **Дополнительные свойства**: часто добавляют поля, например, `code`, `details` или любую бизнес-логику для более удобной обработки.
 
#### Практический контекст
В проектах с большой бизнес-логикой кастомные ошибки помогают разделять ошибки валидации, сетевые ошибки или ошибки доступа. Например, в REST API можно создавать ошибки `ValidationError` с HTTP-кодом 400, а `AuthenticationError` — с кодом 401, чтобы middleware корректно сформировал ответ клиенту.

---

Пример создания простого custom error в JS:

```js
class ValidationError extends Error {
  constructor(message) {
    super(message);
    this.name = 'ValidationError';
    if (Error.captureStackTrace) {
      Error.captureStackTrace(this, ValidationError);
    }
  }
}
```

Такой подход широко используют с React 18+, Node.js 14+ и другими современными стэками для более понятного и удобного управления ошибками.
