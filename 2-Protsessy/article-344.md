# Как маппить доменные ошибки на коды HTTP? (404/400/409/500)

Категория: Процессы

### Краткий ответ

**Как маппить доменные ошибки на HTTP-коды?**  
- Ошибки — абстракция бизнес-логики, HTTP — протокол  
- 400 (Bad Request): ошибки валидации, некорректные входные данные  
- 404 (Not Found): сущность не найдена в хранилище  
- 409 (Conflict): нарушение бизнес-правил, конфликт состояния (например, уникальность)  
- 500 (Internal Server Error): неизвлечённые системные ошибки, баги или исключения  
- Использовать централизованный middleware/фильтр для маппинга  
- Ясные сообщения клиенту + корректный HTTP-код для REST API  
- Такая стратегия упрощает отладку и поддерживает контракт API с клиентом

### Подробный ответ
#### Основной ответ
Маппинг доменных ошибок на HTTP-коды — важная часть построения RESTful API, которая позволяет корректно информировать клиента о результате операции и характере возникшей ошибки. Идея в том, чтобы применять такие клиентские и серверные коды состояний HTTP, которые максимально точно отражают суть доменной ошибки.

#### Ключевые моменты
- **404 Not Found** — подходит для ошибок, когда запрашиваемый ресурс не существует в системе. Например, попытка получить пользователя по несуществующему ID. Это классический случай отсутствия данных.
- **400 Bad Request** — используется, если ошибка вызвана некорректным запросом, например, нарушением валидации данных или формата запроса. Это прямая ответственность клиента, и сервер сообщает о неправильных параметрах.
- **409 Conflict** — применяется для конфликтов в бизнес-логике или состояниях, например при попытке создать ресурс с уникальным полем, которое уже занято, или при нарушении бизнес-правил, таких как дедупликация или ограничение состояния.
- **500 Internal Server Error** — зарезервирован для непредвиденных сбоев на сервере, таких как ошибки инфраструктуры, таймауты, ошибки базы данных и другие исключения, не связанные напрямую с действиями клиента.

#### Практический контекст
В реальных проектах часто реализуют собственную иерархию исключений — например, `ResourceNotFoundException` → 404, `ValidationException` → 400, `ConflictException` → 409 и ловят их в слое контроллера или middleware, маппируя на соответствующие HTTP-статусы. Такой подход улучшает поддержку и однозначность обработки ошибок. Чтобы минимизировать «583 Unknown Error», важно обеспечить хорошее логирование и документировать все коды статусов API (OpenAPI). Использование стандартных кодов HTTP повышает предсказуемость клиента и удобство отладки.
