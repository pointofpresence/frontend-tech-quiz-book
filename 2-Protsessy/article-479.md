# Как на фронтенде реализуется авторизация? (Хранение access/refresh токенов в Cookies / LocalStorage, отправка токена в запросах, обновление токена)

**Как на фронтенде реализуется авторизация?**

- Авторизация — управление доступом пользователя через токены  
- Access токен хранят в **HttpOnly Cookies** для защиты от XSS, либо в памяти (лучше)  
- Refresh токен — в **HttpOnly Secure Cookies** для обновления access без участия пользователя  
- Access токен отправляется в заголовке Authorization: Bearer при API-запросах  
- На истечение access токена — автоматический вызов API обновления с refresh токеном  
- После обновления — новый access токен сохраняется, запросы продолжаются без перезахода  
- Используется **JWT** или OAuth 2.0, ключевой акцент на безопасность: XSS, CSRF защита  

---

**Подробности на интервью:**  
- LocalStorage уязвим к XSS — не рекомендуется хранить access/refresh токены  
- Cookies с HttpOnly не доступны JS — лучший вариант для refresh токена  
- Политика безопасности CORS и SameSite для предотвращения CSRF-атак  
- Важно обрабатывать ошибки обновления (токены истекли/недействительны) и реализовать логаут  
- Применение интерсепторов для автоматической подстановки токена в запросы и рефреша  

---

Это демонстрирует глубокое понимание современных практик безопасной авторизации на клиенте.

# Подробный ответ
## Основной ответ  
Авторизация на фронтенде обычно реализуется через получение и хранение **access** и **refresh токенов**, которые подтверждают аутентификацию пользователя. После успешного входа сервер выдает эти токены, и клиент использует их для авторизованных запросов к API. **Access токен** нужен для аутентификации запросов и имеет короткий срок жизни (~15-60 минут), а **refresh токен** — для получения нового access токена без повторного логина, с более длительным сроком жизни (дни или недели).  

Токены можно хранить как в **HttpOnly Cookies**, так и в **LocalStorage/SessionStorage**, но предпочтительнее использовать HttpOnly cookies, т.к. они защищены от XSS-атак и автоматически отправляются браузером на сервер. При этом access токен обычно хранят в памяти или в non-persistent хранилищах, чтобы минимизировать риски кражи.

## Ключевые моменты  
- **Хранение токенов:**  
  - **HttpOnly Secure Cookies** защищают от XSS, автоматически добавляются в запросы к тому же домену и рекомендуются для refresh токена.  
  - **LocalStorage/SessionStorage** условно простые, но уязвимы к XSS, поэтому лучше там не хранить токены с повышенными правами.  
- **Отправка токенов:**  
  - Access токен обычно передается в заголовке `Authorization: Bearer <token>` при запросах к API.  
  - Refresh токен отправляется при истечении access токена для получения нового, чаще это касается запроса к эндпоинту `/refresh`.  
- **Обновление токена:**  
  - Клиент реализует логику автоматического перехвата 401 ошибок и отправки refresh токена для получения свежего access токена.  
  - Часто используют интерсепторы axios или fetch wrappers для автоматизации этого процесса.

## Практический контекст  
В React 18+ обычно после логина сохраняют refresh токен в HttpOnly cookie с флагами `Secure; SameSite=Strict`, а access токен держат в памяти React state или Redux store. При каждом запросе access токен прикрепляют в заголовок Authorization. Если приходит 401, вызывается эндпоинт обновления токена, и новый access токен обновляет состояние. Этот подход позволяет обеспечить баланс между безопасностью и удобством пользователя. Для мониторинга состояния авторизации часто используют глобальные состояния и уведомления пользователя о необходимости повторной авторизации.
