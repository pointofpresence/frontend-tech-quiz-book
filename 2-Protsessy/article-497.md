# Как работает связка wait(), notify(), notifyAll()?

**Как работает связка wait(), notify(), notifyAll()?**

- Механизм синхронизации в **Java** для **управления потоками**
- Все три метода вызываются только внутри блока **synchronized**
- **wait()** — поток освобождает монитор и переходит в состояние ожидания
- **notify()** — пробуждает один случайный поток из ожидания на этом мониторе
- **notifyAll()** — пробуждает все ожидающие потоки на этом мониторе
- Переход из wait в runnable происходит только после повторного захвата монитора
- Используется для координации потоков при условных ожиданиях (producer-consumer)
- Правильное применение предотвращает **гонки** и **дедлоки**

Этот механизм позволяет реализовать эффективное взаимодействие потоков через **мониторы** и условные переменные.

# Подробный ответ
## Основной ответ
Методы **wait()**, **notify()** и **notifyAll()** — ключевые механизмы работы с потоками в Java, обеспечивающие координацию и синхронизацию между потоками через монитор объекта. Метод **wait()** заставляет текущий поток освободить монитор и перейти в состояние ожидания, пока другой поток не вызовет **notify()** или **notifyAll()** на том же объекте. Метод **notify()** пробуждает один случайный поток, ожидающий этот монитор, в то время как **notifyAll()** пробуждает все ожидающие потоки.

## Ключевые моменты
- **Вызов внутри synchronized блока**: Все три метода нужно вызывать, держа монитор объекта, иначе возникает IllegalMonitorStateException.
- **wait() освобождает монитор**: Поток при вызове wait() освобождает монитор, позволяя другим потокам войти в synchronized блок.
- **notify() пробуждает один поток, notifyAll() — все**: Обычно notifyAll() используют для избежания проблем с "залипшими" потоками, например, при многопоточных очередях.
- **После пробуждения поток не сразу возобновляет работу**, он переходит в runnable, ждет возврата мониторного замка, и только тогда продолжает выполнение после строки, где вызвали wait().

## Практический контекст
Связка используется для реализации условных ожиданий, например, в producer-consumer паттерне: производитель ждёт, пока есть место в буфере, потребитель — пока есть данные. В Java 14+ чаще применяют **java.util.concurrent** пакеты с более современными средствами, но wait/notify остаётся фундаментальным для низкоуровневой синхронизации.
