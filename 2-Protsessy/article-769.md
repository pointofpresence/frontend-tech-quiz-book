# Как работает планировщик горутин (роль GOMAXPROCS, зависимость от ядер ОС)?

**Как работает планировщик горутин (роль GOMAXPROCS, зависимость от ядер ОС)**

- Планировщик Go — **M:N модель**: сопоставляет множество горутин с меньшим числом системных потоков (OS threads).
- Горутину планирует встроенный в runtime **планировщик**, который назначает их потокам для выполнения.
- **GOMAXPROCS** — переменная среды/параметр, задающий число **OS потоков (P)**, одновременно исполняемых планировщиком.
- Значение GOMAXPROCS обычно равно числу **логических ядер CPU** ОС, чтобы максимально загружать CPU без излишней конкуренции.
- Каждому P соответствует очередь горутин, с которой планировщик берет задачи для выполнения.
- Планировщик осуществляет **многозадачность**, переключая горутины между потоками, учитывая блокировки и ожидания.
- При блокировке системного вызова планировщик запускает другую горутину в другом потоке, что поддерживает **высокую параллельность**.
- GOMAXPROCS можно менять во время работы программы, влияя на уровень параллелизма и производительность.
- Планировщик оптимизирован для работы с **современными многоядерными процессорами**, эффективно балансируя нагрузку.

Итого: планировщик Go распределяет горутины по ограниченному числу потоков, заданному **GOMAXPROCS**, который ориентирован на количество ядер ОС для оптимальной параллельной работы.

# Подробный ответ
## Основной ответ  
Планировщик горутин в Go — это **многоуровневый рантайм-компонент**, который управляет исполнением легковесных "нитей" — горутин. Он обеспечивает конкурентное выполнение горутин на доступных системных потоках (M — machine thread), которые, в свою очередь, работают на ядрах ОС. Параметр **GOMAXPROCS** определяет максимальное количество одновременно работающих системных потоков, то есть максимальное число горутин, которые могут реально выполняться параллельно на уровне ОС. Значение этой переменной по умолчанию совпадает с числом логических ядер процессора, доступных ОС.

## Ключевые моменты  
- **Модель M:N**: горутины (G) мультиплексируются на меньшем числе системных потоков (M), которые планируются ОС. Планировщик Go управляет этим распределением, создавая, при необходимости, новые потоки до уровня ограниченного GOMAXPROCS.  
- **GOMAXPROCS** определяет количество потоков, которые одновременно могут выполнять код Go параллельно. Например, при GOMAXPROCS=1 все горутины последовательно разделяют один ОС поток. При увеличении — система использует больше ядер. Можно менять динамически через `runtime.GOMAXPROCS()`.  
- Зависимость от **ядер ОС** — планировщик не может эффективно задействовать больше потоков, чем есть физических или логических ядер, иначе будет лишняя контекстная смена. ОС распределяет потоки по ядрам, а Go-рантайм планирует горутины внутри этих потоков.  

## Практический контекст  
В современных версиях Go (1.5 и выше) планировщик полностью интегрирован в рантайм и оптимизирован под многопроцессорные системы (в т.ч. с гиперпоточностью). Обычно для CPU-bound задач выставляют GOMAXPROCS равным числу ядер системы, чтобы добиться максимального использования ресурсов — например, на сервере с 8 ядрами задают `GOMAXPROCS=8`. Для IO-bound задач это может быть избыточно. Понимание этого помогает оптимизировать производительность и избежать лишнего переключения контекста.
