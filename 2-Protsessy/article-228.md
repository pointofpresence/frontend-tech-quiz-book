# Как работает Dependency Injection (DI) в Spring? Как вы его использовали?

**Как работает Dependency Injection (DI) в Spring? Как вы его использовали?**

- Spring DI — паттерн внедрения зависимостей в IoC-контейнер  
- Управление объектами через **Bean Factory** или **ApplicationContext**  
- Автоматическая инициализация и связывание компонентов по типу/аннотациям (@Autowired, @Inject)  
- Поддержка конструктора, сеттеров и полей для инъекций  
- Позволяет создавать **loosely coupled**, тестируемый и расширяемый код  
- Использовал для конфигурации сервисов и DAO, уменьшения жёстких связей между слоями  
- Применял для упрощения мокирования в unit-тестах и гибкого переключения реализаций  

Так Spring DI обеспечивает контроль над зависимостями и упрощает поддержку кода.

# Подробный ответ
## Основной ответ
**Dependency Injection (DI)** в Spring — это механизм, позволяющий автоматически внедрять зависимости (объекты, компоненты) в классы без явного их создания. Spring контейнер управляет жизненным циклом и связями между объектами, что способствует слабой связанности и улучшает тестируемость кода. DI реализован в Spring через Inversion of Control (IoC), где контроль создания объектов передается контейнеру.

## Ключевые моменты
- **Виды внедрения**: конструкторное, через сеттеры и поля с использованием аннотаций `@Autowired`, `@Inject` (JSR-330), либо через конфигурации `@Bean` в `@Configuration` классах.
- Spring сканирует компоненты (`@Component`, `@Service`, `@Repository`) и создает бин-объекты, автоматически связывая их по типу или по имени (с помощью аннотаций `@Qualifier`).
- DI позволяет легко заменять реализации интерфейсов, например, для mock-объектов в unit-тестах, что повышает модульность и облегчает сопровождение кода.

## Практический контекст
В проектах на Spring 5+ обычно использую **конструкторное внедрение** — оно гарантирует создание полностью инициализированного объекта и проще для unit-тестирования с Mockito. Также активно применял `@Qualifier` для выбора конкретных реализаций в тех случаях, где было несколько кандидатов. Благодаря DI уменьшил связанность компонентов и упростил конфигурацию приложения, сосредоточив бизнес-логику в сервисах без жесткой зависимости от инфраструктуры.
