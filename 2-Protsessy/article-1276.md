# Как работают некластеризованные индексы?

**Как работают некластеризованные индексы?**  
- База данных: индекс для поиска в таблице  
- Содержит **отдельную структуру** (например, B-дерево)  
- Хранит ключи и указатели на строки данных (RID или PK)  
- Не меняет физический порядок строк таблицы  
- Для запроса сначала ищет ключ в индексе, затем читает данные по указателю  
- Ускоряет выборки по неключевым колонкам или с несколькими условиями  
- Минус: дополнительное место и накладные расходы на обновление  
Используются для **ускорения запросов без перестройки таблицы**.

# Подробный ответ
## Основной ответ
**Некластеризованный индекс** — это структура данных, которая хранит в себе отдельную копию столбцов индекса с указателями на соответствующие строки в основной таблице, но в отличие от кластеризованного индекса, порядок данных в таблице не изменяется. Он работает почти как отдельная таблица с ключами и ссылками на физические или логические адреса строк основного хранилища.

Когда выполняется запрос с использованием некластеризованного индекса, СУБД сначала ищет значение в индексе, быстро находя нужный ключ, а затем по ссылке получает соответствующую строку из таблицы. Таким образом, сам индекс ускоряет поиск, но не определяет порядок хранения данных на диске.

## Ключевые моменты  
- **Структура**: обычно реализуются в виде B-дерева (B-tree), где внутренние узлы содержат ключи для навигации, а листовые узлы содержат значения ключей и указатели на строки данных.  
- **Дополнительное хранилище**: индекс занимает дополнительное место, так как хранит копии ключей и ссылки. Это ускоряет чтение, но может замедлить операции вставки, обновления и удаления из-за необходимости поддерживать индекс в актуальном состоянии.  
- **Использование**: применяются для ускорения поиска по неключевым столбцам или когда нужна быстрая фильтрация, но при этом физический порядок таблицы важен для других оптимизаций. В PostgreSQL 14+ и SQL Server 2019 стандартный подход именно некластеризованного индекса.

## Практический контекст  
В реальных проектах часто создают некластеризованные индексы на колонках, участвующих в условиях WHERE и JOIN, чтобы снизить latency запросов до 10–20 мс вместо полного сканирования. Например, для колонки email в таблице пользователей. Однако стоит внимательно выбирать индексируемые поля, чтобы не ухудшать производительность записей.
