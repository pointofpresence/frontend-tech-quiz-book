# Чем отличается абстрактный класс от интерфейса?

Категория: Процессы

### Краткий ответ

**Отличия абстрактного класса и интерфейса**

- ООП, наследование и контракт  
- **Абстрактный класс** — частично реализованный тип  
- Интерфейс — полный **контракт без реализации**  
- Абстрактный класс может содержать **реализацию методов и поля**  
- Интерфейс только объявляет методы, **без поля и реализации** (в классическом понимании)  
- Класс может наследовать **только один абстрактный класс**, но реализовывать **много интерфейсов**  
- Интерфейсы лучше для **множественного наследования поведения**, абстрактные классы — для **общей функциональности**  
- Практика: абстрактные классы используют, когда нужна **базовая реализация**, интерфейсы — для **стандартизации API** и **гибкости**  

Итог: абстрактный класс — это **базовый класс с реализацией**, интерфейс — **чистый набор требований** без реализации.

### Подробный ответ
#### Основной ответ
**Абстрактный класс** и **интерфейс** — это два разных механизма для определения контракта и общей логики в объектно-ориентированном программировании, но они имеют принципиальные отличия. Абстрактный класс может содержать как абстрактные методы (без реализации), так и конкретные методы с кодом, поля (переменные состояния), а интерфейс в классических версиях Java или C# содержит только объявления методов без реализации (хотя в современных версиях появились default-методы с реализацией). Интерфейс служит для строго определения набора методов, которые класс должен реализовать, обеспечивая контракт без состояния и реализации.

#### Ключевые моменты
- **Множественное наследование**: Класс может реализовывать множество интерфейсов, что позволяет гибко комбинировать функционал, тогда как он может наследоваться только от одного абстрактного класса.
- **Состояние и реализация**: Абстрактный класс может содержать поля и реализованные методы, задавая общую логику для всех наследников, интерфейс традиционно не содержит состояния и обычно лишь описывает API.
- **Семантика и применение**: Абстрактный класс хорош для моделирования "является видом" (is-a) с частичной реализацией, а интерфейс — для определения "способности" (capability), например, Serializable, Comparable.

#### Практический контекст
В реальных проектах в Java 8+ часто смешивают эти возможности: интерфейсы с default-методами позволяют избежать жестких иерархий абстрактных классов, а абстрактные классы остаются удобными для совместного использования данных или общей логики. В C# начиная с версии 8 аналогично появились реализации по умолчанию в интерфейсах. Выбор зависит от задачи: если нужна обобщенная реализация с состоянием — абстрактный класс, если нужно лишь гарантировать наличие методов — интерфейс.
