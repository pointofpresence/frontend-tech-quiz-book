# Как проектировать gRPC API так, чтобы оно было backward compatible при эволюции схемы?

Категория: Процессы

### Краткий ответ

**Как проектировать gRPC API для backward compatibility**

- gRPC API — определение через **Protocol Buffers (proto)**
- **НЕ удалять старые поля** — только добавлять новые с уникальными номерами
- используйте **optional** и **repeated** поля для расширяемости
- при изменении типов избегать несовместимых изменений (например, смена int на string)
- расширяйте API через **новые методы** или версии сервисов (например, v1, v2)
- сохранять недефолтное поведение старых клиентов (не ломать логику)
- тестировать совместимость через **integration tests** с разными версиями клиентов

Итог: проектирование через продуманное расширение протоколов и аккуратную миграцию для сохранения совместимости с существующими клиентами.

### Подробный ответ
#### Основной ответ
Проектирование **backward compatible gRPC API** — это важный аспект, который позволяет развивать сервис без нарушения работы клиентов, использующих старые версии. В основе совместимости лежит аккуратное управление **protobuf-схемой** и её изменениями с учётом принципов совместимости.

#### Ключевые моменты
- **Не удалять и не переиспользовать номера полей** в protobuf-сообщениях. Удаление поля следует заменить его пометкой `deprecated = true`. Это предотвращает конфликт при десериализации у старых клиентов.
- **Добавлять новые поля только с уникальными номерами** и использовать значения по умолчанию, чтобы старые клиенты могли игнорировать новые поля без ошибок.
- **Изменять типы полей и их порядок нельзя**, т.к. это ломает бинарный формат. Можно добавлять новые сервисные методы, но не менять сигнатуру уже существующих.
- При изменении enum-типов нельзя удалять значения, а только расширять, чтобы избежать неправильной интерпретации.
- **Использовать versioning API через новые методы или сервисы**, если изменения архитектурные и нарушают совместимость.
- Следить за правилами protobuf (например, не использовать `required` поля, только `optional` и `repeated`), так как `required` поля жёстко ломают backward compatibility.
- **Тестировать backward compatibility с помощью инструментов**: например, grpc-proto-compatibility-checker, чтобы предотвратить нарушение.

#### Практический контекст
В реальных проектах, например в микросервисных архитектурах с gRPC и Protobuf 3+, именно такой подход позволяет разрабатывать сервисы с минимальным временем простоя и позволяет клиентам постепенно мигрировать на новую версию API. Дополнительно стоит документировать изменения и информировать команды, чтобы управление версиями было прозрачным.
