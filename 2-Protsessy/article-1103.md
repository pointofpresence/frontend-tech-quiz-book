# Какие существуют типы вызовов в gRPC?

Категория: Процессы

### Краткий ответ

**Типы вызовов в gRPC**

- gRPC — фреймворк для **удалённых процедурных вызовов**
- Unary RPC: один запрос → один ответ
- Server streaming RPC: один запрос → поток ответов
- Client streaming RPC: поток запросов → один ответ
- Bidirectional streaming RPC: двунаправленный поток запросов и ответов
- Используются HTTP/2 для эффективной передачи данных и мультиплексирования
- Подход подходит для микросервисов и реального времени
- Типы вызовов влияют на архитектуру и стратегию обработки сообщений

### Подробный ответ
#### Основной ответ
В **gRPC** существует четыре основных типа вызовов (RPC), которые поддерживают разные сценарии взаимодействия между клиентом и сервером. Эти типы различаются по направлению и количеству сообщений, которые отправляются и принимаются в рамках одного вызова.

#### Ключевые типы вызовов в gRPC
- **Unary (одноразовый вызов)**: клиент отправляет один запрос, сервер возвращает один ответ. Это самый простой и часто используемый тип вызова, похожий на классический запрос-ответ.
- **Server streaming**: клиент посылает один запрос, а сервер в ответ отправляет поток из нескольких сообщений. Клиент получает данные постепенно, что удобно для передачи больших данных или последовательностей.
- **Client streaming**: клиент отправляет поток сообщений в сервер, сервер обрабатывает весь поток и после окончания передачи возвращает одно итоговое сообщение. Полезно, когда нужно собрать данные с клиента перед ответом.
- **Bidirectional streaming (двунаправленный поток)**: и клиент, и сервер обмениваются потоками сообщений одновременно, что создаёт двунаправленное общение. Используется для real-time коммуникаций, например, чат или live-стриминг.

#### Практический контекст
В React 18+, или при серверной разработке на Go или Java, выбор типа вызова зависит от сценария. **Unary** хорошо подходит для синхронных операций, а **streaming** для обработки больших данных или real-time обмена. Для потоковых вызовов в gRPC применяется HTTP/2, что обеспечивает низкую задержку (~10-50ms) и эффективное использование ресурсов.

Таким образом, понимание этих типов позволяет правильно архитектурировать сервисы под нужды приложения, балансируя между сложностью реализации и производительностью.
