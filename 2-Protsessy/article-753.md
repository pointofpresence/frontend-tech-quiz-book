# Какие типы считаются стабильными (stable) в Compose? Почему List — нестабильный?

**Стабильные типы в Compose. Почему List нестабилен?**

- **Stable (стабильный) тип** — гарантирует, что его состояние не меняется без явного уведомления Compose  
- Обычно: **примитивы, data class с val**, immutable коллекции, классы с правильно реализованными equals/hashCode  
- Compose использует стабильность для **оптимизации** — пропускает перерисовку, если данные не изменились  
- **List** нестабилен, потому что сам по себе — **мутабельный интерфейс**, изменяемую коллекцию нельзя считать неизменной  
- Изменения внутри List Compose не отслеживает, нет гарантий immutability и корректных equals/hashCode  
- Для стабильности списков используют **ImmutableList** или копии списков с val и тщательным управлением изменениями  
- Итог: стабильность важна для **эффективного перерисовывания UI** и предотвращения лишних recompositions в Compose

# Подробный ответ
## Основной ответ
В **Jetpack Compose** типы считаются **стабильными (stable)**, если их значения не меняют состояние без уведомления системы и при этом система может эффективно отслеживать изменения между рекомпозициями. Стабильность важна для оптимизации — compose избегает лишних перерисовок, если понимает, что значение не изменилось. 

Конкретно, **stable типами считаются все базовые примитивные типы (Int, Boolean, Float), строки (String), а также любые иммутабельные объекты, помеченные аннотацией `@Stable` или реализующие интерфейс `Stable`.**

**List** же в Compose обычно нестабильный, поскольку это **мутабельный и изменяемый по содержимому тип** — при изменении элемента внутри списка ссылка на сам список может остаться той же, но содержимое изменится. Compose не может автоматически понять, произошло ли реальное изменение, если просто сравнивать ссылки. Из-за этого списки считаются нестабильным типом, что ведёт к пересозданию UI при каждой рекомпозиции, если не предпринять дополнительных мер (например, использовать `SnapshotStateList` или иммутабельные списки).

## Ключевые моменты
- **Стабильные типы** — иммутабельны или имеют детерминированные изменения, Compose может эффективно сравнивать их по значению и избежать лишней рекомпозиции.
- **List** по умолчанию — mutable, изменение его содержимого не всегда сопровождается сменой ссылки, поэтому Compose не распознаёт изменения автоматически.
- Чтобы сделать списки стабильными, часто используют **immutable списки из Kotlin (например, List<T>)** или оборачивают мутабельные через `SnapshotStateList`, что позволяет Compose отслеживать изменения элементов.
- Аннотация `@Stable` и интерфейс `Stable` — инструменты, помогающие разработчикам явно указать Compose, что объект стабильный, даже если это не очевидно из базовых правил.

## Практический контекст
В реальных приложениях для повышения производительности UI с динамическими списками обычно используют `SnapshotStateList` или immutable коллекции (например, из Kotlinx.collections.immutable). Это позволяет Compose оптимально определять, когда нужна перерисовка, и снижает нагрузку на систему рендеринга. Additionally, обёртывание изменений коллекций в `mutableStateOf` или специальных state-контейнерах помогает эффективно работать с состоянием в Compose.
