# Как работает сборщик мусора в Java?

**Как работает сборщик мусора в Java?**

- Управляет автоматическим освобождением памяти JVM
- Отслеживает объекты без ссылок (недостижимые)
- Делит кучу на поколения: Young, Old, и Permanent (метаданные)
- Основной алгоритм: **mark-and-sweep** (пометка и очистка)
- Young gen — быстрое выделение и сборка (Minor GC)
- Old gen — реже, сложнее (Full GC), требует компактации
- Использует разные алгоритмы: Serial, Parallel, CMS, G1 для оптимизации задержек
- Обеспечивает безопасность через stop-the-world паузы
- Упрощает управление памятью, снижает утечки и ошибки памяти
- Применяется для поддержания производительности и устойчивости приложений Java

# Подробный ответ
## Основной ответ
**Сборщик мусора (Garbage Collector, GC)** в Java — это механизм автоматического управления памятью, который освобождает неиспользуемые объекты в куче, предотвращая утечки памяти и снижая количество ошибок, связанных с ручным управлением памятью. GC работает на уровне JVM и регулярно проверяет, какие объекты больше не достижимы из программы, чтобы затем освободить под них память.

## Ключевые моменты
- **Heap структурирован на поколения**: Young Generation (молодое поколение), Old Generation (старое поколение), а с Java 8+ ещё Metaspace для метаданных. Молодое поколение делится на Eden и два Survivor пространства, что ускоряет очистку недавно созданных объектов.
- **Алгоритмы GC**: типично используется **Generational GC** с двумя важными техниками: **Minor GC** для молодого поколения (частая, быстрая сборка) и **Full (Major) GC** для старого поколения (редкая, но более дорогая). В современных JVM (Java 11+) популярны версии GC — G1 (Garbage-First), ZGC и Shenandoah, оптимизированные для низкой паузы и работы на многопроцессорных машинах.
- **Корневая достижимость (Reachability)**: объекты считаются живыми, если доступны из корневых точек — переменные стека, статические поля, JNI. Недостижимые объекты собираются.
- **Stop-The-World паузы**: GC останавливает выполнение приложения на время очистки, но современные сборщики минимизируют время таких пауз, что критично для latency-sensitive приложений.

## Практический контекст
В проектах с высокой нагрузкой типично выбирают **G1 GC** (включен по умолчанию в Java 11+), который автоматически балансирует между throughput и latency. Для приложений с крайне низкой задержкой актуальны ZGC и Shenandoah (Java 15+), которые обеспечивают паузы в миллисекунды при огромных heap-ах (тестировалось на heap >100GB). Правильная настройка параметров JVM и понимание жизненного цикла объектов помогает оптимизировать использование памяти и снизить накладные расходы GC.
