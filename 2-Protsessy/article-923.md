# Какие способы авторизации применяются при работе с API (например, Bearer Token, Basic Auth)?

**Способы авторизации в API**

- **Basic Auth**: передача логина и пароля в заголовке в Base64, простой, но небезопасный без HTTPS  
- **Bearer Token**: токен доступа (чаще JWT), передается в заголовке Authorization, без пароля, более безопасно  
- **API Key**: уникальный ключ, часто в заголовке или параметрах URL, прост в использовании, но уязвим к перехвату  
- **OAuth 2.0**: протокол авторизации с токенами доступа и обновления, поддерживает делегирование прав и раздельный доступ  
- **Digest Auth**: хеширование учетных данных для защиты, сложнее Basic Auth, не так часто используется  
- **HMAC**: подпись запросов с секретным ключом, обеспечивает целостность и аутентичность данных  
- Важен выбор способа с учетом **безопасности**, **простоты реализации** и **требований проекта**

# Подробный ответ
## Основной ответ
Для авторизации при работе с API чаще всего применяются несколько стандартных методов, среди которых **Bearer Token** и **Basic Auth** — одни из самых распространённых. Основная задача авторизации — подтвердить личность клиента, чтобы сервер мог предоставить доступ к нужным ресурсам и защитить API от несанкционированного использования.

## Ключевые моменты
- **Basic Authentication** — самый простой метод, где клиент передает логин и пароль в заголовке HTTP, закодированные в Base64. Это удобно для быстрых тестов, но небезопасно без использования HTTPS из-за простой уязвимости к перехвату данных.
- **Bearer Token** — в заголовке `Authorization` передается токен доступа (например, JWT или OAuth 2.0 access token). Такой подход позволяет отделить аутентификацию от авторизации, управлять сроком жизни токена и масштабировать безопасность.
- Кроме них часто используют **OAuth 2.0** — протокол авторизации с несколькими потоками (grant types), поддерживающий делегирование доступа, refresh tokens и scopes.
- Еще есть **API Key**, когда клиент получает уникальный ключ, передаваемый в заголовке или параметрах запроса.
- В современных системах популярны **JWT-токены** (JSON Web Tokens), которые позволяют держать закодированную информацию о пользователе прямо в самом токене, что ускоряет валидацию и уменьшает нагрузку на сервер.

## Практический контекст
В реальных проектах для внутренних API часто хватает Basic Auth в сочетании с HTTPS, но для публичных сервисов предпочтительнее Bearer Token в формате OAuth 2.0 с JWT — это позволяет гибко управлять правами доступа, поддерживать автоматическое обновление токенов и масштабировать систему с высокой безопасностью. Например, в REST API сервисы Google и GitHub используют OAuth 2.0 с Bearer Token.
