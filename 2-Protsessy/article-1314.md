# Чем отличаются многопоточность, асинхронность и мультипроцессинг в Python? В чём сильные и слабые стороны каждого подхода?

Категория: Процессы

### Краткий ответ

**Многопоточность, асинхронность и мультипроцессинг в Python**

- **Многопоточность:** параллельное выполнение потоков внутри одного процесса.
  - Потоки разделяют память, лёгкие по ресурсам.
  - Ограничена GIL — одновременное выполнение байткода только в одном потоке.
  - Хороша для IO-bound задач, но слабая для CPU-bound.

- **Асинхронность:** кооперативное многозадачное выполнение на одном потоке через event loop.
  - Использует async/await, не блокирует поток при ожидании IO.
  - Минимальные накладки, высокий масштаб.
  - Подходит для масштабируемого IO-bound кода, менее эффективна для CPU-bound.

- **Мультипроцессинг:** запуск нескольких независимых процессов с собственной памятью.
  - Обходит GIL, поэтому эффективен для CPU-bound.
  - Более ресурсозатратен (память, время запуска).
  - Сложнее обмен данными и синхронизация.

**Сильные стороны:**
- Многопоточность — простота и легковесность для IO.
- Асинхронность — высокая масштабируемость и эффективность при ожидании IO.
- Мультипроцессинг — реальный параллелизм для CPU-задач.

**Слабые стороны:**
- Многопоточность — GIL ограничивает CPU-параллелизм.
- Асинхронность — сложность архитектуры, не подходит для тяжелых вычислений.
- Мультипроцессинг — накладные расходы, сложность коммуникации.

**Итог:** выбирают многопоточность и асинхронность для IO-bound, мультипроцессинг — для CPU-bound задач.

### Подробный ответ
#### Основной ответ
В Python **многопоточность**, **асинхронность** и **мультипроцессинг** — это разные подходы к запуску параллельных или конкурентных задач, которые отличаются по организации выполнения кода и применимым ограничениям. Многопоточность использует несколько потоков внутри одного процесса и подходит для операций, связанных с ожиданием (I/O), но ограничена GIL (Global Interpreter Lock). Асинхронность строится на event loop и позволяет эффективно выполнять большое количество операций ввода-вывода без необходимости заводить много потоков. Мультипроцессинг запускает несколько процессов с отдельной памятью, обходя GIL и эффективно используя многоядерность CPU.

#### Ключевые моменты
- **Многопоточность (threading)** в Python страдает от GIL, который не позволяет выполнять байткод Python в нескольких потоках одновременно, поэтому подходит в первую очередь для задач с блокирующим I/O (сети, файлы). Потоки легковесны, но совместное использование памяти требует синхронизации (Lock, Semaphore).
- **Асинхронность (asyncio)** — это модель с единственным потоком, управляющая задачами через event loop и корутины. Позволяет масштабируемо обрабатывать миллионы сетевых соединений с минимальными накладными расходами. Не подходит для CPU-bound задач, требует иной стиль написания кода.
- **Мультипроцессинг (multiprocessing)** создаёт отдельные процессы с собственной памятью — эффективно для CPU-bound задач, поскольку работает параллельно на нескольких ядрах без ограничений GIL. Но межпроцессное взаимодействие сложнее и дороже из-за сериализации и IPC.

#### Практический контекст
- Многопоточность часто используется с библиотеками для работы с сетью или файловой системой (например, в веб-краулерах с requests + threading).
- Асинхронность популярна в современном Python 3.7+ для высоконагруженных веб-серверов, например, с библиотеками **aiohttp** и frameworks на **asyncio**.
- Мультипроцессинг применяют в дата-науке, ML обучении, где многопоток не даёт прироста из-за GIL — например, с **multiprocessing.Pool** для параллельной обработки данных.

Таким образом, выбор подхода зависит от типа задачи:  
- **I/O-bound**: многопоточность или асинхронность (асинхронность масштабируемее и эффективнее).  
- **CPU-bound**: мультипроцессинг.  

Этот баланс помогает строить производительные и отзывчивые приложения на Python.
