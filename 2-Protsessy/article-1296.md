# Чем отличается изменение состояния в классовых и функциональных компонентах?

Категория: Процессы

### Краткий ответ

**Чем отличается изменение состояния в классовых и функциональных компонентах?**

- React state management  
- В классовых: состояние хранится в this.state  
- Изменение через this.setState (асинхронно, объединяет обновления)  
- В функциональных: состояние с помощью useState hook  
- setState из useState перезаписывает состояние полностью  
- useState обновляет состояние независимо, не объединяя  
- Функциональные компоненты позволяют использовать несколько хуков для разных состояний  
- Классовые — один объект состояния, функциональные — независимые переменные  
- Практика: функциональные компоненты проще для оптимизации и читаемости  

Если вкратце:  
**Классовые** — единый объект, обновления объединяются через setState;  
**Функциональные** — отдельные состояния через хуки, обновления независимы и перезаписывают.

### Подробный ответ
#### Основной ответ
Изменение состояния в **классовых** и **функциональных компонентах** React принципиально отличается из-за их архитектурных особенностей и используемых API. В классовых компонентах состояние хранится в объекте `this.state`, и для его обновления используется метод `this.setState()`, который объединяет новое состояние с текущим и выполняет частичный ререндер. В функциональных компонентах, появившихся с React 16.8 и хуком `useState`, состояние хранится внутри замыканий, а обновление происходит вызовом функции-установщика, возвращаемой хуком — она полностью заменяет предыдущее состояние.

#### Ключевые моменты
- **Механизм обновления**: В классовых компонентах `setState` асинхронен, агрегирует обновления и сливает объекты (shallow merge). В функциональных компонентах `setState` от `useState` полностью заменяет значение и работает с замыканиями, не смешивая объекты автоматически.
- **Императивность vs декларативность**: Классы традиционно более императивны — обновление состояний и жизненный цикл связан с методами. Хуки позволяют декларативно управлять состоянием, благодаря функциональному стилю и чистым функциям.
- **Работа с предыдущим состоянием**: В функциональных компонентах обновления бывают асинхронными, поэтому чаще рекомендуют использовать функцию обновления `setState(prev => newValue)` для гарантии корректного вычисления нового состояния. В классах для этого также есть callback в `setState`.
- **Поведение и производительность**: Функциональные компоненты с хуками также лучше работают с побочными эффектами (хуки `useEffect`), что улучшает контроль жизненного цикла и позволяет избежать типичных проблем классовых компонентов с наследованием и `this`.

#### Практический контекст
Сегодня в React 18+ функциональные компоненты с хуками — стандарт индустрии, так как они проще тестируются, легче рефакторятся и дают более прозрачное управление состоянием и эффектами. Классовые компоненты продолжают поддерживаться и встречаются в legacy-коде, но новые проекты рекомендуется строить на функциях и `useState`, `useReducer` для сложного состояния.
