# Когда вы используете git revert?

Категория: Процессы

### Краткий ответ

**Когда использовать git revert?**  
- **Инструмент отмены коммитов** без удаления истории  
- Создаёт новый коммит, отменяющий изменения выбранного  
- Безопасен для **публичных веток** и коллаборации  
- Не нарушает общую историю, в отличие от git reset  
- Используется для исправления ошибок в опубликованном коде  
- Поддерживает прозрачность и аудит изменений в проекте  
- Практика в командах с **CI/CD** и строгим контролем версий

### Подробный ответ
#### Основной ответ
**Git revert** используется для отмены изменений, которые уже были зафиксированы в репозитории, создавая новый коммит, который «отменяет» эффекты указанного коммита. Это безопасный способ «откатить» изменения в публичной ветке, не переписывая историю, в отличие от `git reset`.

#### Ключевые моменты
- **Безопасность истории**: `git revert` не изменяет существующую историю, а добавляет новый коммит с обратными изменениями. Это критично при работе в команде и совместном использовании веток (например, в `main` или `master`).
- **Используется, когда изменения уже опубликованы**: если коммит уже отправлен в удалённый репозиторий, `git revert` лучше, чем `git reset`, который требует force push и может привести к конфликтам в общей истории.
- **Гранулярное управление**: можно отменить один конкретный коммит, сохранив все остальные, и, если нужно, повторно применять изменения через новые коммиты.

#### Практический контекст
Например, если в продакшен попал баг из-за конкретного коммита, мы используем `git revert <hash>`, чтобы быстро создать исправление, при этом не ломая ветку или историю. В React 18 проектах с CI/CD это типично, когда нельзя позволять force push на главные ветки из-за требований стабильности и аудита.
