# Что происходит при записи в небуферизированный канал из нескольких горутин (блокировки, паника)?

**Что происходит при записи в небуферизированный канал из нескольких горутин (блокировки, паника)?**

- небуферизированный канал — синхронная коммуникация между горутинами  
- при записи (send) горутина блокируется, пока другая не примет значение  
- несколько горутин одновременно записать в канал — race condition без гарантии порядка  
- если нет принимающей горутины, запись блокируется навсегда (deadlock)  
- падение с **паникой** возникает при закрытом канале или невозможности прогресса  
- прием одновременно из нескольких горутин работает по аналогии — блокируется до передачи  
- для избежания deadlock и паники — требуется синхронизация или буферизация канала  

**Итог:** запись в небуферизированный канал блокирует горутину до получения, при отсутствии приёмника — deadlock; несколько отправителей конкурируют на запись, возможна блокировка или panic при закрытии.

# Подробный ответ
## Основной ответ
При записи в **небуферизированный канал (unbuffered channel)** из нескольких горутин, каждая операция отправки (`chan <- value`) блокируется до тех пор, пока другая горутина не выполнит операцию получения (`<-chan`). Это синхронизация "рука об руку" — передача значения возможна только при синхронном присутствии получателя. Если несколько горутин пытаются записать одновременно, они будут блокироваться, ожидая соответствующих приёмников, но не возникает паники или внутренняя гонка, потому что каналы в Go являются потокобезопасными.

## Ключевые моменты
- **Блокировка при записи**: отправляющая горутина блокируется до момента, когда другая горутина выполнит чтение (если канал небуферизированный, буфер отсутствует).
- **Отсутствие паники при одновременных записях**: каналы реализованы с внутренней синхронизацией, они безопасны для конкурентного доступа из нескольких горутин.
- **Порядок передачи и приемников**: каждый `send` ждет своего `receive`, поэтому если приёмников меньше, горутины записи будут жёстко блокироваться.
- **Замечание по закрытию**: если канал закрыть и попытаться записать, будет паника, независимо от количества горутин; но именно одновременная запись в открытый канал безопасна.

## Практический контекст
В реальных проектах при работе с небуферизированными каналами используют этот механизм для синхронизации без необходимости дополнительных мьютексов — например, для передачи сигналов или работы в стиле pipeline. Если необходима асинхронность или буферизация, берут буферизированные каналы, но с ними появляются нюансы с переполнением буфера и потенциальной блокировкой.
