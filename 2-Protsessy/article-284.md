# Как обеспечивали инвалидацию кэша?

**Как обеспечивали инвалидацию кэша?**

- Кэш — **хранилище промежуточных данных** для ускорения доступа  
- Инвалидация важна для **согласованности данных**  
- Стратегии инвалидации:  
  - **TTL (время жизни)**: автоматическая очистка по истечении времени  
  - **Ручная очистка** по событию изменения данных  
  - **Write-through / Write-back**: обновление кэша одновременно с БД  
  - Использование **event-driven** триггеров для синхронизации  
- Выбор зависит от требований по **свежести данных** и нагрузке  
- Практика: комбинированный подход с TTL и точечной ручной инвалидацией  
- Инвалидация минимизирует **конфликты и устаревшие данные**  
- Важно мониторить и тестировать, чтобы избежать **протечек памяти** и **неактуальности**  

Этот подход гарантирует баланс между **производительностью** и **актуальностью данных**.

# Подробный ответ
## Основной ответ
Инвалидация кэша — это процесс обновления или удаления устаревших данных из кэша, чтобы обеспечить консистентность между кэшем и источником правды (например, базой данных). Обеспечение правильной инвалидации критично для сохранения актуальности данных и предотвращения использования «старых» значений.

## Ключевые моменты
- **TTL (time-to-live)** — самый распространённый базовый метод, где каждому кэш-объекту задаётся срок жизни. По истечении TTL объект удаляется или актуализируется. Это простой и эффективный способ, но с ограничением — возможна небольшая прослойка устаревших данных.
- **Инвалидация при записи (Write-through / Write-back)** — обновление/удаление кэша сразу при изменении данных в БД. В Write-through данные сначала записываются в кэш, потом в БД, в Write-back наоборот. Этот метод более ресурсоёмкий, но гарантирует актуальность кэша.
- **Event-driven инвалидация** через публикацию событий (например, в Kafka или Redis Pub/Sub), на которые подписаны кэш-серверы. При изменении данных генерируются события, инициирующие удаление или обновление соответствующих ключей кэша.
- **Групповая инвалидация и версионирование ключей** — при изменении данных меняется версия ключа, и устаревший кэш становится недоступным без явного удаления, что позволяет избежать проблем с race condition.

## Практический контекст
В реальных проектах часто комбинируют TTL и event-driven подход. Например, в React-сервисах (Node.js + Redis) для user sessions ставят небольшой TTL (~30 минут), а при обновлении пользовательских данных отправляют событие в Kafka, на которое подписан кэш-сервис, обеспечивая быструю инвалидацию именно изменённых данных. Такой подход обеспечивает баланс между производительностью и консистентностью данных.
