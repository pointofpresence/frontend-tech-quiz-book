# Почему Dispatchers.Default не подходит для сетевых запросов/работы с БД?

Категория: Процессы

### Краткий ответ

**Почему Dispatchers.Default не подходит для сетевых запросов/работы с БД?**  
- Dispatchers.Default — пул потоков для CPU-нагрузки  
- Оптимизирован для вычислительно тяжелых задач  
- Неэффективен при блокирующих операциях (I/O)  
- Сетевые/БД запросы блокируют поток → снижают производительность  
- Риск истощения пула потоков → задержки и дедлоки  
- Для I/O-операций лучше использовать Dispatchers.IO  
- Dispatchers.IO создаёт расширяемый пул потоков под блокировки  
- Использование Dispatchers.IO обеспечивает стабильность и масштабируемость  

Итого: **Dispatchers.Default тормозит при I/O из-за блокировки CPU-потоков, для таких задач нужен Dispatchers.IO**.

### Подробный ответ
#### Основной ответ
**Dispatchers.Default** в Kotlin Coroutines оптимизирован для CPU-интенсивных задач — параллельных вычислений и обработки данных, используя общий пул потоков размером примерно равным количеству ядер CPU. Для сетевых запросов и работы с базой данных, которые являются **blocking I/O операциями**, этот диспетчер не подходит, поскольку он не предназначен для долгих блокировок, что может привести к исчерпанию пулa потоков и ухудшению производительности.

#### Ключевые моменты
- **Blocking vs Non-blocking**: Dispatchers.Default эффективно работает с неблокирующими задачами, а сетевые запросы/БД-запросы часто блокируют поток, ожидая ответа, что снижает пропускную способность.
- **Риск starvation**: Если блокирующие операции занимают потоки из Dispatchers.Default, задача параллельных вычислений страдает из-за нехватки свободных потоков.
- Для таких задач правильно использовать **Dispatchers.IO**, который использует отдельный расширяемый пул потоков с большим количеством потоков специально для blocking I/O, минимизируя блокировки CPU-нагруженных потоков.
- В React 18+ и аналогичных системах также вынос блокирующих операций в отдельный пул — общепринятая практика.

#### Практический контекст
В реальных проектах для асинхронных сетевых или database запросов обычно назначают `withContext(Dispatchers.IO)`, чтобы гарантировать, что длинные блокировки вызывающих потоков не ухудшат отзывчивость UI или производительность вычислительных задач. Например, в Android приложениях запросы к Retrofit/Room выполняют именно в `Dispatchers.IO`, а тяжелые расчеты — в `Dispatchers.Default`.
