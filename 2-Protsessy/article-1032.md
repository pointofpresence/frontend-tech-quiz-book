# Какие HTTP-методы (GET, POST, PUT, PATCH, DELETE) используются для CRUD-операций?

**HTTP-методы для CRUD-операций**  
- CRUD: Create, Read, Update, Delete  
- **GET** — чтение ресурсов (Read), безопасный, идемпотентный  
- **POST** — создание ресурсов (Create), не идемпотентный  
- **PUT** — полное обновление ресурса (Update), идемпотентный  
- **PATCH** — частичное обновление ресурса (Update), не обязательно идемпотентный  
- **DELETE** — удаление ресурса (Delete), идемпотентный  
- Используются в REST API для строгого соответствия операциям  
- Обеспечивают семантику и оптимизацию клиент-серверного взаимодействия

# Подробный ответ
## Основной ответ
Для реализации CRUD-операций в RESTful API стандартно используют HTTP-методы, которые соответствуют четырём основным действиям с ресурсами: создание, чтение, обновление и удаление. Вот базовое соответствие:

- **GET** — чтение (Read) данных с сервера, обычно запрос без изменения состояния.
- **POST** — создание (Create) нового ресурса, сервер сам генерирует идентификатор.
- **PUT** — полное обновление (Update) существующего ресурса, замена всех его полей.
- **PATCH** — частичное обновление ресурса, меняются только указанные поля.
- **DELETE** — удаление (Delete) ресурса с сервера.

## Ключевые моменты
- **GET** — метод идемпотентный и безопасный, то есть многократные вызовы не изменяют состояние.
- **POST** — не идемпотентен, при повторных вызовах может создавать дубликаты; используется для создания и запуска операций.
- **PUT** — идемпотентный, обновляет ресурс полностью, поэтому в запросе нужно передавать весь объект.
- **PATCH** — более оптимальный для обновления части данных, часто используется в API с версией HTTP/1.1 и выше.
- **DELETE** — идемпотентен, обычно удаляет ресурс по URI, при повторном вызове возвращает статус, что ресурс уже удалён.

## Практический контекст
В современных API, например, в проектах на **Express.js**, **FastAPI** или **Spring Boot**, соблюдение этого распределения методов помогает клиентам понять логику операций и улучшает кеширование и безопасность. Например, GET-запросы к кэшируемым ресурсам можно легко кешировать на стороне CDN, а POST-запросы обрабатываются более строго из-за возможных изменений данных.
