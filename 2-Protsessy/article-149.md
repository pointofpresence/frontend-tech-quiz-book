# Где хранить кэшированные данные? (LocalStorage, IndexedDB). Что делать, если IndexedDB не поддерживается?

**Где хранить кэшированные данные? LocalStorage, IndexedDB. Что делать, если IndexedDB не поддерживается?**

- **IndexedDB** — асинхронное хранилище, подходит для крупных, структурированных данных  
- **LocalStorage** — синхронное, подходит для простых, небольших данных (<5MB)  
- IndexedDB лучше по производительности и гибкости, поддерживает транзакции и индексы  
- Если IndexedDB не поддерживается — fallback на LocalStorage для критичных простых данных  
- Для больших данных и сложных структур — можно использовать in-memory кеш + fallback  
- Альтернативы: WebSQL (устаревшее), sessionStorage (короткоживущий)  
- Всегда реализовать обработку ошибок и проверку поддержки API через feature detection  
- Обеспечить очистку и актуализацию кэша (TTL, версии данных)  
- Применение: офлайн данные, ускорение загрузки, уменьшение запросов к серверу  

**Итог:** IndexedDB — основной выбор для кеша, LocalStorage — резервный, с адаптивной стратегией fallback.

# Подробный ответ
## Основной ответ
Для хранения кэшированных данных в браузере чаще всего используют **IndexedDB** и **LocalStorage**. IndexedDB — это асинхронное, мощное NoSQL хранилище, подходящее для больших объёмов и сложных структур данных, а LocalStorage — синхронное key-value хранилище с ограничением по размеру (~5-10 МБ), идеальное для небольших и простых данных.

Если IndexedDB не поддерживается (редкая ситуация, но возможна в очень старых браузерах или специфичных окружениях), можно использовать LocalStorage как запасной вариант, учитывая его ограничения по объёму и синхронный характер.

## Ключевые моменты
- **IndexedDB** — основной выбор для кэширования сложных или больших данных (например, результаты API, медиа, offline-first приложения), с асинхронным API, что не блокирует основной поток.
- **LocalStorage** хорош для простого, небольшого кэша (например, флаги, конфигурации), но при больших объёмах и частых операциях становится узким местом из-за синхронного API и ограниченного размера.
- Если IndexedDB отсутствует, применяют fallback стратегии:
  - LocalStorage для меньших данных
  - SessionStorage для данных сессии (если подходит)
  - Память внутри JS (in-memory), хотя при перезагрузке страницы данные пропадут
  - В веб-приложениях Progressive Web App можно использовать Service Worker Cache API как альтернативу для кэша ресурсов.

## Практический контекст
В React 18+ или Vue 3 типично используют IndexedDB через библиотеки, например Dexie.js, чтобы абстрагировать сложности работы с API. Если IndexedDB не доступен, приложение переключается на LocalStorage, уведомляя пользователя о возможных ограничениях. Такой fallback важен для поддержки старых браузеров и embedded webview (например, в мобильных приложениях).
