# Можно ли хранить в ref данные, помимо ссылок на DOM-элементы?

Категория: Процессы

### Краткий ответ

**Можно ли хранить в ref данные, помимо ссылок на DOM-элементы?**

- React `ref` — объект с полем `.current`, для сохранения значений
- Используется не только для DOM, но и любых **мутабельных данных**
- Отличие от state: изменения `ref.current` не вызывают ререндер
- Часто применяют для хранения таймеров, предыдущих значений, кэшей
- Позволяет сохранять данные между рендерами без обновления UI
- Важно избегать состояния, от которого зависит визуализация, в ref
- Практическое применение — асинхронные операции и имплементация нестейт-маркерів

### Подробный ответ
#### Основной ответ
Да, в React можно хранить в **ref** не только ссылки на DOM-элементы, но и любые другие данные. Объект, возвращаемый `useRef()`, сохраняет свое значение между рендерами и не вызывает повторной отрисовки при изменении свойства `.current`. Это делает его удобным для хранения mutable состояний или значений, которые нужно сохранить без триггера обновления компонента.

#### Ключевые моменты
- **Mutable container**: `useRef` возвращает объект с полем `.current`, которое можно изменять напрямую, не вызывая повторных рендеров.
- **Хранение произвольных данных**: можно сохранять счетчики, таймеры, предыдущие значения, флаги, веб-сокеты, любые JS-объекты — фактически все, что нужно сохранить между рендерами, не влияя на UI.
- **Отличие от state**: state вызывает перерисовку компонента, если изменяется, ref — нет. Поэтому ref идеален для случаев, когда изменение данных не должно влиять на визуальное состояние.

#### Практический контекст
Типичный кейс — хранение идентификатора таймера (`setTimeout`/`setInterval`), чтобы потом очистить его в `useEffect`. Также часто ref используется для хранения предыдущих пропсов или стейта, чтобы сравнивать значения между рендерами без дополнительного ререндеринга. В React 18+ этот приём часто используют для оптимизации и избегания лишних отрисовок.
