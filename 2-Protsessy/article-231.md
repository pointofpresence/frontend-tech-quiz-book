# Как вы создавали и использовали кастомные запросы в Hibernate?

Категория: Процессы

### Краткий ответ

**Как создавать и использовать кастомные запросы в Hibernate?**

- Работа с Hibernate — ORM для Java, упрощающий доступ к БД  
- Кастомные запросы нужны для сложной выборки, не покрываемой HQL или Criteria API  
- Использую HQL (Hibernate Query Language): запросы на языке, похожем на SQL, но ориентированном на сущности  
- Пример: session.createQuery("from User u where u.age > :age") с передачей параметров  
- Для нативных запросов — создаю SQL-запрос через session.createNativeQuery(), позволяя использовать специфичные функции БД  
- Для динамического построения — применяю Criteria API, который строит запросы через Java-объекты, с типобезопасностью  
- Кастомные запросы интегрирую с транзакциями и кэшированием Hibernate для оптимизации  
- Практическое применение: фильтрация, агрегации, сложные join, когда стандартные методы ограничены  

Итог: кастомные запросы в Hibernate реализуются через HQL, native SQL и Criteria API, позволяют гибко извлекать данные и оптимизировать производительность.

### Подробный ответ
#### Основной ответ
В Hibernate **кастомные запросы** создаются с помощью нескольких подходов: использование HQL (Hibernate Query Language), native SQL, и Criteria API. Для повышения гибкости часто создаю HQL-запросы, которые позволяют писать запросы к базе данных с объектно-ориентированным синтаксисом. Если требуется использовать функционал, недоступный в HQL, применяю native SQL-запросы. Criteria API полезен для динамической генерации запросов на основе условий во время выполнения.

#### Ключевые моменты
- **HQL** — ориентирован на работу с объектами, поддерживает наследование, связи, позволяет использовать именованные параметры для безопасности и читабельности. Например, запросы с JOIN FETCH для оптимизации загрузки связанных сущностей.
- **Native SQL** — пишу, когда нужны специфичные функции базы данных или оптимизация под конкретный движок (PostgreSQL, MySQL). При этом мапплю результат на сущности Hibernate или DTO с помощью ResultSetMapping.
- **Criteria API** — использую для плавного и типобезопасного создания сложных запросов на основе пользовательского ввода. В Hibernate 5+ значительно улучшена поддержка JPA Criteria с более читаемым API.
- Для повторно используемых запросов применяю **именованные запросы** (NamedQuery, NamedNativeQuery), регистрируемые на уровне сущности или XML-конфигурации — это улучшает производительность за счет препарсинга и упрощает поддержку кода.
  
#### Практический контекст
В одном из проектов для отчетности создавал сложные динамические фильтры клиента, используя Criteria API с join-условиями и кастомными агрегациями. Для специфичных аналитических запросов писал native SQL с использованием CTE и оконных функций, затем маппил результат в кастомные DTO для дальнейшего использования в сервисном слое. Это позволило достичь баланса между удобством Hibernate и производительностью конкретных запросов.
