# Как работают дженерики в Kotlin «под капотом»?

Категория: Процессы

### Краткий ответ

**Как работают дженерики в Kotlin «под капотом»?**

- Типы в Kotlin **универсальные** (generic) на уровне **исходного кода**
- За компиляцию отвечает **JVM с type erasure**
- На JVM: дженерики **стираются** — конкретный параметр типа отсутствует в байткоде
- Для ограничения типов используется **ограничение (upper bound)**, влияющее на проверку во время компиляции
- Для сохранения типа можно использовать **reified** параметры в **inline-функциях**
- Для работы с дженериками применяются **castы**, **ограничения** и рефлексия (RuntimeType)
- Практически: дженерики **безопасны**, типы проверяются во время компиляции, но в рантайме JVM не знает точный тип

_Итого_: Kotlin дженерики реализованы через стирание типов JVM с дополнительными механизмами для сохранения типовой информации в compile-time и inline-функциях с reified.

### Подробный ответ
#### Основной ответ
**Дженерики в Kotlin** реализованы с помощью механизма, называемого **type erasure** (стирание типов), аналогично тому, как это происходит в Java. Это значит, что информация о параметрах типа удаляется на этапе компиляции и в рантайме параметры типов не доступны напрямую. Однако Kotlin дополнительно поддерживает **reified** типы в inline-функциях, что позволяет частично обходить ограничение type erasure.

#### Ключевые моменты
- **Type erasure**: при компиляции все параметры дженериков заменяются на верхний bound (обычно `Any?`), что позволяет запускать один и тот же байткод для разных типов, экономит ресурс, но исключает возможность прямой работы с типом в рантайме.
- **Reified типы и inline-функции**: в Kotlin (с версии 1.1+) с помощью модификатора `inline` и ключевого слова `reified` можно сохранить информацию о типе параметра и использовать её внутри функции (например, для `is`-проверок или создания объектов), что невозможно в обычных дженериках.
- **Типобезопасность**: компилятор Kotlin осуществляет проверку типов во время компиляции и генерирует соответствующий байткод с приведениями типов там, где необходима работа с объектами — это баланс между безопасностью и совместимостью с JVM.

#### Практический контекст
Типично при использовании коллекций, например, `List<String>`, в байткоде JVM это будет просто `List` без указания типа элементов. При этом компилятор Kotlin добавит необходимые `cast`-операции, чтобы сохранить безопасность типов. Использование `inline` + `reified` даёт гибкость для API, требующих работу с реальным типом (например, JSON-сериализация или рефлексия), что часто применяется в проектах с Kotlin 1.3+.
