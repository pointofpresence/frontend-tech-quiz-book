# Как организовать стейт-менеджмент при множественных фильтрах (дата, поиск, статус)?

Категория: Процессы

### Краткий ответ

**Как организовать стейт-менеджмент при множественных фильтрах (дата, поиск, статус)?**

- фронтенд: управление UI-состоянием
- использовать централизованный **store** (Redux, Zustand, Context API)
- хранить фильтры в едином объекте (date, search, status)
- обновлять отдельные фильтры через **иммутабельные** действия
- мемоизация селекторов для оптимизации перерендеров
- синхронизация стейта с URL для перезагрузки и shareable links
- применить debounce для поиска, throttle для частых обновлений
- обеспечивает предсказуемость, масштабируемость и тестируемость UI-фильтрации

### Подробный ответ
#### Основной ответ
При организации **стейт-менеджмента с множественными фильтрами** (дата, поиск, статус) важно обеспечить централизованное, атомарное и предсказуемое управление состоянием фильтров, чтобы избежать рассогласований UI и данных. Обычно для этого используют **единичный объект состояния**, отражающий все текущие параметры фильтрации, либо разбивают их на логические части, но с синхронизацией. В frontend-разработке популярным подходом является использование таких инструментов, как **Redux** (в React), **MobX**, **Zustand** или даже локального стейта с **useReducer** для сложной логики.

#### Ключевые моменты
- **Централизованный shared state**: Все фильтры — дата, поиск, статус — хранятся в одном объекте или в нескольких, но в виде связанных фрагментов, чтобы при изменении одного фильтра не терялась консистентность.
- **Immutable updates и actions**: Используя Redux, state обновляется через actions, что упрощает отладку и позволяет легко логировать изменения фильтров, например, `SET_DATE_FILTER`, `SET_SEARCH_QUERY`.
- **Сброс и комбинаторика**: Важно предусмотреть возможности сброса отдельных фильтров и комбинированное применение с правильным приоритетом. Для этого используют селекторы и мемоизацию (reselect в Redux).
- **Синхронизация с URL**: Чтобы пользователи могли делиться ссылками с текущими фильтрами, фильтрable состояние часто синхронизируют с query-параметрами браузера (например, используя React Router и `useSearchParams`).
  
#### Практический контекст
В реальных проектах, скажем, в React 18+, используют **Redux Toolkit** для упрощения boilerplate, создают slice для фильтров сразу с типами (TypeScript), отдельно реализуют thunk или saga для асинхронных запросов с фильтрами. Такой подход позволяет масштабировать приложение и упрощает тестирование логики через unit-тесты стейта и reducers. Если проект небольшой, можно применить локальный state + useReducer с дебаунсом для поиска, чтобы минимизировать перерендеры.
