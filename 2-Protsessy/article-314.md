# Worker threads - профиты. Что происходит при передачи данных в worker thread. Хорошо ли ведут себя на I/O операциях

Категория: Процессы

### Краткий ответ

**Worker threads — преимущества, передача данных, поведение на I/O**

- **Worker threads** — механизм для выполнения **параллельных** задач в Node.js, обходящий однопоточность V8
- Позволяют выполнять **CPU-интенсивные** операции без блокировки основного потока
- **Передача данных** между потоками происходит через **копирование** по сериализации (structured clone), не разделяется память напрямую, исключая гонки
- Можно использовать **SharedArrayBuffer** для общей памяти, но это сложнее
- В I/O задачах worker threads **могут уступать** event loop, т.к. Node оптимизирован под асинхронный I/O в основном потоке без блокировки
- Работают лучше там, где требуется тяжелая обработка данных, но не для обычных асинхронных I/O операций
- Подходят для распределения больших вычислительных нагрузок, но дополнительный оверхед на коммуникацию и создание потоков требует взвешенного решения

Итог: worker threads эффективны для вычислений, но в I/O чаще выгоднее использовать встроенный async/event-driven подход Node.js.

### Подробный ответ
#### Основной ответ  
**Worker threads** в Node.js (начиная с версии 10.5 и активно используемые с 12+) — это полноценные потоки, которые позволяют выполнять CPU-intensive задачи параллельно основному event loop, не блокируя его. Это особенно важно для JavaScript, который по умолчанию однопоточный. Профиты в использовании worker threads включают улучшение производительности при тяжёлых вычислениях, повышение отзывчивости приложения и масштабирование многопоточного исполнения.

При передаче данных в worker thread происходит копирование или клонирование сообщений с помощью механизма **structured clone**, который похож на работу postMessage в браузерах. Это значит, что копируются объекты без сериализации в строки, с поддержкой большинства типов данных, кроме функций и не клонируемых объектов (например, DOM). Для оптимизации передачи больших буферов часто используются **Transferable objects** (например, ArrayBuffer), которые передаются без копирования, а лишь передают "владение" буфером в worker.

Что касается I/O операций, worker threads не добавляют кардинальной пользы, так как Node.js I/O уже асинхронен и не блокирует event loop. Worker threads эффективнее всего именно для CPU-интенсивных задач (шифрование, парсинг больших данных, сжатие), но в I/O-bound задачах запускать worker может даже принести overhead.

#### Ключевые моменты  
- **Параллелизм CPU-bound задач**: worker threads реализуют настоящую параллельную обработку, что ускоряет тяжёлые вычисления, избегая блокировки основного потока.  
- **Structured clone + Transferables**: данные передаются копированием через механизм structured clone; для оптимизации больших бинарных данных можно передавать ownership без копирования.  
- **I/O операции**: для асинхронных I/O worker threads не дают большого выигрыша, т.к. event loop уже оптимизирован для неблокирующих вызовов, и использование медленных синхронных операций в worker не лучший подход.  

#### Практический контекст  
В реальных проектах worker threads применяют для фоновых задач вроде кодирования видео, машинного обучения, сложной математики или обработки больших JSON/XML, сохраняя responsiveness API. Для работы с базами и сетевыми запросами чаще используют встроенный async I/O без воркеров. Также стоит учитывать накладные расходы на создание воркера (~10-30 мс), что оправдано только для достаточно тяжёлых задач.
