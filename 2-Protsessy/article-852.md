# Как найти "тяжёлые" SQL-запросы без метрик?

**Как найти "тяжёлые" SQL-запросы без метрик?**

- анализ логов СУБД (slow query log, general log)
- изучение плана выполнения (EXPLAIN, EXPLAIN ANALYZE)
- поиск запросов с длительным временем выполнения/блокировками
- мониторинг блокировок и ожиданий в транзакциях
- использование системных представлений (pg_stat_statements, sys.dm_exec_query_stats)
- выявление полнотабличных сканов (Seq Scan) и отсутствия индексов
- поиск часто повторяющихся тяжёлых запросов по тексту и статистикам планов

# Подробный ответ
## Основной ответ
Найти "тяжёлые" SQL-запросы без метрик — это задача анализа производительности на основе доступных логов и системных инструментов. Основная идея — выявить запросы, которые потребляют значительные ресурсы (CPU, I/O, время выполнения) по фактическим наблюдениям. Обычно это делается через **логирование медленных запросов**, анализ плана выполнения и мониторинг блокировок.

## Ключевые моменты
- **Логирование медленных запросов**: в большинстве СУБД (PostgreSQL, MySQL) можно включить логирование запросов, превышающих заданный порог времени (например, slow query log с threshold 1с). Это самый практичный способ без метрик, позволяющий выявить долго выполняющиеся запросы по времени.
- **EXPLAIN / EXPLAIN ANALYZE**: если известен подозрительный запрос, вручную анализируем план выполнения, чтобы понять, где узкие места (сканирование всей таблицы, отсутствие индексов, дорогие соединения). Это помогает понять потенциальную тяжесть запросов.
- **Вспомогательные системные инструменты**: мониторинг активности с помощью `pg_stat_activity` (PostgreSQL), `SHOW PROCESSLIST` (MySQL) позволяет в реальном времени увидеть текущие запросы, разбить их по времени выполнения и нагрузке. Можно выявить текущие "тяжёлые" запросы без метрик.
- **Анализ блокировок и ожиданий**: долгие блокировки или время ожидания ресурсов указывают на проблемные запросы, влияющие на общую производительность.
  
## Практический контекст
В реальных проектах часто включают slow query log с порогом в 500-1000 мс, анализируют этот лог линейными скриптами или автоматизированными инструментами (pt-query-digest). Также администраторы используют `pg_stat_statements` в PostgreSQL для агрегированного анализа, но если метрик нет, основа — это логирование и live-снятие состояния.

Таким образом, даже без готовых метрик, используя стандартные возможности СУБД, можно регулярно выявлять и оптимизировать тяжелые запросы, улучшая производительность и стабильность системы.
