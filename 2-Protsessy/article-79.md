# Как называется паттерн проектирования, когда создается абстракция над разными сущностями, которые по смыслу одинаковые, но имеют разные поля?

Категория: Процессы

### Краткий ответ

**Паттерн абстракции над разными сущностями**

- область: проектирование, архитектура ПО  
- паттерн: **Адаптер (Adapter)**  
- суть: предоставляет **единый интерфейс** для разных объектов с разной структурой  
- отличие: скрывает различия в полях через перевод вызовов в понятный формат  
- часто применяют для интеграции **несовместимых API** или моделей  
- также близок паттерн **Фасад**, когда упрощается сложная подсистема  
- практическое применение: унификация доступа к разным источникам данных  

Таким образом, паттерн называется **Adapter** — он создает абстракцию, позволяющую работать с разными сущностями «под одним интерфейсом».

### Подробный ответ
#### Основной ответ
Паттерн проектирования, который предлагает создать общую **абстракцию над разными сущностями**, имеющими одинаковый смысл, но отличающимися полями, чаще всего называют **Adapter (Адаптер)** или **Facade (Фасад)** в зависимости от контекста. Однако если речь именно о моделировании разных типов с общей точкой входа, чаще применяют **Polymorphic Interface / Наследование** через **Абстрактную фабрику** или **Стратегию**. Если же задача объединить разные сущности с различными полями под единый интерфейс — это пример реализации **Adapter**.

#### Ключевые моменты
- **Adapter** позволяет создать общий интерфейс для разных классов с различной внутренней структурой, скрывая различия через единый API.
- **Facade** часто сглаживает сложность взаимодействия с множеством классов, но не меняет структуру самих сущностей.
- Для моделей с разной структурой полезно проектировать с помощью **Наследования и полиморфизма**, где базовый интерфейс описывает поведение, а конкретные сущности реализуют детализацию полей.
- Иногда используется паттерн **Data Transfer Object (DTO)** или **Wrapper**, чтобы унифицировать данные для передачи и обработки.

#### Практический контекст
В реальных проектах, например, при интеграции с разными внешними сервисами, каждый из которых возвращает похожие, но различающиеся данные, часто применяют **Adapter**, чтобы выстроить единый интерфейс взаимодействия. В JavaScript/TypeScript можно создать интерфейс с обязательными методами и адаптировать под него разные структуры через классы-обертки.

Таким образом, для унификации различных по структуре, но сходных по смыслу сущностей, правильнее всего использовать паттерн **Adapter** с поддержкой полиморфизма.
