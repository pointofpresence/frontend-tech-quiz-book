# Какие примитивы синхронизации вы знаете кроме lock (например, ReaderWriterLockSlim, Mutex)?

**Примитивы синхронизации кроме lock**

- **Mutex** — запрещает доступ из других процессов, тяжелее lock
- **Semaphore / SemaphoreSlim** — ограничивает количество одновременно выполняющихся потоков
- **ReaderWriterLockSlim** — позволяет несколько читателей или одного писателя
- **AutoResetEvent / ManualResetEvent** — для уведомления и ожидания событий между потоками
- **Monitor** — основа для lock, поддерживает ожидание и сигнализацию (Wait/Pulse)
- **SpinLock** — активное ожидание без блокировки ядра, для коротких критичных секций
- **Barrier** — синхронизация точек встречи нескольких потоков, барьер для групповых операций

Эффективный выбор зависит от сценария:  
- частое чтение — ReaderWriterLockSlim  
- межпроцессное — Mutex  
- ограничение параллелизма — Semaphore  
- события и сигналы — Event  
- низкая задержка — SpinLock

Используются для управления доступом, предотвращения гонок и обеспечения согласованности данных в многопоточных приложениях.

# Подробный ответ
## Основной ответ  
Помимо классического **lock** в C# есть несколько важных примитивов синхронизации, которые позволяют гибко управлять конкурентным доступом к ресурсам. Среди них — **ReaderWriterLockSlim, Mutex, Semaphore, SemaphoreSlim, AutoResetEvent, ManualResetEvent, SpinLock, CountdownEvent** и др. Каждый из этих примитивов решает разные задачи синхронизации с учетом производительности и сценария использования.

## Ключевые моменты  
- **ReaderWriterLockSlim** позволяет одновременно читать из ресурса нескольким потокам, блокируя запись до освобождения всех читателей. Это улучшает производительность в сценариях с преобладанием операций чтения.  
- **Mutex** — это примитив, обеспечивающий взаимное исключение между потоками, причем даже между процессами (системный mutex). Подходит для межпроцессной синхронизации, но дороже по стоимости по сравнению с lock.  
- **Semaphore и SemaphoreSlim** позволяют ограничить число потоков, одновременно имеющих доступ к ресурсу. Semaphore работает на уровне ОС и поддерживает межпроцессные сценарии, а SemaphoreSlim — более легковесен и оптимизирован для одного процесса.  
- **AutoResetEvent и ManualResetEvent** — это EventWaitHandle-примитивы, которые работают как сигнальные флаги. AutoResetEvent автоматически сбрасывается после освобождения одного потока, а ManualResetEvent требует явного сброса, что полезно для уведомления множества потоков.  
- **SpinLock** — низкоуровневый примитив, который активно ждет освобождения блокировки без переключения контекста, эффективен при очень коротких критических секциях.  
- **CountdownEvent** полезен, когда необходимо ждать завершения определённого числа операций перед продолжением.  

## Практический контекст  
В реальных проектах часто комбинируют эти примитивы в зависимости от требований к производительности и характеру задач. Например, ReaderWriterLockSlim широко используется при частом многопоточном чтении данных, Mutex — для синхронизации между процессами (например, доступ к файлу). SemaphoreSlim часто применяют в асинхронных сценариях для контроля параллелизма, а AutoResetEvent — для реализации producer-consumer паттерна. Понимание этих примитивов позволяет выбирать оптимальные решения под конкретные потребности и достигать баланс между безопасностью и производительностью.
