# Какие ограничения можно задать для столбца в таблице?

**Ограничения для столбца в таблице**

- область SQL, реляционные БД  
- **Constraints** обеспечивают целостность данных  
- **NOT NULL** — запрещает пустые значения  
- **UNIQUE** — гарантирует уникальность значений  
- **PRIMARY KEY** — уникален, не NULL, индексирует  
- **FOREIGN KEY** — связь с другим столбцом для целостности ссылок  
- **CHECK** — выражение для ограничения значений (например, возраст > 0)  
- **DEFAULT** — задаёт значение по умолчанию, если вставлено NULL  
- критично для предотвращения ошибок и обеспечения бизнес-правил в данных

# Подробный ответ
## Основной ответ
Ограничения (constraints) в колонках таблицы баз данных — это правила, которые гарантируют корректность и целостность данных на уровне самой БД. Они помогают предотвратить ввод некорректных или дублирующихся значений и обеспечивают бизнес-логику на уровне схемы.

## Ключевые моменты
- **NOT NULL** — запрещает хранить NULL, т.е. значение обязательно должно быть заполнено. Например, для обязательных полей email или имя пользователя.
- **UNIQUE** — гарантирует уникальность значений в столбце, предотвращая дублирование. Например, username или email, если они должны быть уникальными.
- **PRIMARY KEY** — особый вид ограничения, который обеспечивает уникальность и NOT NULL одновременно, идентифицируя запись однозначно. Обычно задаётся на одном или нескольких столбцах.
- **FOREIGN KEY** — ограничение целостности ссылочной связи, заставляет значения столбца соответствовать значениям в другой таблице, обеспечивая связь между ними.
- **CHECK** — позволяет задать произвольное условие на значения столбца, например `CHECK (age >= 18)` для ограничений по возрасту.
- **DEFAULT** — задаёт значение по умолчанию, которое будет подставлено, если при вставке не передали значение в этот столбец.

## Практический контекст
Например, в PostgreSQL 14+ при проектировании таблицы users часто применяют комбинацию **PRIMARY KEY (id)**, **UNIQUE (email)**, **NOT NULL (name, email)** и **CHECK (length(password) >= 8)** — так одновременно контролируют уникальность, обязательность и минимальную длину пароля. Это минимизирует ошибки на уровне данных и упрощает логику на стороне приложения.
