# Чем по смыслу отличаются 400 и 500 ошибки HTTP?

Категория: Процессы

### Краткий ответ

**Чем по смыслу отличаются 400 и 500 ошибки HTTP?**

- HTTP-статусы — категории ошибок  
- **400 Bad Request** — ошибка на стороне клиента  
- сигнализирует о **некорректном запросе** (синтаксис, параметры)  
- клиент отправил **недопустимые данные**  
- **500 Internal Server Error** — ошибка на сервере  
- означает сбой или исключение во время обработки запроса  
- клиент мог отправить валидный запрос, но сервер не смог его обработать  
- важно разделять для отладки и улучшения UX: 400 — проверить запрос, 500 — исправлять сервер  
- практика REST: сообщать конкретные 4xx и обрабатываемые 5xx ошибки для клиента  
- помогает разделять ответственность между клиентом и сервером  

Это ключевое различие: **400 — вина клиента, 500 — вина сервера**.

### Подробный ответ
#### Основной ответ
HTTP ошибки **400 (Client Error)** и **500 (Server Error)** различаются по причине возникновения проблемы: код 400 означает, что ошибка произошла на стороне клиента — запрос сформирован некорректно или содержит неверные данные; код 500 сигнализирует о проблемах на стороне сервера, когда запрос правильный, но ошибка возникает внутри самого сервера.

#### Ключевые моменты
- **400 Bad Request** — клиент отправил неправильный или неполный запрос, например, синтаксическая ошибка, неверный формат JSON, отсутствие обязательных параметров. Это означает, что клиент должен исправить запрос перед повторной отправкой.
- **500 Internal Server Error** — сервер столкнулся с неожиданной проблемой (например, исключение в коде, ошибки базы данных, перегрузка), и не может обработать даже корректный запрос. Часто требуется исправление на серверной стороне.
- Эти коды помогают разделять ответственность: клиент отвечает за корректность запроса (4xx), сервер — за надежность и обработку (5xx). Это ключ к отладке и мониторингу в production.

#### Практический контекст
В проекте с API важно четко разграничивать эти ошибки: например, валидировать данные на клиенте, чтобы избегать 400, и логировать 500 ошибки для быстрого реагирования. В React 18+ часто используют библиотеку axios, где легко фильтровать эти статусы для user-friendly сообщений или retry-механизмов.
