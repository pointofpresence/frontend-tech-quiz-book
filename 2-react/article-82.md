# В каких случаях тяжелые операции тормозят React?

### Категории

Async, Optimization, React, Performance

### Краткий ответ

**В каких случаях тяжелые операции тормозят React?**  
- Работа с **большими объёмами данных** в рендере  
- Выполнение **тяжелых вычислений** внутри рендер-функций  
- Частые вызовы методов, вызывающих **повторные рендеры** (например, setState)  
- Отсутствие оптимизации, как **memo**, **shouldComponentUpdate**  
- Исполнение кода в основном **главном потоке** браузера (JS однопоточен)  
- Работа с большим количеством компонентов или сложной иерархией  
- Манипуляции с **DOM** вне виртуального DOM  
- Отсутствие ленивой загрузки и дебаунсинга (например, для событий ввода)

Итого: тормоза возникают, когда тяжелые синхронные процессы блокируют главный поток, мешая React быстро обновлять виртуальный DOM и применять изменения в браузере.

### Подробный ответ
#### Основной ответ
Тяжёлые операции тормозят React в тех случаях, когда они блокируют главный поток JavaScript, отвечающий за рендеринг компонентов и обработку событий. React работает в однопоточном окружении браузера, поэтому любая длительная синхронная операция, например, вычисления или обработка больших данных прямо в render-функциях, приводят к задержкам обновления интерфейса и ухудшению UX.

#### Ключевые моменты
- **Вычисления в рендере**: если во время вызова `render` или в компонентах вызываются тяжёлые вычисления (например, сортировка больших массивов, операции с интенсивными циклами), это блокирует обновление DOM и препятствует плавной отрисовке.
- **Синхронные операции в event handlers**: длительные синхронные задачи в обработчиках событий пользовательского интерфейса замедляют реакцию приложения — браузер не сможет обработать последующие события или анимации.
- **Большие и частые перерисовки**: если React рендерит много компонентов часто без оптимизаций (например, без мемоизации через `React.memo` или `useMemo`), общее время на вычисления возрастает и тормозит интерфейс.
- Начиная с **React 18**, введён концепт concurrent режимов, которые позволяют более эффективно планировать обновления и уменьшать лаги, но тяжёлые операции в синхронном коде всё равно будут узким местом.

#### Практический контекст
Обычно тяжёлые операции выносят из рендера в отдельные web workers или кешируют результаты с помощью `useMemo`/`React.memo`. Для асинхронных вызовов применяют дебаунсинг и ленивую загрузку данных (code splitting). В реальных проектах используют инструменты профилирования React DevTools и браузерные Performance панели, чтобы найти узкие места и оптимизировать рендеринг.
