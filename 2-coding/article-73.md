# Какие методы использовали для оптимизации SQL-запросов (индексы, уменьшение JOIN, проверка на NULL)?

### Категории

Coding, Indexing, Null check, Performance, Indexes, Algorithms, Backend, Database, Joins, Optimization, Live coding, Null, Join, SQL

### Краткий ответ

**Методы оптимизации SQL-запросов**  
- **Индексы**: создание на колонках с частыми фильтрами и JOIN, уменьшение полного сканирования таблиц  
- **Минимизация JOIN**: избегание избыточных соединений, использование предфильтрации и подзапросов  
- **Проверка на NULL**: минимизация проверок NULL в условиях, использование COALESCE/DEFAULT для ускорения сравнения  
- **Выборочные поля**: выбор только нужных колонок вместо SELECT *  
- **Использование EXPLAIN**: анализ плана выполнения для выявления узких мест  
- **Партиционирование и оптимизация запросов с WHERE**: сокращение объёма данных для обработки  
- **Кэширование и денормализация**: снижение нагрузки при сложных агрегациях и частых запросах  
- Итог: комплексный подход позволяет значительно повысить скорость и снизить нагрузку на БД.

### Подробный ответ
#### Основной ответ
Для оптимизации SQL-запросов я использую комплексный подход, начиная с правильного индексирования и заканчивая рефакторингом самих запросов. Индексы позволяют существенно ускорить поиск и фильтрацию данных, особенно при работе с большими таблицами. При этом важно выбирать тип индекса (B-tree, Hash, GiST и т.д.) исходя из характера запросов. Помимо индексов, я стараюсь минимизировать количество **JOIN**, особенно сложных и превышающих 3-4 таблицы, так как они сильно влияют на производительность — при необходимости разбиваю задачи на несколько простых запросов. Ещё один важный аспект — правильная работа с условиями, например, избегание проверки на **NULL** в двухсторонних фильтрах или использование IS NOT NULL и COALESCE, чтобы избежать полных сканирований таблиц.

#### Ключевые моменты
- **Индексация**: создание покрывающих (covering) индексов, включающих только необходимые поля для конкретных запросов, что снижает обращение к данным таблиц.
- **Оптимизация JOIN**: выборочный join вместо общего, использование EXISTS/IN для подзапросов, а также фильтрация данных до выполнения соединения.
- **Условия и NULL**: избегание функций или выражений, которые препятствуют использованию индексов; например, проверка на NULL может блокировать индекс и привести к full scan.
- Анализ плана выполнения запросов с помощью EXPLAIN ANALYZE (PostgreSQL 14+, MySQL 8) для понимания узких мест и принятия обоснованных решений по оптимизации.

#### Практический контекст
В проектах с большими объёмами данных (от сотен тысяч строк и выше) я использовал покрывающие индексы на часто фильтруемых колонках, что снижало latency запросов с ~300ms до ~50ms. Для аналитических запросов часто разбивал многоступенчатые JOIN на последовательные преобразования (ETL-подобно), а в OLTP обращался к query planner для выбора правильной стратегии и избегал NULL checks, меняя архитектуру данных. Также применял материализованные представления для предрасчётов сложных агрегаций.
