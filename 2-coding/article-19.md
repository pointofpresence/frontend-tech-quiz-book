# Чем отличается Read Committed от Repeatable Read?

### Категории

Coding, High load, Architecture, Concurrency, Isolation level, Transaction isolation, Algorithms, Backend, Database, Transaction, Distributed systems

### Краткий ответ

**Чем отличается Read Committed от Repeatable Read?**

- уровень изоляции в **СУБД** (ACID)
- Read Committed: видит только **зафиксированные** данные
- предотвращает **грязные чтения**
- позволяет **неповторяющиеся чтения** (non-repeatable reads) и фантомные чтения
- Repeatable Read: данные при повторных чтениях **не меняются**
- предотвращает **грязные** и **неповторяющиеся чтения**
- фантомные чтения возможны в зависимости от СУБД (например, в MySQL InnoDB — запрещены)
- Repeatable Read требует **блокировок** или MVCC для стабильности
- Read Committed быстрее, но менее консистентен
- выбирается, если важна более высокая **консистентность** чтения  

Итог: **Read Committed** защищает от грязных чтений, **Repeatable Read** — от грязных и неповторяющихся, приближая к сериализуемости.

### Подробный ответ
#### Основной ответ
**Read Committed** и **Repeatable Read** — это два уровня изоляции транзакций в реляционных базах данных, определяющие степень видимости изменений данных внутри и между транзакциями. **Read Committed** гарантирует, что транзакция видит только зафиксированные данные на момент конкретного чтения, но может видеть разные данные при повторных чтениях. **Repeatable Read** обеспечивает, что повторные чтения одних и тех же строк транзакцией будут идентичными, предотвращая неповторяющиеся чтения, но не всегда исключая фантомные чтения.

#### Ключевые моменты
- **Read Committed**: при каждом запросе транзакция видит только зафиксированные другим транзакциями данные. Это предотвращает "грязное чтение" (dirty read), но допускает "неповторяющееся чтение" (non-repeatable read), когда один и тот же запрос внутри транзакции может вернуть разные результаты.
- **Repeatable Read**: фиксация снимка данных на момент первого чтения, обеспечивающая, что все последующие чтения тех же строк в рамках транзакции будут возвращать идентичные данные. При этом фантомные чтения (появление новых строк в результате запроса) в некоторых СУБД, например в MySQL InnoDB, могут продолжать возникать без дополнительной блокировки.
- В PostgreSQL 14+ и большинстве современных СУБД **Repeatable Read** обычно достигается через MVCC (многоверсионность) и предотвращает неповторяемые чтения, давая более сильную гарантию согласованности данных по сравнению с Read Committed.
  
#### Практический контекст
В системах с высокой конкуренцией **Read Committed** используется для большей пропускной способности при меньшем уровне изоляции, подходящем для чтения данных без жестких требований к повторяемости. **Repeatable Read** оправдан, если необходима консистентность в рамках транзакции с минимизацией риска непредсказуемых смен данных при множественных чтениях (например, финансовые операции). В PostgreSQL для исключения фантомного чтения стоит рассматривать уровень **Serializable**.
