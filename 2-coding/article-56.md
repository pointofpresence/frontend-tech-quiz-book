# Какие сетевые (TCP/UDP) и веб-протоколы (HTTP, gRPC) ты знаешь?

### Категории

Coding, Backend, Optimization, Data structures, Live coding, Networking, Protocols

### Краткий ответ

**Какие сетевые (TCP/UDP) и веб-протоколы (HTTP, gRPC) ты знаешь?**

- Сетевые: **TCP** — надёжный, установочный, с контролем потока; **UDP** — быстрый, без установления соединения, без гарантий доставки  
- HTTP: **текстовый** протокол приложений, работает поверх TCP, поддерживает методы GET, POST, PUT, DELETE  
- HTTP/2: бинарный, мультиплексирование запросов, сжатие заголовков для скорости  
- HTTP/3: базируется на QUIC, использует UDP для уменьшения задержек и потерь  
- gRPC: современный фреймворк RPC, основан на HTTP/2, с поддержкой стриминга и протокола Protocol Buffers для компактности  
- WebSocket: двунаправленный протокол для постоянного соединения поверх HTTP/HTTPS  
- REST (веб-стиль): архитектура использования HTTP с ресурсно-ориентированными API  
- TLS: защита TCP с помощью шифрования, широко применяется в HTTPS

---  
**Обоснование:** описаны ключевые протоколы передачи данных, их свойства и практические применения, продемонстрирована разница между транспортными и прикладными уровнями, учтены современные веб-решения.

### Подробный ответ
#### Основной ответ  
Я знаком с основными сетевыми протоколами TCP и UDP, а также с веб-протоколами HTTP и gRPC. **TCP** — это протокол с установлением соединения, обеспечивающий надежную доставку данных, контроль порядка и исправление ошибок, часто используется там, где важна целостность данных (например, HTTP, FTP). **UDP** — безсоединительный, не гарантирует доставку или порядок, но очень быстрый и легковесный, применяется для real-time видео, VoIP, игровых серверов.

Что касается веб-протоколов, **HTTP/1.1** и **HTTP/2** — базовые протоколы передачи в интернете, поддерживают работу с REST API, где HTTP-запросы и ответы обычно в формате JSON или XML. В HTTP/2 добавлена мультиплексированная передача, использование бинарного протокола и сжатие заголовков, что снижает latency. **gRPC** — это современный RPC-фреймворк поверх HTTP/2 с поддержкой бинарного формата Protobuf, обеспечивает высокую производительность и строго типизированные контракты на уровне сервисов, широко используется для микросервисной архитектуры.

#### Ключевые моменты  
- **TCP** гарантирует доставку и порядок пакетов, но с небольшими задержками, подходят для надёжных сервисов. **UDP** быстрее, но не гарантирует порядок и доставку, что критично для real-time приложений.  
- **HTTP/1.1** — текстовый, более простой, но медленный из-за лишних сетевых рунд-трипов. **HTTP/2** снижает задержки, поддерживает мультиплексирование и эффективное управление потоками.  
- **gRPC** позволяет создавать высокопроизводительные сервисы с контрактами, используя Protobuf и HTTP/2, что делает его оптимальным для распределённых систем с низкой латентностью.

#### Практический контекст  
В реальных проектах я использовал TCP для построения надёжных клиент-серверных приложений, UDP для трансляции real-time данных. HTTP/2 и REST API широко применял для внешних интеграций, а gRPC использовал для общения между микросервисами в распределённой системе, где важна высокая скорость и сжатие трафика.
