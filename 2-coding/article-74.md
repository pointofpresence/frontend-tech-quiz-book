# Как работают кластеризованные и некластеризованные индексы? Когда их стоит применять?

### Категории

Coding, Algorithms, Backend, Database, Optimization, Data structures, Live coding

### Краткий ответ

**Как работают кластеризованные и некластеризованные индексы? Применение**

- **Кластеризованный индекс** — определяет **физический порядок** строк в таблице по ключу, т.е. данные хранятся упорядоченно  
- По таблице может быть только **один кластеризованный индекс** из-за физического ограничения  
- Поиск по кластеризованному индексу быстрый при диапазонных запросах (например, BETWEEN) и сортировке

- **Некластеризованный индекс** — отдельная структура с указателями на строки, физический порядок таблицы сохраняется  
- Может быть много некластеризованных индексов, что ускоряет **выборочные запросы по разным колонкам**  
- При вставке обновлении индекс требует поддержания, что может влиять на производительность

- **Применение**:  
  - Кластеризованный — когда важна быстрая выборка по ключу с сортировкой или диапазону (например, PK, дата)  
  - Некластеризованный — для ускорения частых запросов по другим колонкам, не влияющих на порядок хранения  

- Баланс между количеством индексов и производительностью записи — ключевой фактор в выборе стратегии  
- Современные СУБД часто комбинируют оба для оптимальной скорости чтения и записи  

Итого: кластеризованный индекс **упорядочивает таблицу**, некластеризованный — **создает дополнительную структуру доступа**; выбор зависит от паттернов запросов и нагрузки.

### Подробный ответ
#### Основной ответ
**Кластеризованный индекс** — это индекс, который определяет физический порядок строк в таблице согласно ключу индекса. Фактически, данные таблицы отсортированы и хранятся в соответствии с этим индексом. В отличие от него, **некластеризованный индекс** — это отдельная структура данных, которая содержит ключи индекса и указатели на физические строчки таблицы (или на кластеризованный индекс), но сами данные хранятся вне индекса и не сортированы в его порядке.

#### Ключевые моменты
- **Кластеризованный индекс** в реляционных БД (например, SQL Server, MySQL InnoDB) бывает только один на таблицу, так как данные можно физически отсортировать только по одному критерию. Он эффективен для запросов с диапазонами, сортировками и выборками большого количества последовательных данных.
- **Некластеризованный индекс** может быть много и служит для быстрого поиска по другим колонкам, не изменяя физический порядок хранения. Он создаёт дополнительную структуру (B-дерево), ускоряющую точечные запросы и операции поиска по нестандартным полям.
- При применении кластеризованного индекса операции вставки/обновления могут быть чуть дороже из-за необходимости поддерживать физический порядок, но чтение данных по ключу происходит очень быстро. Некластеризованные индексы экономят место и позволяют создать множество дополнительных путей доступа, но требуют дополнительных переходов по указателям при запросах.

#### Практический контекст
В реальных базах данных рекомендуется использовать **кластеризованный индекс** на поле с уникальными значениями, часто используемое в условии соединений или сортировок (например, первичный ключ). **Некластеризованные индексы** создают для столбцов, по которым часто делают фильтрацию или выборку, но которые не подходят для сортировки данных или не уникальны, например, индексы по датам, статусам, внешним ключам. В PostgreSQL, например, кластеризованный индекс можно создать с помощью команды CLUSTER, а в MySQL InnoDB первичный ключ по умолчанию — кластеризованный.
