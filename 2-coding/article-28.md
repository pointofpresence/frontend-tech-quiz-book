# Что такое фантомное чтение (phantom read)? Решает ли эту проблему Repeatable Read?

### Категории

Coding, High load, Concurrency, Repeatable read, Backend, Database, Phantom read, Transactions, Isolation levels, Message broker, Networking, DevOps

### Краткий ответ

**Что такое фантомное чтение? Решает ли Repeatable Read?**  
- фантомное чтение — проблема в **транзакциях** и **изоляции**  
- возникает при повторном запросе, когда **новые строки** появляются/исчезают из результата  
- **Repeatable Read** гарантирует повторяемость строк, но не всегда предотвращает фантомы  
- фантомы решаются только на уровне **сериализуемости** или **специфических блокировок (например, Predicate Locks)**  
- в популярных СУБД Repeatable Read часто **не полностью защищает** от фантомов из-за реализации  
- для полного решения нужен уровень изоляции **Serializable**  
- практическое значение — важность выбора правильного уровня изоляции для целостности данных

### Подробный ответ
#### Основной ответ
**Фантомное чтение (phantom read)** — это явление в транзакционной системе, когда в ходе одной транзакции при повторном выполнении одного и того же запроса к базе данных появляются новые строки, которые не были видны при первом запросе. То есть, данные, на которые опирается транзакция, меняются "снаружи" из-за операций других транзакций, что приводит к непредсказуемому поведению. 

**Repeatable Read** из стандартов транзакций SQL пытается гарантировать, что чтение одних и тех же строк будет стабильным, но фантомные чтения при этом возможны — новые строки, соответствующие критериям запроса, могут появиться. Для полного устранения фантомных чтений нужен уровень изоляции **Serializable**, который блокирует добавление таких новых строк.

#### Ключевые моменты
- В **Repeatable Read** блокируются модификации уже прочитанных строк, но не предотвращается добавление новых строк, подходящих под критерии выборки.
- Фантомные чтения характерны для ситуаций с запросами типа `SELECT … WHERE` — при повторном выполнении запроса могут появиться новые строки.
- В **Serializable** уровне транзакций используется более строгая блокировка (например, gap locking в InnoDB), она предотвращает появление фантомных записей, гарантируя полную изоляцию.

#### Практический контекст
В PostgreSQL уровень Repeatable Read имплементовался начиная с версии 9.1 и гарантирует отсутствие не повторяемых чтений, но фантомы теоретически возможны из-за использования MVCC, и их предотвращение обеспечивает только Serializable режим. В MySQL/InnoDB Repeatable Read блокирует gap locks, что устраняет фантомы, но это частный случай реализации. Поэтому при проектировании высоконадежных систем важно корректно выбирать уровень изоляции, исходя из бизнес-требований.
