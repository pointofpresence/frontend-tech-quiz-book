# Вызывают ли изменения props ререндеры в React?

### Категории

Coding, Algorithm complexity, React, Event loop, Props, Debugging, Rerender, TypeScript

### Краткий ответ

**Вызывают ли изменения props ререндеры в React?**

- React — библиотека для UI с **виртуальным DOM**
- Изменение **props** родительского компонента передаётся в дочерний
- При изменении props дочерний компонент **триггерит ререндер**
- React сравнивает новые props с предыдущими (**shallow comparison**)
- В функциональных компонентах ререндер происходит при любом изменении props
- Для оптимизации используют **React.memo** (проверка props для избежания ререндеров)
- Ререндеры нужны для обновления UI на основе новых данных, обеспечивая реактивность интерфейса

### Подробный ответ
#### Основной ответ
Да, в **React** изменения **props** по умолчанию вызывают ререндер компонента. Когда компонента получает новые пропсы, React запускает процесс обновления, чтобы отобразить изменения в UI. Однако это не означает, что весь дочерний компонент отрисовывается заново в DOM — React использует виртуальный DOM и эффективное сравнение (diffing), чтобы минимизировать реальные обновления.

#### Ключевые моменты
- **Props — источник данных для компонента:** когда родитель передает новые значения, React считает, что UI мог измениться, и инициирует ререндер.
- **PureComponent и React.memo:** эти оптимизации позволяют избегать ререндеров, если props не изменились поверхностно (shallow compare). Например, при одинаковых примитивных значениях ререндеры отменяются.
- **Изменение структуры объектов:** если props — объекты или массивы, даже если содержимое не меняется, но ссылка новая, React воспринимает это как изменение и ререндерит компонент. Это часто приводит к ненужным обновлениям без мемоизации.

#### Практический контекст
В React 18+ с автоматическим batching эффективность ререндеров улучшилась, но правильное управление props, использование `React.memo` и правильное обновление состояния (immutability) остаются обязательными для оптимизации производительности. Это критично в больших приложениях, где ререндеры дорогие (например, сложные вычисления или тяжелые DOM-деревья).
