# Что будет, если смешать асинхронный код с CPU-bound задачей?

### Категории

Backend, JavaScript, Async, Performance, Concurrency, CPU-bound

### Краткий ответ

**Что будет, если смешать асинхронный код с CPU-bound задачей?**

- тема: асинхронность и CPU-bound  
- асинхронность эффективна для **I/O операций**, а не для CPU-bound  
- CPU-bound задачи **блокируют** поток, тормозя асинхронный цикл событий  
- асинхронный код ждет освобождения CPU, снижается **параллелизм**  
- увеличивается латентность и общее время ответа  
- решение: выносить CPU-bound задачи в отдельные **потоки** или процессы  
- итог: без разделения блокировка снижает эффективность асинхронной модели

### Подробный ответ
#### Основной ответ
Если смешать **асинхронный код** с **CPU-bound задачей**, то эффективность асинхронности сильно пострадает. Асинхронный подход (например, в Node.js или Python с asyncio) хорошо работает для I/O-bound операций, не блокируя event loop. Но CPU-bound задачи, требующие интенсивных вычислений, блокируют поток, в котором выполняются, препятствуя обработке асинхронных событий.

#### Ключевые моменты
- **Блокировка event loop**: CPU-bound задачи занимают процессорное время, не освобождая event loop для обработки других асинхронных задач, из-за чего теряется преимущество асинхронности и возникает задержка.
- **Отсутствие параллелизма** в однопоточном окружении: если асинхронный код работает в одном потоке (например, Node.js 14 или Python 3.8 в стандартном event loop), то CPU-intensive задачи "замораживают" всю обработку событий.
- **Решения**: стоит выносить тяжелые CPU-задания в отдельные worker threads, процессы (например, через worker_threads в Node.js или multiprocessing в Python), либо использовать специализированные очереди задач и балансировщики нагрузки.

#### Практический контекст
В продакшене, чтобы сохранить скорость отклика асинхронных API на уровне **latency ~50ms**, CPU-задания часто делегируют на фоновые воркеры или выделяют отдельные сервисы, чтобы не блокировать основной event loop. Например, в Node.js 18+ активно используют worker threads для таких задач, а в Python — Celery и multiprocessing.
