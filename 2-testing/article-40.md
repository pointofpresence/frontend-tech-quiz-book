# Почему в проекте были только UI-тесты? Как покрывали бизнес-логику?

### Категории

UI testing, Mobile, Android, Testing, Business logic

### Краткий ответ

**Почему в проекте были только UI-тесты? Как покрывали бизнес-логику?**

- ограничение времени/ресурсов → фокус на UI для быстрой проверки  
- UI-тесты покрывают **end-to-end** сценарии, включая бизнес-логику  
- бизнес-логика проверялась через **интеграционные тесты** на уровне UI  
- отсутствие unit-тестов могло быть вызвано **уникальной архитектурой** или legacy-кодом  
- параллельно использовались ручные/автоматизированные проверки бизнес-правил  
- планировалось/были в работе выделение unit-тестов для критичных компонентов  
- практика 👉 UI-тесты дают бизнес-ценность, сокращая баги в пользовательском потоке

### Подробный ответ
#### Основной ответ
Если в проекте были только **UI-тесты**, это могло быть связано с приоритетом проверки конечного пользовательского опыта и интеграции компонентов на высоком уровне. Бизнес-логику при этом часто покрывают на уровне **юнит-тестов** или **интеграционных тестов** отдельно, но в вашем случае, возможно, она проверялась косвенно через UI-тесты, которые эмулируют поведение пользователя и проходят полный путь данных через все слои приложения.

Такой подход иногда бывает вызван ограничениями по времени, ресурсам или архитектурным особенностям — например, если бизнес-логика тесно переплетена с UI или если в проекте используется low-code/no-code платформа. Однако одних UI-тестов, как правило, недостаточно для полной уверенности в корректности бизнес-логики из-за их медлительности и хрупкости.

#### Ключевые моменты
- **UI-тесты** обычно покрывают end-to-end сценарии, гарантируя, что пользовательский сценарий работает целиком — от интерфейса до результата.
- Для проверки бизнес-логики традиционно используют **юнит-тесты** (например, на Jest, JUnit) и **модульные интеграционные тесты**, которые быстрее и менее подвержены нестабильности, чем UI-тесты.
- В проекте могли применять **параллельное покрытие логики через статический анализ**, code review и QA процессы, либо бизнес-логику строили по паттернам (например, чистая архитектура), которые упрощают тестируемость на уровне UI.

#### Практический контекст
В реальных проектах, например на React 18+ с Redux, часто комбинируют **юнит-тесты для редьюсеров и селекторов**, интеграционные тесты сервисов и UI e2e-тесты через Cypress или Playwright. Если же бюджет позволял только UI-тестирование, то тогда старался максимально покрывать ключевые бизнес-сценарии через эти тесты и уделял внимание качественному код-ревью и автоматизации проверки данных на бекенде.
