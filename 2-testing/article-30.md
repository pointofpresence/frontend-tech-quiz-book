# Какие HTTP-статусы (200/300/400/500) чаще всего встречаются при тестировании API?

### Категории

Aqua, Backend, Automation, Database, HTTP status, API, HTTP status codes, Integration testing, Testing, Microservices, Payment systems

### Краткий ответ

**Частые HTTP-статусы при тестировании API**  
- 200: успешный запрос, данные возвращены  
- 201: создание ресурса (POST)  
- 204: успех без тела ответа (DELETE/PUT)  
- 301/302: редиректы (переадресация)  
- 400: ошибка клиента, неверный запрос  
- 401/403: проблемы с авторизацией/доступом  
- 404: ресурс не найден  
- 500: внутренняя ошибка сервера  
- 503: сервер временно недоступен  
- Используют для проверки **корректности работы** и **обработки ошибок**  
- Важно тестировать разные группы для надежности API

### Подробный ответ
#### Основной ответ
При тестировании **API** чаще всего встречаются HTTP-статусы из категорий **200, 400 и 500**. Код **200 OK** указывает на успешное выполнение запроса, что является базой для проверки корректности работы API. Статусы в диапазоне **400** свидетельствуют о проблемах на стороне клиента — например, неверный формат запроса или отсутствие обязательных параметров. Статусы **500** отображают ошибки на сервере, которые важны для выявления сбоев и недоработок backend-логики.

#### Ключевые моменты
- **200 OK** — основной индикатор успешного ответа. В REST API это значит, что запрос обработан корректно, и данные возвращены. Обычно тестируется с конкретными сценариями запросов, проверяется структура и содержание ответа.
- **4xx** (например, 400 Bad Request, 401 Unauthorized, 404 Not Found) — критически важны при валидации входных данных и проверки аутентификации/авторизации. Они помогают убедиться, что API адекватно обрабатывает неправильные запросы.
- **5xx** (например, 500 Internal Server Error, 503 Service Unavailable) — указывают на внутренние ошибки сервера и должны сигнализировать о проблемах в коде, зависимостях или инфраструктуре. Повторяются при тестах устойчивости и обработки исключений.

#### Практический контекст
В реальных проектах тестировщики и разработчики уделяют особое внимание **401/403** при интеграции с аутентификацией, а также **429 Too Many Requests** в сервисах с ограничением доступа. Мониторинг и логирование этих статусов помогает быстро выявлять и устранять проблемы в продакшене, а покрытие тестами — гарантирует стабильность API.
