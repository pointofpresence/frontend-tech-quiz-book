# Какие типы очередей предоставляет GCD?

Категория: Технологии

### Краткий ответ

**Типы очередей в GCD**

- GCD (Grand Central Dispatch) — система управления потоками в iOS/macOS  
- **Сериализованные** (Serial) очереди — выполняют задачи последовательно, по одной, гарантируют порядок  
- **Параллельные** (Concurrent) очереди — выполняют несколько задач одновременно, порядок запуска сохраняется, порядок завершения — нет  
- **Главная очередь** (Main queue) — сериализованная, выполняется в основном потоке UI, нужна для обновления интерфейса  
- **Глобальные очереди** (Global queues) — параллельные, системные, с разными уровнями приоритета (QoS), для фоновых задач  
- Пользовательские очереди создаются с использованием serial или concurrent параметров, настройка приоритета необязательна  
- Используются для оптимизации и асинхронности, улучшения отзывчивости приложений

### Подробный ответ
#### Основной ответ
**Grand Central Dispatch (GCD)** предоставляет несколько типов очередей для организации параллельного выполнения задач в iOS и macOS. Основных типов две: **serial queues** (последовательные) и **concurrent queues** (конкурентные). Помимо этого есть системные глобальные очереди с различными качествами обслуживания (QoS).

#### Ключевые моменты
- **Серийные очереди (Serial Queues)**: выполняют задачи последовательно, одна за другой, что упрощает синхронизацию данных без использования блокировок. Используются для упорядоченного выполнения кода.
- **Конкурентные очереди (Concurrent Queues)**: позволяют запускать несколько задач одновременно, когда доступно несколько ядер процессора. Задачи стартуют почти одновременно, но завершаются в произвольном порядке.
- **Глобальные очереди (Global Queues)**: представляют собой заранее сконфигурированные конкурентные очереди с разным уровнем приоритета/качества обслуживания (например, userInteractive, userInitiated, utility, background). С GCD 4+ рекомендовано управлять приоритетами через QoS классы.
- **Main Queue (Главная очередь)**: это уникальная серийная очередь, связанная с главным потоком UI, где выполняется весь пользовательский интерфейс. Любые обновления UI должны выполняться в этой очереди.

#### Практический контекст
В реальных приложениях часто создают собственные серийные очереди для последовательных задач (например, запись в базу данных), используют глобальные конкурентные очереди для сетевых запросов и background задач, а всю работу с UI синхронизируют через главную очередь. Такое разделение помогает эффективно использовать ресурсы CPU, гарантируя безопасность данных и отзывчивость интерфейса.
