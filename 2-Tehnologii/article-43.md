# Поддерживает ли Kafka репликацию данных и как работает восстановление при падении лидер-реплики?

**Поддержка репликации и восстановление в Kafka**

- Kafka — распределённая система с **репликацией** топиков
- Каждая часть топика (partition) имеет **лидера** и **фолловеры**
- Все записи идут через лидера, фолловеры **ассинхронно копируют** данные
- Реплики синхронизируют через **ISR (in-sync replicas)** — список актуальных фолловеров
- При падении лидера, Kafka выбирает нового лидера из ISR
- Новый лидер обеспечивает **консистентность** данных и продолжает обслуживание
- Репликация обеспечивает **устойчивость** к сбоям и сохранность данных  

**Итог:** Kafka поддерживает репликацию с автоматическим восстановлением лидера, гарантируя высокую доступность и целостность данных.

# Подробный ответ
## Основной ответ
Да, **Apache Kafka** поддерживает встроенную **репликацию данных** для обеспечения высокой доступности и устойчивости к отказам. Каждый топик разбивается на партиции, и каждая партиция реплицируется на несколько брокеров (называются репликами). Из них одна реплика назначается лидером, а остальные — followers. Все операции записи и чтения происходят с лидером, а followers синхронизируют данные асинхронно.

При падении лидера происходит автоматический failover: одна из синхронизированных followers становится новым лидером. Этот процесс контролирует встроенный компонент Kafka — **controller**, который поддерживает консенсус и следит за состоянием брокеров через **Zookeeper** (в Kafka 2.x) или через собственный координационный протокол в Kafka 3.x+ (KRaft mode).

## Ключевые моменты
- **Репликация** — асинхронная, с настройкой параметров `min.insync.replicas`, чтобы гарантировать, что запись считается успешной, только если данные подтверждены минимальным количеством реплик, что повышает надежность.
- **ISR (In-Sync Replicas)** — набор реплик, которые полностью синхронизированы с лидером. Только из этого набора выбирается следующий лидер при сбое.
- **Failover** происходит быстро (обычно миллисекунды — секунды), при этом новый лидер берет управление на себя без потери данных, если старая реплика была в ISR, иначе возможна потеря данных.

## Практический контекст
В реальных проектах Kafka широко применяется для обработки критичных данных с требованием 99.9% uptime. Репликация обеспечивает отказоустойчивость к сбоям узлов. Например, в продуктивных системах настроена репликация с тремя репликами и `min.insync.replicas=2`, чтобы минимизировать риск потери сообщений и обеспечить консистентность даже при сбое одного брокера. Мониторинг lag реплик через JMX и Prometheus помогает вовремя выявлять проблемы с репликацией.
