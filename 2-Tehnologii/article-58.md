# Какие основные сущности в Apache Kafka вы знаете и в чём её принцип работы?

**Основные сущности и принцип работы Apache Kafka**

- Kafka — распределённая **система потоковой передачи данных**
- Основные сущности:
  - **Producer** — отправляет данные (сообщения)
  - **Consumer** — читает данные, подписан на темы
  - **Topic** — логическая категория для сообщений
  - **Partition** — часть топика, обеспечивает масштабируемость и параллелизм
  - **Broker** — сервер Kafka, хранит данные партиций
  - **ZooKeeper** (в классических версиях) — хранит метаданные и координирует кластер
- Принцип работы:
  - Producer пишет сообщения в топики
  - Сообщения разбиваются на партиции, упорядочены в каждый
  - Consumer читает сообщения из партиций, контролируя оффсеты
  - Партиции реплицируются для отказоустойчивости
- Kafka обеспечивает **высокую доступность**, масштабируемость и гарантии порядка внутри партиции
- Применяется для **стриминга**, логирования и ETL в реальном времени

# Подробный ответ
## Основной ответ
**Apache Kafka** — это распределённая платформа потоковой передачи данных, работающая по модели *publish-subscribe*. Основные сущности в Kafka — это **topic (топик)**, **partition (партиция)**, **producer (производитель)**, **consumer (потребитель)**, **broker (брокер)** и **zookeeper/kafka controller**. Работа Kafka основана на надежном размещении сообщений в топиках, где они разбиваются на партиции для параллелизма и масштабируемости. Продюсеры записывают сообщения в топики, а консьюмеры выбирают их для обработки с возможностью гарантировать порядок и доставку.

## Ключевые моменты
- **Topic и Partition**: Топик — это логическая категория сообщений. Каждый топик делится на несколько партиций, что позволяет масштабировать обработку и обеспечивать упорядоченную запись внутри каждой партиции.
- **Producer и Consumer**: Продюсеры отправляют события в партиции топиков, а консьюмеры читают данные из них, при этом консьюмер может быть частью группы для балансировки нагрузки и параллелизма.
- **Broker и Zookeeper (Kafka controller)**: Broker — сервер Kafka, хранящий партиции и обслуживающий запросы. Zookeeper раньше управлял метаданными и состоянием кластера, но в последних версиях Kafka постепенно переходит на собственный Kafka Controller для управления кворумом.
- **Основной принцип работы** — append-only лог с гарантией упорядоченности внутри партиции, поддержка репликации для устойчивости к ошибкам, а также consumer offset'ы для отслеживания прогресса чтения.

## Практический контекст
Kafka широко используют для построения real-time аналитики, event-driven архитектур и интеграции микросервисов. Например, в системе онлайн-торговли топики могут хранить события заказов, которые потребляют разные сервисы (биллинг, логистика) с высокой пропускной способностью и низкой задержкой (~миллисекунды).
