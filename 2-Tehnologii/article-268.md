# Настройка воркеров: Supervisor / PHP-FPM в контексте очередей

**Настройка воркеров: Supervisor / PHP-FPM в контексте очередей**  
- очередь — фоновая обработка задач (PHP)  
- **Supervisor** — менеджер процессов для воркеров  
- контролирует запуск, рестарт воркеров, **автоматизация**  
- **PHP-FPM** — пул воркеров для обработки HTTP-запросов  
- для очередей PHP-FPM не подходит, используется для веб-запросов  
- очередь запускает воркеров CLI через Supervisor (или systemd)  
- настройка: количество воркеров, рестарт на сбои, логирование, **мониторинг**  
- оптимизация по нагрузке, RAM и CPU, чтобы избежать блокировок  
- практическое значение: стабильная и отказоустойчивая обработка задач в фоне

# Подробный ответ
## Основной ответ  
Настройка воркеров в контексте очередей часто требует выбора между использованием **Supervisor** и **PHP-FPM**. Оба инструмента обеспечивают управление процессами, но решают разные задачи и имеют существенно различающийся подход.

**Supervisor** — это универсальный менеджер процессов, который позволяет запускать, перезапускать и мониторить воркеры, обрабатывающие задачи из очередей (например, Laravel Queue, Symfony Messenger). Он отлично подходит для CLI-воркеров очередей, которые работают постоянно и в фоновом режиме.

**PHP-FPM** — FastCGI Process Manager, в основном оптимизирован для веб-запросов PHP. Он управляет пулом PHP-процессов, принимающих HTTP-запросы на фронтенде, но не предназначен для длительных CLI задач.

## Ключевые моменты  
- **Supervisor** позволяет легко контролировать воркеры: автоматический рестарт при падении, логирование, гибкая настройка числа процессов, приоритизация. Воркеры запускаются как независимые CLI процессы, что идеально для асинхронной обработки задач в очередях.  
- **PHP-FPM** управляет процессами, которые обрабатывают веб-запросы в режиме быстрых транзакций, с ограничением по времени выполнения. По умолчанию не подходит для постоянной работы воркеров очереди из-за ограничений времени и природу работы.  
- Часто в продакшене рекомендуется запускать очередь через **Supervisor**, так как это более стабильное и управляемое решение для долгоживущих фоновых процессов, в то время как **PHP-FPM** отвечает исключительно за HTTP-сервер.

## Практический контекст  
В Laravel или других PHP-фреймворках CLI воркеры (например, `php artisan queue:work`) обычно поднимаются через Supervisor с настройкой concurrency и логированием. Таким образом, при перезагрузке сервера или сбоев воркеры автоматически восстанавливаются. PHP-FPM остается сфокусированным на обслуживании веб-трафика, не влияя на асинхронную обработку. Такая разграниченность повышает отказоустойчивость и масштабируемость системы.
