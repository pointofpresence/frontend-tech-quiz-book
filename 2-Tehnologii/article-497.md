# Что такое CORS (Cross-Origin Resource Sharing), и как он относится к встроенным элементам типа `iframe`, `img`, и `script`?

Категория: Технологии

### Краткий ответ

**Что такое CORS и как он связан с `iframe`, `img`, `script`?**

- CORS — механизм безопасности браузера для **контроля доступа к ресурсам с другого домена**  
- Срабатывает, когда скрипты делают **кросс-доменные запросы** (XHR, fetch)  
- `<script>` и `<img>` — **автоматически могут загружать ресурсы с других доменов без CORS**  
- `<iframe>` — загружает документы с другого домена, но для доступа к содержимому нужно соблюдать **политику одного источника (SOP)**  
- CORS защищает от **попыток JavaScript получить данные с другого домена без разрешения**  
- Заголовки CORS (Access-Control-Allow-Origin) определяют, какие домены разрешены к вызову API или чтению данных  
- Практически: `<script>` и `<img>` не используют CORS для загрузки, но `<iframe>` ограничен политикой безопасности и CORS важен для AJAX внутри него  

Итог: CORS регулирует **программный доступ к чужим ресурсам**, но встроенные теги `<img>` и `<script>` работают иначе — загружают контент без CORS, `<iframe>` ограничен политиками безопасности и требует CORS для взаимодействия внутри.

### Подробный ответ
#### Основной ответ
**CORS (Cross-Origin Resource Sharing)** — это механизм безопасности браузеров, который регулирует, как веб-страницы с одного домена могут запрашивать ресурсы с другого домена (кросс-доменные запросы). Он помогает предотвратить несанкционированный доступ к ресурсам между разными origin, реализуя контроль через HTTP-заголовки (например, `Access-Control-Allow-Origin`). Без CORS браузер блокирует большинство кросс-доменных AJAX-запросов по политике Same Origin Policy.

Относительно **встроенных элементов** `<iframe>`, `<img>`, `<script>` CORS работает иначе: их загрузка ресурсов из других доменов обычно разрешена без ограничений, так как это не AJAX-запросы, а загрузка встроенных ресурсов. Однако, если скрипт пытается делать XHR/fetch запросы, CORS применяется.

#### Ключевые моменты
- **`iframe`**: загрузка внешнего домена разрешена, но доступ к содержимому iframe из другого origin заблокирован (SOP). CORS не контролирует загрузку, а доступ к DOM — запрещён.
- **`img` и `script`**: можно загружать ресурсы без CORS, но для `<script>` важна политика безопасности (например, CORS влияет на поведение `crossorigin` атрибута, ответственный за загрузку с разрешением доступа к ошибкам или кэшированию).
- Для динамических запросов AJAX/fetch CORS строго необходим, и сервер должен выставлять разрешающие заголовки.
- Иногда используется атрибут `crossorigin` на тегах для указания, как обрабатываются CORS-запросы (например, `crossorigin="anonymous"`).

#### Практический контекст
В React-приложениях или SPA при запросах API с фронтенда на другой домен сервер должен вернуть корректные CORS-заголовки, иначе браузер заблокирует ответ. Но для загрузки изображений через `<img>` или скриптов из CDN часто CORS не мешает, лишь влияет на доступ к ошибкам и кэшированию. Встраивание стороннего контента через `<iframe>` требует учитывать sandbox и политику происхождения при манипуляции содержимым.
