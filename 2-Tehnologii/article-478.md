# Почему использование []rune эффективнее конкатенации строк?

**Почему использование []rune эффективнее конкатенации строк?**

- строки в Go **неизменяемы** (immutable)
- конкатенация создает новые копии строк — **затратно по памяти**
- []rune — **массив изменяемых символов (runes)**
- манипуляции с []rune — **операции слайсов, модификация без копирования**
- преобразование []rune в строку происходит **один раз** в конце
- итог — **меньше аллокаций, лучше производительность** при множественных конкатенациях
- подходит для **работы с юникод-символами** (runes) и динамического построения строки

# Подробный ответ
## Основной ответ
Использование **среза []rune** для построения строки часто эффективнее, чем конкатенация строк напрямую, потому что строки в Go являются неизменяемыми. При конкатенации происходит выделение новой памяти и копирование существующих данных, что существенно замедляет процесс при множествах операций. Срезы []rune, наоборот, позволяют накапливать символы в динамическом массиве без частых аллокаций, а затем преобразовать их в строку один раз в конце.

## Ключевые моменты
- **Строки в Go неизменяемы** — конкатенация через оператор `+` при каждом шаге создаёт новую строку и копирует данные, что приводит к квадратичной по времени сложности при множестве операций.
- **Использование []rune (или []byte)** предоставляет возможность увеличивать буфер без копирований при каждом добавлении, так как срезы управляют емкостью и только при её превышении происходит reallocation.
- В современной практике рекомендуют применять **strings.Builder** (в Go 1.10+) для эффективного построения строк, но если нужен контроль над Unicode символами, []rune удобен для работы с кодовыми точками.

## Практический контекст
В реальных проектах, например при парсинге текста или обработке Unicode, сначала накапливают []rune для корректной работы с многобайтовыми символами, а затем итоговую строку создают единожды. Это снижает нагрузку на сборщик мусора и улучшает производительность при большом количестве операций конкатенации.
