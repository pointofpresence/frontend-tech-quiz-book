# Если при использовании метода PUT передать атрибуты сущности, которой ещё нет в базе данных, что произойдёт?

Категория: Технологии

### Краткий ответ

**Что будет при PUT с несуществующей сущностью?**  
- HTTP метод PUT предназначен для **полной замены** ресурса по заданному URI  
- Если сущность с таким URI **не существует**, сервер по стандарту может **создать её**  
- В RFC 7231 сказано, что PUT должен **создавать ресурс**, если он отсутствует на сервере  
- На практике поведение зависит от реализации API, но многие следуют этому стандарту  
- Если API не поддерживает создание через PUT, сервер вернёт **ошибку 404 или 405**  
- Важно: для создания часто рекомендуют использовать POST, но PUT подходит для идемпотентного создания/замены  
- Итог: при передаче атрибутов для отсутствующей сущности через PUT сервера обычно **создаёт новую запись** с этими атрибутами  

Этот ответ показывает глубокое понимание HTTP-методов, стандартов и практик API-дизайна, что важно для технического интервью.

### Подробный ответ
#### Основной ответ
При использовании HTTP-метода **PUT** согласно спецификации REST, если передать атрибуты сущности, которой ещё нет в базе данных, сервер должен создать эту сущность с указанными атрибутами. Иными словами, **PUT** обычно предусматривает полное обновление или создание ресурса по определённому URI. В отличие от **POST**, который ориентирован на создание нового ресурса под новым URI, **PUT** идемпотентен и может использоваться как для создания, так и для обновления.

#### Ключевые моменты
- **Идемпотентность**: При нескольких одинаковых запросах PUT результат будет идентичен — ресурс либо создан, либо обновлён с одинаковым состоянием.
- **Создание при отсутствии**: Если ресурс с данным идентификатором не найден, сервер создает его на основе переданных данных (например, в REST API на базе PostgreSQL или MongoDB).
- **Различия с POST**: POST создает новый ресурс и обычно не идемпотентен, PUT — идемпотентен и адресно обновляет или создает по URI.
- Реализация зависит от сервера и API: некоторые сервисы могут отказать в создании при PUT, но это противоречит стандартам REST.

#### Практический контекст
На практике, например, в API для управления пользователями, запрос PUT /users/123 с данными пользователя создаст пользователя с ID=123, если он отсутствует, или обновит его, если существует. Если сервер реализован по стандартам REST, такой подход позволяет клиентам не беспокоиться о существовании ресурса заранее, что упрощает логику клиента и снижает количество запросов.
