# Какие виды коммуникаций между сервисами ты знаешь?

**Виды коммуникаций между сервисами**

- синхронная (HTTP REST, gRPC) — запрос-ответ, блокирует вызывающий сервис  
- асинхронная (сообщения в очередях, Kafka, RabbitMQ) — публикация и обработка без ожидания  
- событийнная — реакция на события, декуплированные через Event Bus или системы публикации-подписки  
- потоковая (streaming) — постоянная передача данных (Kafka Streams, WebSocket)  
- бинарные протоколы (gRPC) для высокой производительности и строгой типизации  
- использование API Gateway для унификации вызовов и балансировки нагрузки  
- выбор протокола зависит от требований к задержкам, надежности и масштабированию  

Комбинация зависит от бизнес-логики и архитектуры, чтобы обеспечить отказоустойчивость и масштабируемость.

# Подробный ответ
## Основной ответ
Существует несколько основных видов коммуникаций между микросервисами, которые можно разделить на **синхронные** и **асинхронные**. Выбор зависит от требований к времени отклика, надежности, масштабируемости и архитектурных предпочтений.

## Ключевые моменты
- **Синхронная коммуникация** — обычно реализуется через протоколы типа HTTP/REST или gRPC. Клиент делает запрос и ждет ответ, что подходит для случаев, когда нужен быстрый feedback. В React 18+ и современных backend-системах gRPC с Protobuf обеспечивает меньшую задержку и эффективное бинарное взаимодействие.
- **Асинхронная коммуникация** — используется брокеры сообщений (Kafka, RabbitMQ, NATS), где сервисы публикуют события или ставят задачи в очередь, а другие подписываются и обрабатывают по мере готовности. Такой подход повышает отказоустойчивость и масштабируемость, убирает tight coupling.
- **Event-driven communication** — отдельный подвид асинхронной, где микросервисы обмениваются событиями через event bus или event streaming platform (например, Kafka Streams, Debezium), что позволяет интегрировать сервисы без прямых вызовов.

## Практический контекст
В реальных проектах часто комбинируют оба подхода: синхронные REST/gRPC вызовы для user-facing операций с latency около 50-100ms и асинхронные события для фоновых задач, обработки данных и обеспечения eventual consistency. Например, заказ в e-commerce создается через REST API, а последующая рассылка уведомлений и обновление отчетности происходит через асинхронные события.
