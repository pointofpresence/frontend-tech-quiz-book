# Hash-индекс — когда использовать, какие операции поддерживает (только equality)?

**Hash-индекс — когда использовать, какие операции поддерживает (только equality)?**

- тип индекса для быстрого поиска по ключу
- использует хеш-функцию для доступа к данным
- эффективен для операций **равенства** (equality): `=`
- не поддерживает диапазонные запросы (`<`, `>`, `BETWEEN`)
- быстрый доступ, примерно **O(1)** на поиск и вставку
- подходит для точного поиска по уникальным или часто фильтруемым значениям
- используется в базах данных для ускорения выборок по равенствам, например, поиск по ID или уникальному ключу

# Подробный ответ
## Основной ответ
**Hash-индекс** — это структура данных для быстрого поиска записей по ключу с помощью хеширования. Основное применение — операции точного сравнения (equality), где запросы типа `WHERE column = value` выполняются очень эффективно благодаря прямому доступу к хешу. В отличие от B-tree индексов, hash-индексы не подходят для диапазонных или сортировочных операций.

## Ключевые моменты
- **Поддерживаемые операции**: hash-индексы оптимизированы только для операций точного соответствия (`=`). Запросы с операторами `>`, `<`, `LIKE` (кроме точно совпадающих), или диапазонные (`BETWEEN`) работать не будут, индекс не ускорит такие операции.
- **Производительность**: при точном поиске hash-индексы обеспечивают амортизированное время доступа O(1), что часто быстрее B-tree, но они чувствительны к коллизиям и требуют хорошей хеш-функции.
- **Использование в реальных СУБД**: в PostgreSQL hash-индексы долго были нестабильны, начиная с версии 10 их стабильность улучшена, но всё равно часто предпочтительнее B-tree за универсальность. Hash-индексы эффективны, когда в таблице очень много равенств и нет необходимости в сортировке.

## Практический контекст
Типично используют hash-индексы для колонок с частыми запросами по точному совпадению — например, для поиска по уникальному идентификатору или токену. Когда запросы требуют сортировки или диапазонов, лучше использовать B-tree. В системах с heavy equality search и огромным объёмом данных hash-индексы помогают снизить latency до порядка единиц миллисекунд.
