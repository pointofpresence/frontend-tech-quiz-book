# Какие подходы знаешь для обновления данных на странице в реальном времени?

### Категории

Async, Frontend, Real-time, Networking, Event loop, Design system, Performance, Practical tasks

### Краткий ответ

**Обновление данных на странице в реальном времени**

- клиент-серверное взаимодействие  
- **Polling** — периодические запросы к серверу  
- **Long Polling** — удержание соединения до новых данных  
- **WebSocket** — двунаправленное постоянное соединение  
- **Server-Sent Events (SSE)** — однонаправленные серверные обновления  
- технологии: REST API, Socket.IO, EventSource  
- выбирают по нагрузке, задержкам и сложности  
- применение: чаты, ленты новостей, игры, дашборды

### Подробный ответ
#### Основной ответ  
Для обновления данных на странице в реальном времени обычно используют несколько основных подходов, каждый из которых подходит под разные требования по производительности, сложности и масштабируемости. Ключевые технологии — **Polling**, **Long Polling**, **WebSockets** и **Server-Sent Events (SSE)**.  

#### Ключевые моменты  
- **Polling** — периодические запросы клиента к серверу с фиксированным интервалом (например, каждые 5 секунд). Очень простой, но неэффективный при малом числе изменений и создает избыточную нагрузку.  
- **Long Polling** — усовершенствованный вариант, где HTTP-запрос удерживается открытым на сервере до появления новых данных, после чего сервер отвечает и клиент сразу же посылает следующий запрос. Меньше нагрузка, чем обычный polling, но всё равно используется HTTP-запросы.  
- **WebSockets** — двунаправленный, постоянно открытый TCP-сокет между клиентом и сервером (в React 18+, Node.js и других), позволяющий серверу мгновенно отправлять обновления без дополнительного опроса. Идеально подходит для сложных real-time приложений (чат, игры, дашборды). Требует серверной поддержки и дополнительных усилий по масштабированию.  
- **Server-Sent Events (SSE)** — uni-directional канал от сервера к клиенту через EventSource API. Легко поддерживать, хорош для вещания обновлений с низкой частотой, но не поддерживает двунаправленное общение.  

#### Практический контекст  
В реальных проектах часто комбинируют подходы: например, WebSockets для критичных real-time данных (курсы валют, чаты), SSE для логов и уведомлений, polling — для менее чувствительных данных или для простых решений. При использовании WebSockets часто применяют библиотеки типа **Socket.IO** или **ws** (Node.js), а для масштабирования — Redis Pub/Sub или Kafka для передачи сообщений между нодами.  

Такой выбор зависит от задачи: если надо минимальное потребление ресурсов и быстрое обновление — WebSockets. Если приложения однонаправленные и несложные — SSE будет проще. Если возможностей сервера мало — длительный polling тоже приемлем.
