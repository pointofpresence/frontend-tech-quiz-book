# Как устроено клиент-серверное взаимодействие в протоколе OAuth 2.0?

### Категории

Authentication, Interview, Authorization, API, Networking, Live coding, Client-server, Security

### Краткий ответ

**Как устроено клиент-серверное взаимодействие в протоколе OAuth 2.0?**

- протокол авторизации для доступа к ресурсам
- три ключевых участника: клиент (приложение), сервер авторизации, ресурсный сервер
- клиент запрашивает **разрешение** у пользователя (ресурсного владельца)
- сервер авторизации выдаёт **токен доступа** после аутентификации пользователя
- клиент использует токен для доступа к ресурсному серверу
- токен подтверждает права доступа без передачи пароля
- обеспечивает **безопасный делегированный доступ** к API и ресурсам пользователя

### Подробный ответ
#### Основной ответ
**OAuth 2.0** — это протокол авторизации, который позволяет клиентским приложениям получать ограниченный доступ к ресурсам пользователя на сервере, не передавая напрямую его учетные данные. Клиент-серверное взаимодействие в OAuth 2.0 строится вокруг передачи и проверки **токенов доступа**, что повышает безопасность и управляемость авторизацией.

#### Ключевые моменты
- Взаимодействие включает 4 основных участника: **Resource Owner (Пользователь)**, **Client (Клиентское приложение)**, **Authorization Server (Сервер авторизации)** и **Resource Server (Сервер ресурсов)**. Клиент запрашивает у сервера авторизации разрешение от имени пользователя, а после получает токен доступа.
- Основные шаги взаимодействия: клиент получает у пользователя разрешение, отправляет запрос на сервер авторизации с кодом или другими учетными данными, сервер выдает **access token** (и опционально **refresh token**), клиент использует его для доступа к ресурсу.
- Протокол предусматривает несколько **грантов (flows)** — Authorization Code Flow (для серверных приложений с backend), Implicit Flow (для SPA, но устаревший), Client Credentials Flow (для сервер-сервер без пользователя), Resource Owner Password Credentials (редко используется, небезопасен). Каждый flow решает задачи безопасности, удобства и контекста приложения.

#### Практический контекст
В реальных проектах, например, в React 18 SPA с OAuth 2.0 часто используют Authorization Code Flow с PKCE для защиты от атак перехвата кода. Сервер авторизации, как правило, реализован на основе OpenID Connect (надстройка над OAuth 2.0), например с помощью Keycloak или Auth0. Access token хранится короткое время (например, 5-15 минут), refresh token — дольше для обновления без повторного логина. На Resource Server устанавливаются проверки токенов (JWT или introspection) для доступа к API.
