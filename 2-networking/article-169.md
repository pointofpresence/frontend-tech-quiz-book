# При каких механизмах авторизации существует опасность CSRF атаки?

### Категории

Authorization, Frontend, CSRF, Networking, Web, Security

### Краткий ответ

**При каких механизмах авторизации существует опасность CSRF атаки?**

- Контекст: безопасность веб-приложений, авторизация  
- CSRF атакует сессии, основанные на автоматической отправке аутентификационных данных браузером  
- Уязвимы механизмы с **cookie-based** аутентификацией, где браузер автоматически отправляет cookies  
- Опасность возникает при использовании **cookie сессий** без дополнительных защит (CSRF-токенов)  
- Менее уязвимы **токены в заголовках** (например, Authorization Bearer) — требуют явной передачи  
- OAuth и JWT уязвимы, если токен автоматически добавляется браузером в запросы  
- Ключ: наличие автоматического и скрытого отправления аутентификации браузером  
- Практическое решение: CSRF-токены, SameSite cookies, проверка Origin/Referer  

Итого: CSRF актуален при **cookie-based** авторизации без защиты, где браузер автоматически добавляет аутентификационные данные в запрос.

### Подробный ответ
#### Основной ответ
Опасность **CSRF (Cross-Site Request Forgery)** возникает в тех механизмах авторизации, где браузер автоматически отправляет идентификационные данные — особенно с использованием **куки (cookies)**, так как браузер присылает их при каждом запросе к соответствующему домену без дополнительной проверки. В частности, уязвимы механизмы авторизации на основе **cookie-сессий** и **HTTP Basic Authentication с сохранёнными в браузере учетными данными**.

#### Ключевые моменты
- **Cookie-based авторизация**: куки автоматически прикрепляются к HTTP-запросам, поэтому злоумышленник с другого сайта может инициировать запросы от имени пользователя, если нет дополнительных защит (например, CSRF-токенов).
- **Token-based авторизация в HTTP Header** (например, Bearer Token в Authorization header) менее подвержена CSRF, потому что JavaScript другого домена не имеет доступа к таким токенам из-за политики CORS, и токен не отправляется автоматически, а должен передаваться явно.
- **CSRF защищают** через использование уникальных и проверяемых CSRF-токенов, SameSite куки (начиная с современных браузеров), а также проверку Origin/Referer заголовков.
  
#### Практический контекст
В современных веб-приложениях сессии чаще всего хранятся в куки, потому CSRF-атаки остаются актуальной проблемой, особенно без использования флага `SameSite=Lax/Strict` или CSRF-токенов в формах. В API, где авторизация реализована через заголовки `Authorization` с токенами (например, JWT в локальном хранилище), риск CSRF существенно снижается, но растёт угроза XSS, поэтому важно применять комплекс мер безопасности.
