# Как работает идемпотентность под капотом — где и как хранятся ключи, с чем сравнивается запрос?

### Категории

SOLID, Security, API, Idempotency, Architecture, Web, Database, Technical

### Краткий ответ

**Как работает идемпотентность: механизм и хранение ключей**

- идемпотентность — гарантия одинакового результата при повторных вызовах
- ключи идемпотентности обычно сгенерированы клиентом или сервером, уникальны для операции
- ключи хранятся в **persistent store**: базы данных, кеши (Redis, Memcached) или специализированные хранилища
- при запросе сначала ищется ключ; если найден — возвращается сохранённый ответ или статус
- если ключ отсутствует — операция выполняется, результат сохраняется с ключом
- сравнение запроса идёт по ключу, а не по содержимому запроса для эффективности
- важна механика **тайм-аутов** и **очистки старых ключей** для предотвращения ростос хранения

**Итог:** ключи в durable storage, проверка по ключу — позволяет повторять операции без дублирования эффекта и гарантирует консистентность.

### Подробный ответ
#### Основной ответ
**Идемпотентность** — это свойство операции, при котором её повторное выполнение не изменяет результат системы сверх первоначального. В веб-разработке идемпотентность важна для безопасного повторения запросов без риска неправильных побочных эффектов, например, при сбоях сети. Под капотом это обычно реализуется через хранение **уникальных ключей идемпотентности** (идентификаторов запросов) и сравнение их с уже обработанными запросами.

Когда приходит новый запрос с идемпотентным идентификатором (например, `Idempotency-Key` в HTTP-заголовке), сервер смотрит в **специальное хранилище** (чаще всего в базу данных или кэш вроде Redis), есть ли уже для этого ключа сохранён результат ранее выполненной операции. Если ключ найден, возвращается сохранённый ответ, а новое выполнение не происходит. Если ключа нет — запрос обрабатывается, результат сохраняется вместе с ключом, и возвращается клиенту.

#### Ключевые моменты
- **Хранение ключей и результата:** ключи идемпотентности обычно сохраняются в быстром хранилище с TTL (временем жизни), чтобы ограничить размер базы. Это может быть SQL таблица с колонками `idempotency_key, response_data, status, created_at` или Redis с хранимыми сериализованными ответами.
- **Сравнение запросов:** ключи обычно рассчитаны клиентом и уникальны для конкретной операции, поэтому сравнивается именно ключ, а не все параметры запроса. Это исключает необходимость глубокого сравнения тела запроса.
- **Атомарность и согласованность:** чтобы избежать race conditions, операции записи ключа и ответа должны выполняться атомарно, например, через транзакции в БД или Lua-скрипты в Redis. Иначе два одинаковых запроса могут быть выполнены параллельно.
- **Управление сроком жизни:** ключи и результаты хранятся ограниченное время (например, несколько часов или суток) с помощью TTL, после чего удаляются для освобождения ресурсов.

#### Практический контекст
В реальных системах (например, в Stripe API или платежных шлюзах) клиенты задают `Idempotency-Key` при оплате, чтобы при повторных запросах избежать двойного списания. Ключ и ответ сохраняются в Redis с TTL ~24 часа. За счёт этого при повторном запросе запрос мгновенно отклоняется с тем же ответом, что гарантирует консистентность и безопасность без дополнительных операций.
