# Что такое VACUUM в PostgreSQL

### Категории

Profiling, Concurrency, Performance, Backend, Database, Optimization, Fault tolerance, Technical, SQL

### Краткий ответ

**Что такое VACUUM в PostgreSQL**

- **Управление памятью** и **удаление "мертвых" строк** (dead tuples)
- Реализует **чистку и сжатие** таблиц после операций UPDATE/DELETE
- Обновляет **статистику** для оптимизатора запросов
- Предотвращает рост **раздувания (bloat)** таблиц и индексов
- Обеспечивает **доступность** видимых данных и поддерживает MVCC
- Имеет **автоматический** и ручной режимы (autovacuum/manual)
- Критически важен для **производительности** и целостности БД

### Подробный ответ
#### Основной ответ
**VACUUM** в PostgreSQL — это команда очистки, которая удаляет "мёртвые" строки из таблиц, освободив место для повторного использования и предотвращая рост размера базы данных. PostgreSQL использует **MVCC (Multiversion Concurrency Control)**, и при обновлении или удалении строки они не стираются сразу, а помечаются как устаревшие (dead tuples). VACUUM очищает эти устаревшие записи, освобождая дисковое пространство и поддерживая производительность.

#### Ключевые моменты
- **MVCC и мёртвые строки**: Каждое изменение создаёт новые версии строк, чтобы обеспечить изоляцию транзакций; VACUUM удаляет старые версии, которые больше не нужны.
- **Виды VACUUM**: обычный (стандартный) и **VACUUM FULL** — второй реорганизует таблицу, физически сжимая её, но вызывает блокировки и более длительное выполнение.
- **Автоматизация**: В PostgreSQL есть **autovacuum daemon** — фоновый процесс, который автоматически запускает VACUUM по необходимости с учётом статистики, что критично для стабильной работы системы.
- **Анализ статистики** после VACUUM (особенно ANALYZE) улучшает планирование запросов оптимизатором.

#### Практический контекст
В реальных проектах регулярный autovacuum поддерживает размер баз данных на приемлемом уровне и поддерживает низкую задержку запросов, например в PostgreSQL 14+ autovacuum настроен более агрессивно для OLTP систем с большим количеством обновлений. При больших таблицах с интенсивной записью могут запускать ручной VACUUM FULL на оффлайн периодах для дефрагментации.
