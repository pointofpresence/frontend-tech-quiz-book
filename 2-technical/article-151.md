# Как работают JWT-токены и зачем они нужны?

### Категории

SOLID, Technical, Authentication, Database, Architecture, API, Security

### Краткий ответ

**Как работают JWT-токены и зачем они нужны?**

- JWT (JSON Web Token) — стандарт для компактного, URL-безопасного токена аутентификации  
- Структура: header (тип, алгоритм) + payload (данные, claims) + signature (подпись)  
- Подпись создаётся с помощью секретного ключа или приватного ключа (HMAC / RSA) для проверки подлинности  
- Клиент хранит токен, отправляет его в HTTP-заголовке (обычно Authorization: Bearer)  
- Сервер проверяет подпись и извлекает данные без запроса к БД — **stateless аутентификация**  
- Основное назначение — безопасная передача информации о пользователе между клиентом и сервером  
- Позволяют масштабировать приложения без состояния сессии на сервере  
- Используются для аутентификации, авторизации и передачи короткоживущих данных (claims)  

Итог: JWT — безопасный, масштабируемый механизм **stateless аутентификации и авторизации** через зашифрованные, самодостаточные токены.

### Подробный ответ
#### Основной ответ
**JWT (JSON Web Token)** — это компактный, URL-безопасный токен в формате JSON, который используется для передачи информации между клиентом и сервером. Основная цель JWT — безопасная аутентификация и авторизация пользователей без необходимости хранить сессию на сервере. JWT состоит из трёх частей: заголовка (header), полезной нагрузки (payload) и подписи (signature), каждая часть кодируется в Base64 и разделяется точками.

#### Ключевые моменты
- **Структура**: header содержит тип токена и алгоритм подписи (например, HMAC SHA-256), payload — пользовательские данные или claims (например, userId, role, срок жизни), signature подтверждает подлинность и целостность токена.
- **Без состояния**: благодаря тому, что вся информация хранится в самом токене, сервер не хранит сессии, что упрощает масштабирование и уменьшает нагрузку на сервер.
- **Безопасность**: подделать токен практически невозможно без секретного ключа; при использовании JWT важно использовать HTTPS для защиты от MITM атак и правильно настраивать срок жизни токена и методы обновления.

#### Практический контекст
В рективных приложениях на React 18+ популярно использовать JWT для хранения токена в HttpOnly cookie или localStorage (с осторожностью), а на сервере (например, Node.js с Express) с помощью middleware проверять подпись и права доступа. Это обеспечивает удобный stateless механизм аутентификации с производительностью ~милисекунды при валидировании.
