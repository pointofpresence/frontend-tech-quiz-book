# Какие типы авторизации для API вам известны?

### Категории

Authorization, Technical, Authentication, API, Security

### Краткий ответ

**Типы авторизации для API**

- Basic Auth: передача логина и пароля в заголовке (base64)
- Bearer Token: токен доступа в заголовке Authorization
- OAuth 2.0: делегированная авторизация с токенами (access, refresh)
- API Key: уникальный ключ, часто в параметрах запроса
- JWT (JSON Web Token): самодостаточный токен с данными пользователя
- Digest Auth: улучшенный Basic, с хэшированием пароля
- OpenID Connect: надстройка над OAuth 2.0 для идентификации
- Применяется в зависимости от безопасности и удобства интеграции

### Подробный ответ
#### Основной ответ
Существует несколько основных типов **авторизации для API**, каждый из которых имеет свои особенности и сценарии применения. Наиболее распространённые — это **Basic Auth**, **Token-based Auth (например, JWT)**, **OAuth 2.0** и **API ключи**. Авторизация — это процесс проверки прав доступа пользователя или системы к ресурсу после аутентификации.

#### Ключевые моменты
- **Basic Auth** — передает логин и пароль в заголовках HTTP, работает на простых и малоопасных сценариях, но требует HTTPS, так как данные не шифруются.
- **Token-based авторизация (JWT)** — после аутентификации сервер выдает **JSON Web Token**, который клиент использует в запросах. Удобно для stateless архитектуры RESTful API, масштабируемо, поддерживает embeded claims и expiry.
- **OAuth 2.0** — стандарт для делегирования доступа, особенно для сторонних приложений. Позволяет безопасно выдавать временные токены доступа с ограниченными правами без передачи пароля.
- **API Keys** — простой механизм, где ключ используется как уникальный идентификатор клиента. Обычно его включают в заголовок или query-параметр. Хорошо подходит для сервисов с ограниченной безопасностью и простыми ограничениями.

#### Практический контекст
В современных проектах часто комбинируют **OAuth 2.0 с JWT** для управления доступом к микросервисам. Для внутреннего API — используются API ключи с rate limiting через Redis. В React или мобильных приложениях для аутентификации — типично применять Token-based авторизацию с refresh tokens для долгосрочной сессии.
