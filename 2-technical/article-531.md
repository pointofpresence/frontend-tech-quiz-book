# Что означает идемпотентность HTTP-методов? Какие методы считаются идемпотентными?

### Категории

HTTP, QA, API, Idempotency, Network, Integration, Test design, Web, Technical, Testing

### Краткий ответ

**Что означает идемпотентность HTTP-методов? Какие методы считаются идемпотентными?**

- идемпотентность — свойство операции не изменять результат при повторных вызовах  
- повторный запрос не должен иметь побочных эффектов, отличных от первого  
- важна для надёжности сети и корректного повторения запросов  
- **идемпотентные методы:** GET, PUT, DELETE, HEAD, OPTIONS, TRACE  
- GET — только чтение, не изменяет состояние  
- PUT — заменяет ресурс, одинаковый эффект при повторе  
- DELETE — удаляет ресурс, повторный вызов не изменит результат  
- HEAD, OPTIONS, TRACE — информационные, не меняют данные  
- неидемпотентные — POST (создание/модификация с эффектом)  
- идемпотентность критична для безопасного повторения запросов и работы прокси/кешей

### Подробный ответ
#### Основной ответ
**Идемпотентность HTTP-методов** означает, что многократное выполнение одного и того же запроса приведёт к одному и тому же результату и состоянию сервера, как если бы запрос был выполнен единожды. Это важно для обеспечения надёжности и предсказуемости API, особенно при повторных запросах из-за сетевых сбоев.

#### Ключевые моменты
- **Идемпотентные методы HTTP** — это **GET, PUT, DELETE, HEAD, OPTIONS, TRACE**. Каждый из них гарантирует, что несколько одинаковых запросов не изменят состояние ресурса дополнительно.
- Метод **GET** запрашивает данные без изменения состояния сервера, поэтому всегда идемпотентен.
- Метод **PUT** обновляет ресурс полностью и повторное выполнение создаст или заменит ресурс тем же способом.
- Метод **DELETE** при повторном вызове удаляет ресурс — если ресурс уже удалён, повторный DELETE не меняет состояние.
- Напротив, метод **POST** не является идемпотентным, так как при повторных вызовах может создавать новые ресурсы или выполнять другие операции, изменяющие сервер.

#### Практический контекст
Идемпотентность важна для систем с повторяющимися запросами или retry-механизмами (например, в микросервисах или REST API). Клиенты или прокси могут повторять запросы без риска некорректных изменений данных. В React 18 и облачных API это особенно критично для устойчивости и консистентности.
