# Что нужно оборачивать в using?

### Категории

Disposable pattern, Resource management, Memory management, Using statement, Technical

### Краткий ответ

**Что нужно оборачивать в using?**  
- работа с объектами, реализующими **IDisposable**  
- освобождение **неуправляемых ресурсов** (файлы, сокеты, соединения с БД)  
- автоматический вызов метода **Dispose()** по завершении блока  
- предотвращение утечек памяти и блокировок  
- повышение читаемости и безопасности кода  
- сокращает boilerplate для очистки ресурсов  
- ключевой паттерн в управлении **жизненным циклом ресурсов** в .NET

### Подробный ответ
#### Основной ответ
В **C#** ключевое слово **using** используют для автоматического управления временем жизни объектов, которые реализуют интерфейс **IDisposable**. В блоке `using` объект создаётся, а после выхода из области видимости автоматически вызывается метод `Dispose()`, освобождающий неуправляемые ресурсы — например, файловые дескрипторы, соединения с базой данных, сетевые сокеты и т.д.

#### Ключевые моменты
- В **using** нужно оборачивать объекты, которые используют **неуправляемые ресурсы** и реализуют **IDisposable**. Примеры: классы из `System.IO` (FileStream, StreamReader), подключения к БД (SqlConnection, DbContext), объекты для работы с графикой и др.
- Использование `using` гарантирует вызов `Dispose()` даже при возникновении исключения, что критично для предотвращения утечек ресурсов.
- В C# 8.0 и выше появился **using declaration** (без блока), который облегчает синтаксис, сохраняя те же гарантии освобождения ресурсов.

#### Практический контекст
В реальных проектах, например, при работе с базой данных через **Entity Framework Core 6+**, вы оборачиваете контекст `DbContext` в `using`, чтобы корректно закрыть соединение и освободить внутренние буферы. Также при работе с файлами или сетевыми потоками `using` помогает избежать блокировок и исчерпания системных ресурсов. Это практика стандарта профессиональной разработки на C#.
