# Какие способы общения между микросервисами?

### Категории

Microservices, Architecture, Communication, Message queue, Asynchronous, Backend, Database, Messaging, Queues, Technical

### Краткий ответ

**Какие способы общения между микросервисами?**

- HTTP/REST: синхронное, простое, широко поддерживается  
- gRPC: высокопроизводительный, бинарный протокол, подходит для быстрых вызовов  
- Message Broker (RabbitMQ, Kafka): асинхронное, через очереди, повышает устойчивость  
- Event-driven: микросервисы реагируют на события, снижает связность  
- Shared database: совместное использование БД (редко, антипаттерн)  
- WebSockets/Streaming: для реального времени, двунаправленная связь  
- GraphQL Gateway: агрегация данных из разных сервисов для удобства клиента  
Используются в зависимости от требуемой задержки, надежности и архитектуры системы.

### Подробный ответ
#### Основной ответ
Между микросервисами общение организуют двумя основными способами: **синхронное** и **асинхронное взаимодействие**. Синхронное построено на прямом запрос-ответ (например, HTTP/REST или gRPC), когда сервисы ждут ответ перед продолжением работы. Асинхронное основано на обмене сообщениями через брокеры (например, Kafka, RabbitMQ), позволяя сервисам работать независимо и обрабатывать сообщения в своем ритме.

#### Ключевые моменты
- **Синхронные коммуникации** часто реализуются по протоколам HTTP/REST или gRPC. REST распространён за счёт простоты и HTTP/JSON, gRPC обеспечивает высокую производительность и контрактное взаимодействие через protobuf с поддержкой потоков (streaming) с версии 1.0+. Минус — высокая связность, задержки передают задержки.
- **Асинхронные коммуникации** предполагают обмен сообщениями через брокеры или сервисы очередей. Это уменьшает связанность, повышает устойчивость и масштабируемость, разрешая сервисам обрабатывать события и задачи в собственном темпе.
- Модели обмена сообщениями включают **Publish/Subscribe**, **Event-driven архитектуру**, **Command/Queue** и **Sagas** для управления распространением событий и согласованностью в распределённой системе.
- Стоит учитывать компромисс между консистентностью и доступностью (CAP-теорема) и выбирать способ коммуникации исходя из требований по задержкам, отказоустойчивости и сложностям обработки ошибок.

#### Практический контекст
В проектах на React 18+ с backend на Kubernetes часто используют REST с OpenAPI для публичных API, а внутри системы — Kafka для передачи событий и RabbitMQ для команд с подтверждениями. Это обеспечивает гибкость и разделение ответственности между микросервисами, а также позволяет выдерживать SLA с latency ~50-100ms в синхронных вызовах и повышенной нагрузкой в асинхронных обработках.
