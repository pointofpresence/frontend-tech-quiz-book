# Можно ли в SQL в WHERE использовать агрегатные функции?

### Категории

Backend, Database, Query, Optimization, Technical, SQL

### Краткий ответ

**Можно ли в SQL в WHERE использовать агрегатные функции?**

- SQL, оператор **WHERE** фильтрует строки до группировки  
- агрегатные функции применяются к **группам** данных  
- для условий на агрегаты используется **HAVING**  
- **WHERE и HAVING работают на разных этапах**: WHERE – перед группировкой, HAVING – после  
- пример: WHERE фильтрует записи, HAVING – агрегированные результаты  
- использование агрегатов в WHERE вызовет **синтаксическую ошибку**  
- практическое правило: применять агрегаты только в HAVING для фильтрации агрегированных данных

### Подробный ответ
#### Основной ответ
В SQL **агрегатные функции** (например, `SUM()`, `COUNT()`, `AVG()`) **нельзя напрямую использовать в секции `WHERE`**, потому что `WHERE` фильтрует строки до этапа группировки данных. Агрегатные функции работают уже после того, как строки сгруппированы, то есть на уровне результатов группировки. Для фильтрации результатов агрегации используется секция **`HAVING`**, которая предназначена именно для работы с агрегатами.

#### Ключевые моменты
- **`WHERE` фильтрует отдельные строки до агрегации**, поэтому агрегатные функции там неприменимы.
- **`HAVING` фильтрует сгруппированные данные**, позволяя использовать агрегатные условия, например, `HAVING COUNT(*) > 1`.
- В некоторых СУБД можно использовать агрегатные функции в подзапросах, которые возвращаются в `WHERE`, например, через конструкции с коррелированными подзапросами.

#### Практический контекст
Типичный пример: если нужно выбрать клиентов, у которых сумма заказов больше 10000, мы пишем запрос с `GROUP BY` по клиенту и условием `HAVING SUM(order_amount) > 10000`. Это стандартный подход в **PostgreSQL**, **MySQL 8+**, **SQL Server** и других популярных СУБД. Попытка поместить агрегат напрямую в `WHERE` вызовет синтаксическую ошибку.
