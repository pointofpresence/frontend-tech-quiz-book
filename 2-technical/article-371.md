# Как обеспечить уникальность необязательного поля (например, рабочего телефона) с учётом NULL-значений?

### Категории

SOLID, Uniqueness, Security, Architecture, Backend, Database, Null values, Technical

### Краткий ответ

**Как обеспечить уникальность необязательного поля с NULL**

- Контекст: работа с БД и уникальными ограничениями
- NULL по стандарту SQL считается **не равен другому NULL**
- Уникальное ограничение не запрещает несколько строк с NULL
- Решения:
  - **Частичное уникальное индексирование**: индекс с условием IS NOT NULL (PostgreSQL, Oracle)
  - Использовать **уникальный индекс с COALESCE** (MySQL, SQL Server) — заменять NULL на фиктивное уникальное значение
  - Триггеры или проверка в приложении — **принудительный контроль** уникальности NULL
- Выбор зависит от СУБД и требований к производительности
- Практическое применение: исключить дублирование рабочих телефонов при вводе данных  
- Рекомендация: предпочитать частичное индексирование если поддерживается, для чистоты модели и производительности

### Подробный ответ  
#### Основной ответ  
Обеспечение уникальности необязательного поля с `NULL` в реляционных базах данных — классическая задача, связанная с трактовкой `NULL` в уникальных индексах. В большинстве СУБД (например, PostgreSQL, Oracle) уникальные ограничения позволяют несколько строк с `NULL` в данном поле, потому что `NULL` считается неопределённым значением и не сравнивается с другими `NULL`. То есть уникальность не распространяется на `NULL`.  

Чтобы обеспечить уникальность на уровне немножества ненулевых значений, нужно применять специальные приёмы:  
- В PostgreSQL (версии 9.2+) можно использовать **partial unique index**, индексирующий только строки, где поле не `NULL`.  
- В других СУБД можно попытаться заменить `NULL` на уникальный маркер через вычисляемое поле и сделать уникальный индекс на него, либо с помощью триггеров проверять дубли.  

#### Ключевые моменты  
- **Partial unique indexes:** В PostgreSQL такой индекс может выглядеть так:  
  ```sql  
  CREATE UNIQUE INDEX idx_unique_work_phone ON employees(work_phone) WHERE work_phone IS NOT NULL;  
  ```  
  Это позволяет множеству строк с `NULL` в `work_phone` иметь дубликаты, но запрещает дубликаты среди настоящих номеров.  
- **Альтернативы:** В MySQL (до 8.0) и других СУБД можно создать вычисляемый столбец, например, `COALESCE(work_phone, '##NULL##')`, и наложить уникальный индекс на него, но тогда нужно гарантировать, что этот маркер не пересекается с реальными значениями.  
- **Триггеры/проверочные процедуры:** Если СУБД не поддерживает partial indexes, можно написать триггер, который при вставке/обновлении проверяет наличие дубликата рабочего телефона, игнорируя `NULL`. Минус — усложнение логики и потенциальный performance hit.  

#### Практический контекст  
В реальных проектах на PostgreSQL используют partial indexes, чтобы минимизировать накладные расходы и поддерживать целостность на уровне базы. В MySQL 8+ можно использовать функциональные индексы с аналогичным эффектом. В NoSQL-базах или ORM-слоях часто реализуют эту проверку на уровне приложения или через транзакционные блокировки. Такой подход позволяет избежать дублирования данных с реальными номерами и сохраняет возможность оставлять поле пустым.
