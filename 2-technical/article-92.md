# Что должно передаваться на вход и возвращаться в ответе для каждого из CRUD-методов?

### Категории

Technical, Design, Database, CRUD, API

### Краткий ответ

**CRUD-методы: вход и ответ**

- **Create (POST):**
  - на вход: объект с данными для создания (валидный JSON)
  - в ответе: созданный объект с новым уникальным ID и статус 201

- **Read (GET):**
  - на вход: идентификатор ресурса или параметры фильтрации
  - в ответе: объект(ы) данных с запрошенной информацией, статус 200

- **Update (PUT/PATCH):**
  - на вход: идентификатор ресурса + объект с изменениями (PUT — полный, PATCH — частичный)
  - в ответе: обновлённый объект или подтверждение обновления, статус 200/204

- **Delete (DELETE):**
  - на вход: идентификатор ресурса
  - в ответе: подтверждение удаления, часто пустой ответ со статусом 204

- Важный момент: ответы содержат HTTP-статусы для однозначного результата операции  
- Формат обмена данных — обычно JSON для API  
- Логика валидации и ошибок осуществляется через соответствующие коды и сообщения

### Подробный ответ
#### Основной ответ
Для каждого из **CRUD-методов** (Create, Read, Update, Delete) важно четко определить, какие данные мы принимаем на вход и что возвращаем в ответе, чтобы обеспечить однозначное и предсказуемое поведение API.

- **Create**: на вход передается объект с необходимыми полями для создания ресурса (например, JSON с атрибутами). В ответ обычно возвращается созданный объект, дополненный уникальным идентификатором (ID) и метаданными (даты создания, версии).
- **Read**: на вход передается идентификатор ресурса (ID) или параметры фильтрации/поиска. В ответ возвращается запрашиваемый объект или список объектов, либо ошибка, если ресурс не найден.
- **Update**: на вход — ID ресурса и набор полей для изменения (частичный или полный объект). В ответ возвращается обновленный объект или подтверждение успешного обновления.
- **Delete**: на вход — ID ресурса. В ответ обычно возвращается подтверждение удаления (например, статус 204 No Content), иногда — удаленный объект или метаинформация.

#### Ключевые моменты
- Входные данные для Create и Update должны валидироваться, чтобы не допускать неконсистентных данных.
- В ответах желательно отдавать не только данные, но и метаинформацию (временные метки, версии) для упрощения дальнейшей работы с ресурсом, особенно в RESTful API.
- Обработка ошибок (например, 404 для Read/Delete несуществующего ресурса, 400 при неверных данных) должна быть стандартизирована и возвращать понятный клиенту формат.

#### Практический контекст
В современных REST API для каждого метода принято использовать четкий контракт JSON-схемы для входных данных и стандартизированный формат ответа. Например, в системе на PostgreSQL + Node.js (Express) для Create endpoint принимается JSON с обязательными полями, возвращается объект с ID и HTTP статус 201 Created, что облегчает интеграцию и автоматическое тестирование.
