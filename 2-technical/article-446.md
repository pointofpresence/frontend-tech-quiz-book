# Как бороться с дублированием данных в ClickHouse?

### Категории

Architecture, Backend, Database, Data duplication, Technical

### Краткий ответ

**Как бороться с дублированием данных в ClickHouse?**

- ClickHouse — колонко-ориентированная СУБД, оптимизированная для аналитики
- Нет встроенной уникальной индексации для предотвращения дублирования
- Основной метод — **Deduplication** на уровне вставок через **ENGINE ReplacingMergeTree**
- ReplacingMergeTree хранит несколько версий строк, во время мёржа оставляет последнюю по ключу
- Можно задать **версионное поле** для выбора нужной версии записи
- Для явного удаления дубликатов — использовать **INSERT SELECT DISTINCT** или агрегацию перед вставкой
- В архитектуре — избегать дублирующих источников данных, контролировать ETL pipeline  
- При обновлениях — использовать **TTL с реплейсерами** для очистки устаревших дубликатов
- Практика: комбинировать логику на уровне бизнес-приложений + ReplacingMergeTree для устойчивости
- Итог: борьба с дубликатами — баланс между архитектурой данных и механизмами ReplacingMergeTree, агрегации и контроля данных на входе

### Подробный ответ
#### Основной ответ
Дублирование данных в **ClickHouse** может возникать по разным причинам: повторная отправка данных, параллельные вставки без контроля, проблемы на уровне источника данных. Для борьбы с дублированием используют несколько подходов, включая контроль на уровне схемы, обработку данных на этапе вставки и оптимизацию ETL-процессов.

#### Ключевые моменты
- **Использование движка MergeTree с ключом первичного индекса (primary key)**. Он не предотвращает дубликаты напрямую, но позволяет быстро выполнять операции агрегирования и фильтрации для удаления дубликатов в запросах. Например, `GROUP BY` по уникальному ключу после выборки.
- **Deduplication на уровне вставки через специализированные механизмы**: если данные поступают через Kafka, используется движок `Kafka` совместно с `Materialized View` и `ReplacingMergeTree`, который позволяет хранить только последние версии записей по ключу (замена дубликатов). При вставке в `ReplacingMergeTree` с указанием `version`-столбца можно легко оставить только уникальные записи по ключу с максимальным значением версии.
- **Обработка дубликатов средствами ETL и batch-обновлений**. Перед загрузкой данных в ClickHouse часто проводят агрегацию и фильтрацию уникальных записей на уровне промежуточных систем (например, в Spark, Flink, или прямо в SQL). Это предотвращает попадание дубликатов в базу.
- **Использование TTL и логики очистки**. В некоторых случаях проводят периодическую очистку или архивирование старых дубликатов с помощью TTL-столбцов и запросов DELETE (начиная с версий ClickHouse 21+).

#### Практический контекст
Типичный production-подход — использовать **ReplacingMergeTree** с уникальным первичным ключом и version-колонкой. Например, для событий с идентификатором пользователя и временной меткой в качестве версии: повторные события с меньшим version будут автоматически "перезаписаны". В сочетании с Materialized Views и Kafka это дает надёжный пайплайн с минимальным дублированием. Также в ClickHouse 21+ появилась поддержка первичных ключей с deduplication на уровне вставки через специальные функции, что облегчает поддержание чистоты данных.
