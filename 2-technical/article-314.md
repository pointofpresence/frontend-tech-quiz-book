# Какое покрытие кода тестами (test coverage) считается достаточным?

### Категории

Test coverage, GC, Query optimization, Async, Indexes, Database, Decorators, Transactions, Caching, Vacuum, Technical, Testing

### Краткий ответ

**Какое покрытие кода тестами считается достаточным?**

- **Контекст:** качество покрытия тестами (тестирование ПО)  
- Идеальное покрытие — **100%** (цель, но редко достижимо)  
- Практически достаточным считают **70%-80%** (баланс затрат и риска)  
- Важно покрыть **ключевые бизнес- и критические части** кода  
- Покрытие не равнозначно качеству тестов (важнее **качество тестов**)  
- Высокое покрытие без проработки случаев — **ложное чувство безопасности**  
- Рекомендуется анализировать **код, важный для безопасности и логики**  
- Используют разные типы покрытия: **юнит-, интеграционные и e2e тесты**  
- Критическое: покрытие **регрессий и ошибок** для стабильности системы  
- В итоге: хорошее покрытие — **индекс качества, а не цель сама по себе**

### Подробный ответ
#### Основной ответ
Достаточное покрытие кода тестами — это не столько фиксированный процент, сколько понимание качества и полноты проверки критически важных частей приложения. Обычно минимальный ориентир — около **70-80%** покрытия, однако слепое стремление к высокому проценту (например, 100%) без учета качества тестов и бизнес-логики часто неэффективно. Важнее, чтобы покрывались основные сценарии, граничные случаи и наиболее уязвимые места в коде.

#### Ключевые моменты
- **Качество важнее количества**: 100% coverage с тестами, которые не проверяют реальные бизнес-правила или не выявляют баги, мало что дают. Рекомендуется сосредоточиться на покрытии критичных компонентов — моделей, сервисов, контроллеров.
- **Разные виды coverage и их значимость**: line coverage показывает, какие строки кода исполнялись, но branch coverage и path coverage важнее для проверки всех возможных условий и ветвлений.
- **Баланс с усилиями и сроками**: в промышленности типично 70–80% — оптимум для поддерживаемого уровня уверенности без перерасхода ресурсов. Покрытие меньше этого — риск возникновения скрытых багов, больше — требует значительных затрат и иногда дублирует проверки.

#### Практический контекст
В проектах на React обычно стремятся к 80% покрытию компонентов и логики с Jest и Testing Library, при этом критичные модули серверной части (Node.js, Java) покрываются интеграционными и unit-тестами тщательнее. Используются инструменты вроде Istanbul или Jacoco для контроля метрик, а Code Review и CI/CD pipeline блокируют слияние при падении ниже порога.
