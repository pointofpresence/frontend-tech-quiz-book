# Как следует поступать с полями в DTO, которые рассчитываются на лету на фронтенде или не хранятся напрямую в базе данных?

### Категории

Data transfer, Architecture, Database, Frontend, Technical

### Краткий ответ

**Обработка вычисляемых полей в DTO**

- ориентация на **разделение ответственности**: DTO передают данные, а вычисления — на фронтенде или слое бизнес-логики  
- **исключать вычисляемые поля из DTO**, если они не хранятся в базе и не нужны для передачи на бекенд  
- для вычисляемых полей создавать **отдельные свойства или методы** на фронтенде  
- если вычисляемые данные нужны на бекенде — использовать **отдельные View Models или Data Mappers**  
- избегать смешения **чистой модели данных** с представлением или логикой вычислений  
- преимущества: улучшает **поддерживаемость**, снижает **зависимости**  
- практическое применение: фронтенд рассчитывает значения клиентски, backend остается чистым и легковесным  

Итог: **DTO должны содержать только статичные данные, вычисления — в отдельном слое.**

### Подробный ответ
#### Основной ответ
При работе с **DTO (Data Transfer Object)** важен чёткий контракт между слоями приложения. Если поля рассчитываются на лету на фронтенде или не хранятся в базе, то в DTO их хранить напрямую не обязательно. Обычно такие поля либо исключают из DTO, либо включают как вычисляемые свойства, которые не сериализуются и не сохраняются. В API лучше явно разделять поля, приходящие из БД, и вычисляемые на клиенте, чтобы избежать путаницы и ошибок при передаче данных.

#### Ключевые моменты
- **Отделение данных и вычислений**: DTO должен отражать только те данные, которые реально передаются между слоями, а вычисляемые поля лучше формировать на клиенте или в бизнес-логике, чтобы не смешивать ответственность.
- **Валидация и сериализация**: такие поля можно объявлять как виртуальные (например, в TypeScript через getter) или опциональные, чтобы избежать ошибок при сериализации либо в бэкенде не включать их в модели базы данных.
- **Явность контрактов API**: хорошо документировать, какие поля клиент должен рассчитывать самостоятельно, чтобы не было дублей логики и не возникало конфликтов при обновлении данных.

#### Практический контекст
В проектах на React + Node.js нередко DTO из БД приходит без вычисляемых полей, а на фронте с помощью селекторов (например, в Redux Toolkit) добавляют вычисляемые значения. В backend можно использовать отдельные сериализаторы/мапперы для API-ответов, которые умеют формировать необходимые дополнительные поля либо возвращать строго DTO из модели. Это улучшает поддерживаемость, снижает избыточность и делает систему более предсказуемой.
