# Как защитить данные в LocalStorage от стороннего JS?

### Категории

Storage, Client-side, Technical, Browser, Web, Security

### Краткий ответ

**Как защитить данные в LocalStorage от стороннего JS?**

- Security context: client-side storage, web security  
- LocalStorage не шифруется и доступен любым скриптам в том же домене  
- Изоляция по origin (протокол + домен + порт) — основа защиты  
- Защита от XSS: самый критичный вектор для доступа к LocalStorage  
- Основные меры:  
  - Content Security Policy (CSP) для блокировки неавторизованных скриптов  
  - Тщательная очистка и экранирование пользовательского ввода (предотвращение инъекций)  
  - Минимизация использования inline-скриптов и сторонних библиотек  
- Шифрование данных в LocalStorage на клиенте (например, AES) — ограниченная защита, зависит от безопасности ключа  
- Рассмотреть альтернативы: хранение чувствительных данных на сервере или в HttpOnly куках  
- Итог: защита LocalStorage – это в первую очередь предотвращение внедрения стороннего JS через XSS, а не сам LocalStorage  

Эффективная защита требует комплексного подхода к безопасности фронтенда, а прямой защиты LocalStorage от сторонних скриптов без ограничения запуска JS практически нет.

### Подробный ответ
#### Основной ответ
**LocalStorage** — это клиентское хранилище данных в браузере, которое по дизайну доступно любому JavaScript-коду в рамках одного домена (origin). Это значит, что изолировать данные от сторонних скриптов на том же домене через LocalStorage напрямую невозможно, так как политика безопасности браузера (SOP) не позволяет ограничить доступ между скриптами одного origin. Следовательно, защита данных от стороннего JS требует комплексных мер.

#### Ключевые моменты
- **Изоляция по домену:** браузеры не предоставляют встроенных средств для ограничения доступа к LocalStorage внутри одного origin — все скрипты под этим origin'ом имеют равные права.
- **Шифрование данных:** дополнительная защита достигается через хранение зашифрованных данных в LocalStorage. При этом шифрование и дешифрование происходит в самом приложении, используя ключ, который не хранится в LocalStorage (например, через session или ввод пользователя). Это снижает риск утечки, даже если сторонний скрипт получит доступ к LocalStorage.
- **Content Security Policy (CSP) и Subresource Integrity (SRI):** настроенные корректно CSP позволяют ограничить загрузку и выполнение сторонних скриптов, сводя к минимуму возможность внедрения вредоносного кода, который может получить доступ к LocalStorage. SRI обеспечивает целостность загружаемых ресурсов.
- **Разделение логики:** использование iframe с другим origin или современного API, например, **IndexedDB** с контролем доступа, а также хранение чувствительных данных на сервере и их передача по защищенным каналам.
- **Отказ от LocalStorage для чувствительных данных:** рекомендуют хранить критичные данные в более защищённых механизмах, например, HttpOnly cookie (недоступных для JS) или в памяти, чтобы минимизировать риск утечек.

#### Практический контекст
В реальных проектах часто используют шифрование данных перед сохранением в LocalStorage, например, с помощью Crypto API в браузере (AES-GCM) и управлением ключами на стороне приложения. CSP-заголовки на уровне сервера запрещают запуск неподписанных скриптов, защищая от XSS-атак, которые часто становятся причиной доступа сторонних скриптов к LocalStorage. Для очень чувствительных данных предпочтительнее серверное хранение и получение через запросы с авторизацией.
