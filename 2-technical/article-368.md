# Почему нет настоящей параллельности в Python?

### Категории

Async, Concurrency, OOP, Python, Backend, GILs, Parallelism, Technical

### Краткий ответ

**Почему нет настоящей параллельности в Python?**

- ограничение **GIL (Global Interpreter Lock)** в CPython  
- GIL предотвращает **одновременное выполнение байт-кода** несколькими потоками  
- обеспечивает **безопасность доступа к объектам** без сложной синхронизации  
- поэтому многопоточные программы работают **конкурентно, а не параллельно**  
- настоящая параллельность возможна через **многопроцессность** (multiprocessing)  
- альтернативы: реализации без GIL — **Jython, IronPython, PyPy STM**  
- практическое применение: CPU-интенсивные задачи — многопроцессность, I/O — многопоточность  

Эссенция: GIL — причина отсутствия параллельного исполнения потоков в стандартном Python.

### Подробный ответ
#### Основной ответ
Настоящей параллельности в стандартном Python (CPython) практически нет из-за **Global Interpreter Lock (GIL)** — глобальной блокировки интерпретатора, которая позволяет одновременно выполнять в Python только один поток байткода, даже если у вас многоядерный процессор. Это связано с особенностями реализации CPython, где GIL гарантирует безопасность работы с внутренними структурами данных без сложной синхронизации.

#### Ключевые моменты
- **Global Interpreter Lock (GIL)** создан для упрощения управления памятью (счётчик ссылок) и предотвращения условий гонки на уровне интерпретатора, но ограничивает параллельность в многопоточных приложениях.
- Потоки в Python выполняются **конкурентно**, а не одновременно — переключение между ними происходит быстро, но в каждый момент времени активен только один поток.
- Параллельность достигается через **многопроцессность (multiprocessing)**, где для каждого процесса создаётся отдельный интерпретатор без общей GIL, что позволяет использовать все ядра CPU, но с большими затратами на IPC и память.
- Для вычислительно интенсивных задач часто используют модули на C, которые могут освобождать GIL во время выполнений тяжёлых операций (например, NumPy), или альтернативные реализации Python типа **Jython, IronPython** и **PyPy STM**, где GIL нет.

#### Практический контекст
В реальных проектах, если требуется высокая вычислительная производительность с использованием всех ядер, обычно выбирают архитектуру на основе **multiprocessing**, распределенных систем, либо пишут критичные части на С или Cython с ручным управлением GIL. Для I/O-ориентированных приложений в Python 3.7+ популярность набирает асинхронное программирование (asyncio), позволяющее эффективно масштабировать нагрузку без настоящей параллельности.
