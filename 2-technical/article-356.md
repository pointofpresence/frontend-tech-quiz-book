# Первичный ключ vs уникальный индекс — сходства, различия, когда какой применять?

### Категории

SOLID, Indexing, Security, Architecture, Web, Database, Unique index, Primary key, Technical

### Краткий ответ

**Первичный ключ vs уникальный индекс — сходства, различия, когда применять**

- Сходства: оба обеспечивают **уникальность** значений в столбце(ах)
- Первичный ключ (PK): **логический идентификатор** строки, ограничение целостности
- Уникальный индекс (Unique index): структурный объект БД для **ускорения поиска** и контроля уникальности
- Отличия:
  - PK: единственный на таблицу, автоматически создает кластерный индекс (в реляционных БД как SQL Server)
  - Unique index: может быть много на таблицу, не всегда кластерный
  - PK не допускает **NULL**, unique index допускает (зависит от СУБД)
- Применение:
  - PK для **основного идентификатора** сущности, обязательная уникальность и не-null
  - Unique index для задания дополнительных уникальных ограничений (например, email, телефон)
- Важно для дизайна: PK связывает таблицы через внешние ключи, unique индекс — для бизнес-правил уникальности

Итог: PK — для **идентификации строки**, unique индекс — для **дополнительных уникальных ограничений и ускорения запросов**.

### Подробный ответ
#### Основной ответ
**Первичный ключ (Primary Key)** — это уникальный идентификатор строки в таблице, который строго гарантирует уникальность и не допускает NULL-значений. При этом **уникальный индекс (Unique Index)** обеспечивает уникальность значений в одном или нескольких столбцах, но допускает наличие NULL, если СУБД это поддерживает. В сущности, первичный ключ и уникальный индекс оба обеспечивают уникальность, но различаются по семантике и ограничениям использования.

#### Ключевые моменты
- **Первичный ключ** — это концептуальное ограничение: он определяет "основной идентификатор" записи. Такая колонка (или сочетание колонок) обычно используется для построения связей между таблицами (foreign key). В большинстве СУБД по первичному ключу автоматически строится кластерный индекс (например, в SQL Server и InnoDB MySQL).
- **Уникальный индекс** — это физический объект базы данных, который поддерживает уникальность. Можно иметь несколько уникальных индексов в таблице, но только один первичный ключ.
- По умолчанию **первичный ключ не допускает NULL**, тогда как уникальный индекс может допускать (в PostgreSQL уникальность проверяется по не-NULL значениям, в Oracle NULL считается уникальным).
  
#### Когда что применять
- **Первичный ключ** применяют, когда нужно задать основной идентификатор для сущности — в таблице должен быть один и только один первичный ключ, который недвусмысленно идентифицирует строку.
- **Уникальные индексы** создают для столбцов, где важна уникальность, но данные не являются основным идентификатором (например, email пользователя или серийный номер продукта). Можно иметь несколько таких ограничений.
  
#### Практический контекст
В реальных проектах стандартный подход — определить **первичный ключ** как стабильный, чаще всего числовой ID (auto-increment или UUID), который гарантирует быстрый поиск и связь. Для бизнес-логики и валидации уникальности новшеств применяют **уникальные индексы**. Например, в PostgreSQL 14+ часто создают первичный ключ на ID, а уникальные индексы — на email или username, чтобы избежать дублирования, но разрешить NULL, когда это нужно.
