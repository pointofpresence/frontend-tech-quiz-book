# Какие альтернативы локальному storage существуют?

### Категории

Message brokers, Architecture, Storage, Backend, Database, Web applications, Queues, Technical

### Краткий ответ

**Альтернативы локальному storage**

- Cookie  
  - хранит данные с ограничением по размеру (~4Кб)  
  - автоматически отправляется на сервер при каждом запросе  
- Session Storage  
  - хранит данные в рамках одной сессии браузера  
  - очищается при закрытии вкладки или браузера  
- IndexedDB  
  - объектно-ориентированная DB в браузере  
  - хранит большие объемы структурированных данных  
  - асинхронный API, подходит для офлайн-приложений  
- Web SQL (устарел, не поддерживается современными браузерами)  
- Cache Storage (Service Workers)  
  - хранение ответов для офлайн-доступа  
  - используется в PWA  
- Серверное хранение (через API)  
  - данные сохраняются на сервере, доступ через REST/GraphQL  
- Memory Storage  
  - временное хранение в JS-памяти (не сохраняется при обновлении страницы)  

**Итог:** выбор зависит от объема, срока жизни данных, необходимости доступности на сервере и поддержки браузерами. IndexedDB – лучший выбор для больших и структурированных данных; cookie – для маленьких и тех, что должны идти на сервер; sessionStorage – для временных данных сессии.

### Подробный ответ
#### Основной ответ
Альтернативы **локальному хранилищу (Local Storage)** в веб-разработке включают ряд других клиентских и серверных решений для хранения данных, которые обеспечивают более гибкий, масштабируемый или безопасный доступ к данным. Основная причина использования альтернатив — ограничения Local Storage по объему (около 5 МБ), синхронному API и отсутствию шифрования.

#### Ключевые моменты
- **Session Storage** — очень похож на Local Storage, но данные живут только в рамках одной вкладки или сессии браузера и удаляются при закрытии вкладки. Используется, если нужно хранить данные временно.
- **IndexedDB** — объектно-ориентированная база данных на стороне клиента, поддерживающая хранение структурированных данных, большие объемы (до сотен мегабайт) и асинхронный API. Отлично подходит для offline-first приложений или хранения сложных данных.
- **Cookies** — подходят для хранения небольших данных, которые нужно передавать на сервер автоматически с каждым HTTP-запросом. Однако имеют ограниченный размер (обычно ~4 КБ) и низкую производительность.
- **Web SQL** — устаревшая технология, практически не используется и больше не поддерживается современными браузерами.
- **Service Workers + Cache Storage** — позволяют кэшировать HTTP-ресурсы и обеспечивать offline-доступ к ним, чаще используются для PWA.
- **Серверное хранение** (например, сессии на Redis, базы данных) — если данные критичны и должны быть централизованными и безопасными, тогда frontend хранит только токен или идентификатор, а сами данные — на сервере.
- **Web Storage API alternatives с шифрованием** — библиотеки, которые обеспечивают шифрование данных перед сохранением в Local Storage для повышения безопасности.

#### Практический контекст
В React-приложениях часто используют **IndexedDB** через библиотеки, например, Dexie.js для кэширования данных приложения и обеспечения работы offline. **Session Storage** подходит для хранения временных форм или промежуточных данных. В high-security приложениях предпочтительно использовать серверное хранение с минимальным хранением приватных данных на клиенте.
