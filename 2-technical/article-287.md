# Как на самом деле выполняется оператор UPDATE в PostgreSQL?

### Категории

GC, Testing, Async, Optimization, Indexes, Database, Decorators, Transactions, Caching, Vacuum, Technical, SQL

### Краткий ответ

**Как выполняется оператор UPDATE в PostgreSQL?**

- Система управления **реляционной БД**
- PostgreSQL не обновляет строки напрямую  
- Выполняет **MVCC**: старая строка помечается как удалённая  
- Создаётся **новая версия** строки с изменениями  
- Обеспечивается **консистентность** параллельных транзакций  
- Требует последующей **очистки (VACUUM)** старых версий  
- Поддерживает **триггеры**, индексы обновляются для новой строки  
- Используется для надежного и изолированного **обновления данных**

### Подробный ответ
#### Основной ответ
В **PostgreSQL** оператор **UPDATE** фактически не изменяет строку на месте. Вместо этого происходит создание новой версии строки (tuple) с обновлёнными значениями, а старая помечается как устаревшая (dead tuple). Такой подход реализует модель **MVCC (Multi-Version Concurrency Control)** — обеспечивает параллельную работу транзакций без блокировок чтения. По сути, UPDATE в PostgreSQL — это комбинация вставки новой строки и пометки старой на удаление.

#### Ключевые моменты
- При выполнении UPDATE PostgreSQL создаёт новую запись с обновлёнными данными и устанавливает для старой отметку "обновлённой" (infomation в заголовке tuple), при этом старая версия всё ещё существует до завершения vacuum.
- **MVCC** гарантирует, что другие транзакции видят полноценную консистентную картину данных, не блокируя чтение.
- Старые версии строк не удаляются сразу — за ними следит процесс **vacuum** который очищает мёртвые tuple, восстанавливая место на диске.
- Если обновляется строка с индексируемыми столбцами, PostgreSQL также создаст новые индексные записи для новой версии строки и оставит старые до очистки.
- Это поведение ведёт к потенциальному росту размера таблицы и индексов при частых обновлениях (явление bloat).

#### Практический контекст
Такой механизм улучшает производительность в многопользовательских системах с высокой конкурентностью (например, банковские системы, OLTP). Практически всегда нужно учитывать влияние частых UPDATE на потребление пространства и запускать **autovacuum** для поддержания работоспособности БД. Понимание работы UPDATE важно для оптимизации и правильного проектирования БД в PostgreSQL 14+ и новее.
