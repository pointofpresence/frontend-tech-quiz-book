# Сборщик мусора и счётчик ссылок?

### Категории

Async, Memory management, Backend, Technical, Python internals

### Краткий ответ

**Сборщик мусора и счётчик ссылок**

- Сборщик мусора (GC) — автоматическое управление памятью  
- **Счётчик ссылок** — освобождение объекта при 0 ссылок  
- GC обнаруживает **неиспользуемые объекты** через обход графа  
- Счётчик ссылок не ловит циклические ссылки, GC — ловит  
- GC может приводить к паузам, счётчик ссылок — чаще без задержек  
- GC используется в JVM, .NET, JavaScript, счётчик — в Objective-C, Python  
- Практическая ценность: стабильное управление памятью без утечек и ошибок доступа

### Подробный ответ
#### Основной ответ  
**Сборщик мусора (Garbage Collector, GC)** и **счётчик ссылок (Reference Counting)** — это две разные технологии управления памятью, используемые для автоматического освобождения неиспользуемых объектов в программах. Сборщик мусора работает, периодически находя и удаляя объекты, до которых невозможно добраться из корневых ссылок (roots), тогда как счётчик ссылок ведёт подсчёт количества активных ссылок на объект и освобождает память, когда этот счётчик становится нулём.

#### Ключевые моменты  
- **Сборщик мусора** обычно базируется на алгоритмах трассировки, таких как Mark-and-Sweep, Mark-and-Compact или поколенческий GC. Он отключает объекты, которые не достижимы, что позволяет собирать объекты в циклах ссылок. Такой подход используется в Java (с версии 8 — G1, в Java 17 — ZGC), .NET, Go.  
- **Счётчик ссылок** поддерживает счётчик в каждом объекте, увеличивая его при создании новых ссылок и уменьшая при их удалении. Когда счётчик падает до нуля, объект немедленно освобождается. Преимущество — низкая задержка освобождения (не нужно ждать пауз GC), но есть большой недостаток — невозможность освобождения циклических ссылок, что требует дополнительных механизмов (например, периодического поиска циклов, как в CPython).  
- **Trade-off**: сборщик мусора обычно лучше справляется с сложными графами объектов и циклическими ссылками, но может вызывать паузы и повышенную нагрузку на CPU во время сборки. Счётчик ссылок проще реализовать и предсказуем по времени освобождения, но усложняет архитектуру из-за проблем с циклами и накладных расходов на обновление счётчиков.

#### Практический контекст  
В современных языках, например, в Swift и Objective-C, живёт активное использование счётчиков ссылок (ARC — automatic reference counting), при этом иногда дополняется средствами устранения циклов (weak, unowned ссылки). В то же время Java и C# преимущественно полагаются на поколения сборщиков мусора для производительности и управления большим объемом объектов с минимальными задержками. Понимание этих механизмов важно для оптимизации использования памяти и устранения утечек в реальных проектах.
