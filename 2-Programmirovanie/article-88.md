# Какова сложность доступа к элементу по индексу в List<T> и почему?

Категория: Программирование

### Краткий ответ

**Сложность доступа к элементу по индексу в List<T> и почему**

- List<T> в .NET реализован как **динамический массив**  
- Элементы хранятся в **непрерывном блоке памяти**  
- Доступ по индексу — это операция **прямого адреса** (индексация)  
- Сложность доступа — **O(1)** (константное время)  
- За счёт адресации элемента по формуле *(базовый адрес + индекс × размер элемента)*  
- При вставке/удалении сложность другая, но для доступа важна именно константность  
- Применение: быстрый прямой доступ, удобен для индексируемых коллекций  

Итого: доступ по индексу в List<T> — **O(1)** из-за внутреннего представления в виде массива с непрерывной памятью.

### Подробный ответ
#### Основной ответ
В **List<T>** (например, в C# или Java) доступ к элементу по индексу происходит за **O(1)** — константное время. Это связано с тем, что внутренне List реализован как массив, то есть элементы располагаются в памяти последовательно, и можно сразу вычислить адрес нужного элемента, зная индекс.

#### Ключевые моменты
- **Прямая адресация**: благодаря тому, что элементы хранятся в contiguous memory block (непрерывном блоке памяти), для доступа к элементу по индексу достаточно взять адрес начала массива и прибавить индекс, умноженный на размер элемента.
- **Отличие от связных структур**: в отличие от связного списка (LinkedList), где для доступа к элементу нужно последовательно пройтись по узлам (O(n)), в List это немедленный доступ.
- **Реализация расширяемого массива**: List<T> обычно расширяет внутренний массив динамически, но это не влияет на сложность доступа — оно всегда остается O(1), даже при ресайзе.

#### Практический контекст
Такое поведение обеспечивает высокую производительность при частом случайном доступе к элементам коллекции, например, в алгоритмах сортировки или выборки данных. В проектах с интенсивной работой с индексами предпочтительно использовать List<T>, а не связные структуры.
