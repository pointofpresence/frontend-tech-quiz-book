# В чём отличие Kafka от RabbitMQ?

**Отличие Kafka от RabbitMQ**

- **Архитектура**: Kafka — распределённый журнал сообщений (log), RabbitMQ — брокер очередей
- **Модель доставки**: Kafka — потоковая передача, хранит сообщения долго; RabbitMQ — push-модель, удаляет после подтверждения
- **Производительность**: Kafka — высокая пропускная способность, оптимизирован под большие потоки; RabbitMQ — меньше throughput, но гибче роутинг
- **Гарантии доставки**: Kafka — "в точности один раз" с помощью партиций и оффсетов; RabbitMQ — "по крайней мере один раз", поддержка транзакций и подтверждений
- **Роутинг**: RabbitMQ поддерживает сложный роутинг через exchange (direct, topic, fanout), Kafka — только топики и партиции
- **Хранение**: Kafka хранит логи длительное время для повторного чтения; RabbitMQ — хранит кратковременно, ориентирован на немедленную доставку
- **Использование**: Kafka подходит для потоковой обработки, аналитики и event sourcing; RabbitMQ лучше для традиционных задач очередей и универсальной интеграции
- В итоге: Kafka — система для масштабируемой потоковой передачи с долгим хранением сообщений, RabbitMQ — гибкий брокер для управления очередями и маршрутизацией сообщений.

# Подробный ответ
## Основной ответ  
**Kafka** и **RabbitMQ** — это два популярных брокера сообщений, но они существенно различаются по архитектуре и применению. Kafka — это распределённая платформа стриминга данных с ориентацией на высокую пропускную способность и долговременное хранение сообщений. RabbitMQ — классический message broker с поддержкой различных моделей маршрутизации и гибкими очередями, ориентированный на гарантированную доставку и надежную обработку сообщений в режиме очередей.

## Ключевые моменты  
- **Архитектура и модель данных**: Kafka хранит сообщения в топиках как неизменяемые журналы с возможностью повторного чтения, что идеально для event sourcing, ETL и больших потоков данных. RabbitMQ реализует классические очереди сообщений с удалением после подтверждения доставки.  
- **Пропускная способность и масштабирование**: Kafka масштабируется горизонтально, поддерживает десятки тысяч сообщений в секунду с низкой задержкой (~ms), оптимизирован для потоковой передачи в режиме pub/sub. RabbitMQ хорош в разнонаправленных workflow с гибкой маршрутизацией, но менее эффективен при экстремальных нагрузках и требует более сложной настройки для масштабирования.  
- **Гарантии доставки и упорядочение**: RabbitMQ обеспечивает надежную доставку с поддержкой подтверждений (ack) и различными политиками QOS, идеально для transactional систем. Kafka гарантирует упорядочение сообщений внутри партиций и обеспечивает at-least-once delivery, но с более сложным управлением потребителями и offset'ами.

## Практический контекст  
В проектах Kafka часто используют для масштабируемой потоковой аналитики, обработки логов и event-driven архитектуры с большим объемом данных в реальном времени (например, в бизнесе с высокочастотной телеметрией). RabbitMQ предпочтительнее для классических enterprise-интеграций, микросервисов с сложной маршрутизацией сообщений и когда важна гарантия доставки и порядок обработки каждого сообщения.
