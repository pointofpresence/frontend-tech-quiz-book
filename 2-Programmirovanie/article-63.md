# Как оптимизировать запрос для поиска самого длинного видео из топ-10 коротких?

**Оптимизация запроса: самый длинный из топ-10 коротких видео**

- Контекст: SQL-запросы для поиска и сортировки видео по длине  
- Основная задача: найти самое длинное видео среди коротких, предварительно ограничившись топ-10 по длине  
- Ключевой момент: разбить задачу на два шага — сначала выбрать топ-10 **коротких** видео, затем из них — самый длинный  
- Использовать **подзапрос** или **CTE (WITH)** для выделения топ-10 коротких видео с сортировкой по длине (по возрастанию или по заданному критерию "коротких")  
- Внешний запрос выбрать видео с максимальной длиной из результата подзапроса  
- Для ускорения — создать и использовать **индекс по длине видео** (например, на поле duration или length)  
- Важно минимизировать количество строк в подзапросе, чтобы не сортировать всю таблицу полностью  
- Практика: в PostgreSQL можно написать  
  ```sql
  WITH TopShortVideos AS (
    SELECT * FROM videos
    WHERE duration < <короткий_порог>
    ORDER BY duration ASC
    LIMIT 10
  )
  SELECT * FROM TopShortVideos
  ORDER BY duration DESC
  LIMIT 1;
  ```  
- Такой подход позволяет эффективно сначала выборку коротких видео ограничить топ-10 самой короткой длины, а затем быстро найти среди них самое длинное видео  
- Альтернатива: использовать оконные функции (ROW_NUMBER/PARTITION) если нужны дополнительные фильтры или условия  
- Итог: разделение задачи на этапы + использование индексов + минимальный объём данных для верхнего запроса решают задачу оптимально  

Это позволит получить ответ быстро и с минимальной нагрузкой на базу.

# Подробный ответ
## Основной ответ
Чтобы оптимизировать запрос для поиска самого длинного видео из топ-10 коротких, сначала нужно корректно определить критерии "коротких" и понять способ их отбора (например, по рейтингу, дате, просмотрам). Предполагается, что существует некая метрика ранжирования видео (например, популярность), и нам нужно из топ-10 таких видео с короткой длительностью найти максимальную по длине. Оптимизация сводится к минимизации объёма обрабатываемых данных и использованию индексов.

## Ключевые моменты
- **Выборка ограниченного поднабора**: сначала выбрать топ-10 коротких видео (например, `ORDER BY rating DESC LIMIT 10` с условием `duration < X`), чтобы ограничить объем данных для последующего поиска.
- **Индексация полей**: иметь индекс по полю длительности (`duration`) и по полю для сортировки топ-10 (например, `rating`) — это ускорит фильтрацию и сортировку.
- **Использование подзапроса или CTE**: нельзя сразу искать максимальную длительность во всей таблице — сначала удобно получить поднабор топ-10 коротких видео, затем найти среди них максимальное по длине.
- **Минимизация количества обрабатываемых строк**: без ограничения топ-10 запрос провалится в полный скан таблицы.
  
Пример SQL (PostgreSQL 14+):

```sql
WITH top_short_videos AS (
  SELECT * FROM videos
  WHERE duration < 300 -- критерий "коротких" в секундах
  ORDER BY rating DESC
  LIMIT 10
)
SELECT * FROM top_short_videos
ORDER BY duration DESC
LIMIT 1;
```

## Практический контекст
В реальных проектах с большими каталогами видео важно иметь правильные индексы (например, B-tree по `duration`, `rating`). Иногда кэшируют топ-10 коротких видео в Redis для мгновенного поиска самого длинного, что снижает нагрузку на БД. Также можно комбинировать с материализованными представлениями, если данные редко меняются, чтобы повысить производительность и снизить latency (~50ms для подобных запросов в продакшене считается хорошим результатом).
