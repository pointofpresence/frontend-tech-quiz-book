# Как обеспечивается гарантия доставки сообщений в Kafka?

### Категории

Processes, Behavioral, QA, Microservices, Testing, Async, Architecture, Message queue, Onboarding, Reliability, Messaging, Guarantee, Distributed systems, Guarantee delivery, Message delivery

### Краткий ответ

**Как обеспечивается гарантия доставки сообщений в Kafka?**

- Kafka гарантирует доставку через **репликацию** данных в кластере
- Использует концепцию **ISR (in-sync replicas)** — только синхронизированные реплики подтверждают запись
- Сообщения считаются записанными после получения **acknowledgment** от нужного числа реплик (настройка ack: 0, 1, all)
- При использовании ack=all достигается гарантия **"трижды надежной" записи** (синхронизация между всеми ISR)
- Для консумеров есть настройка **offset commit** — чтобы сохранить прочитанные позиции и избежать дублирования или потери
- Настройка **min.insync.replicas** определяет минимальное число реплик для подтверждения записи, повышая стойкость
- Архитектура поддерживает **ат-леаст-ван** (at-least-once) доставку, комбинируя репликацию и подтверждения, что минимизирует риск потери сообщений

Итог: Сила гарантии достигается через репликацию, подтверждение записи всех синхронизированных реплик и контроль смещений у потребителей, что обеспечивает надежность доставки в масштабах распределённой системы.

### Подробный ответ
#### Основной ответ
Гарантия доставки сообщений в **Apache Kafka** обеспечивается сочетанием нескольких механизмов: подтверждения продюсера (acks), идемпотентной записи, синхронизации с репликами и обработкой сдвигов (offsets) на стороне консьюмера. Kafka обеспечивает три уровня гарантий доставки: **at most once** (не более одного раза), **at least once** (минимум один раз) и **exactly once** (ровно один раз) — с помощью комбинации этих механизмов.

#### Ключевые моменты
- **Продюсерские подтверждения (acks)**: параметр `acks` может принимать значения 0 (без подтверждений), 1 (запись подтверждена лидером партиции) или `all` (ждущие подтверждения всех реплик). Значение `all` обеспечивает устойчивость к сбоям узлов и защищает от потери данных.
- **Идемпотентность продюсера (idempotent producer)**: начиная с Kafka 0.11, продюсер может назначать sequence number для каждого сообщения, предотвращая дублирование при повторных попытках отправки. Это критично для Exactly-once semantics.
- **Репликация и ISR (in-sync replicas)**: сообщения записываются на лидера и реплицируются на followers, входящие в ISR. Если подтверждения ждут от всех in-sync реплик (`acks=all`), данные не теряются при сбое.
- **Обработка сдвигов (offsets) у консьюмеров**: для управления доставкой используется явное коммитирование offset’ов, что позволяет обеспечивать повторную обработку при сбоях и выбирать стратегию at-least-once или exactly-once.
- **Транзакции**: с Kafka 0.11+ можно использовать транзакции, объединяющие запись сообщений и коммит offset'ов в одну атомарную операцию. Это ключ к exactly-once processing в сложных конвейерах.

#### Практический контекст
В реальных проектах для надежности используют `acks=all`, включают идемпотентность продюсера, а на стороне консьюмера — управляют offset коммитами вручную либо с помощью транзакций и Kafka consumer groups. Для критичных данных обычно настраивается кластер с минимальной задержкой репликации и мониторингом состояния ISR через Prometheus и Kafka Manager.
