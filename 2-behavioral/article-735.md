# Как реализована персистентность в Redis?

### Категории

Behavioral, Persistence, Async, Backend, Database, Messaging

### Краткий ответ

**Как реализована персистентность в Redis?**

- персистентность — гарантия **сохранения данных** на диск  
- два основных механизма:   
  - **RDB (Snapshotting)**: снимает **моментальные снимки** памяти через интервалы, создает компактные бинарные файлы, эффективен по ресурсам, но возможна потеря последних изменений  
  - **AOF (Append-Only File)**: логирует **каждую операцию записи** в файл, более надёжно, позволяет восстановить точное состояние, но файл растёт и требует периодической перезаписи (фрагментации)  
- можно использовать **совместно** RDB и AOF для баланса скорости и безопасности  
- при запуске Redis восстанавливает данные с диска, применяя последний сохранённый RDB-файл и/или AOF  
- поддержка настроек частоты сохранения, fsync для AOF для контроля баланса производительности и риска потери данных  
- персистентность отключаемая, при выключении Redis работает как in-memory DB  
- используется для обеспечения устойчивости в продакшене, бэкапов, failover и восстановления после сбоев

### Подробный ответ
#### Основной ответ
В **Redis** персистентность реализована через два основных механизма: **RDB (Redis Database Backup)** — создание снимков базы в определённые моменты времени, и **AOF (Append-Only File)** — лог всех команд, изменяющих состояние, записываемый последовательно. Это позволяет сохранять данные на диск и восстанавливаться после перезапуска сервера.

#### Ключевые моменты
- **RDB** создаёт точки сохранения за счёт периодических снапов. Это эффективно по производительности и подходит, если допустима потеря последних изменений (например, до минуты). Этот метод не блокирует сервер, кроме короткого момента записи.
- **AOF** логирует каждую команду модификации данных в отдельный файл, что обеспечивает более точное восстановление данных (почти без потерь). При старте Redis воспроизводит логи для восстановления состояния.
- Можно комбинировать оба метода, настроив Redis на работу в режиме RDB + AOF, чтобы получить баланс между быстрым восстановлением (RDB) и минимальными потерями данных (AOF).
- В Redis 6+ добавлены улучшения, такие как **AOF rewrite** — автоматическая очистка и сжатие файла AOF, чтобы файл не становился слишком большим и не ухудшалась производительность.
- Кроме этого, Redis поддерживает **псевдо-персистентность** через **RDB+snapshotting** для минимизации влияния на latency, а также имеет настройки для асинхронной записи на диск (fsync при записи файла AOF или RDB).

#### Практический контекст
В реальных проектах часто используют комбинацию: RDB для регулярных контрольных точек, а AOF — для минимизации потерь данных при сбоях. В высоконагруженных системах применяют Redis в режиме с AOF с частым fsync, чтобы обеспечить баланс между пропускной способностью и надёжностью. Например, в Redis 6+ для оптимизации latency и отказоустойчивости в кластерах.
