# Как сохранить состояние при смене конфигурации?

### Категории

Configuration change, Behavioral, Payment processing, Mobile, Security, Android lifecycle, State management

### Краткий ответ

**Как сохранить состояние при смене конфигурации?**  
- область: мобильная разработка (Android)  
- конфигурация — смена ориентации, языка, ресурсов  
- Android пересоздаёт Activity → теряется UI-состояние  
- использовать onSaveInstanceState(Bundle) для сохранения данных  
- восстанавливать данные в onCreate(Bundle) или onRestoreInstanceState(Bundle)  
- ViewModel хранит состояние через конфигурационные изменения без пересоздания  
- сохранение через retained Fragment или сохранение в базы/файлы при критичных данных  
- для UI: непроизводительно сохранять всё, лучше минимальный набор ключевых данных  
- практическое значение — обеспечить непрерывность UX при повороте и изменениях  
---

**Подробности:**  
При изменении конфигурации Android уничтожает и создаёт заново Activity, что приводит к потере состояния. onSaveInstanceState вызывается перед уничтожением, куда можно записать ключевые данные (например, текст из EditText, позицию списка). Затем эти данные восстанавливаются в onCreate или onRestoreInstanceState. Однако для более сложного состояния лучше использовать ViewModel из Android Architecture Components, который переживает смену конфигурации и хранит логику и данные без необходимости пересоздавать. Также можно использовать retained Fragment с setRetainInstance(true), но этот подход устаревший. Для долговременных данных используют БД, SharedPreferences или файлы. Важно сохранять только критичные данные, не перегружая Bundle, чтобы не снизить производительность. Таким образом, сохраняется пользовательский опыт бесперебойной работы после поворота экрана или смены конфигурации.

### Подробный ответ
#### Основной ответ
Сохранение состояния при смене конфигурации — это классическая задача, часто встречающаяся в мобильной разработке (например, Android) или в приложениях с динамическими настройками. Основная идея — гарантировать, что пользовательские данные или состояние UI не теряются при изменении конфигурации, например, поворот экрана, смена темы, локали или параметров ресурса.

#### Ключевые моменты
- **Использование жизненного цикла**: в Android, например, состояние сохраняется через методы `onSaveInstanceState()` и восстанавливается в `onRestoreInstanceState()` или `onCreate()`. Это позволяет сериализовать необходимые данные в `Bundle` и восстановить после пересоздания Activity.
- **ViewModel и сохранение состояния**: с появлением Android Jetpack `ViewModel` помогает сохранять состояние между пересозданиями UI-компонентов, так как ViewModel переживает смену конфигурации. Для долгосрочного хранения данных можно использовать `SavedStateHandle`.
- **Изоляция настроек и использование реактивных подходов**: при изменении конфигурации (например, смене темы) стоит отделять состояние UI от конфигурации, сохраняя данные в отдельных слоях — модели или хранилище. Выбор реактивных шаблонов (LiveData, StateFlow) значительно упрощает обновление UI без потери состояния.
- **Сериализация и хранение на стороне клиента**: иногда состояние нужно сохранять в SharedPreferences, в базе данных или через файловую систему, если объем данных большой или они должны пережить перезапуск приложения.
  
#### Практический контекст
В React Native и вебе можно использовать локальное состояние с локальной памятью (localStorage, IndexedDB) или управлять состоянием через Redux/Persist, чтобы сохранить его при смене пользовательской сессии или конфигурации. В Android же типично применяют `ViewModel` и `SavedStateHandle` (AndroidX 2.2+) вместе с `onSaveInstanceState` для комплексного сохранения небольших UI-данных. Такой подход обеспечивает не только сохранность, но и удобство тестирования и поддержки кода.
