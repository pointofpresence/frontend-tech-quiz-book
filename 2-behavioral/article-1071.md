# Почему GET-запрос считается менее безопасным для передачи данных?

### Категории

HTTP, Security, Behavioral, API, Testing

### Краткий ответ

**Почему GET-запрос считается менее безопасным для передачи данных?**

- HTTP метод для **получения** данных, не для передачи секретов
- Параметры передаются через URL → **видны в адресной строке**
- URL логируется браузером, прокси и сервером → риски утечки данных
- Ограничение длины URL → нельзя скрыть крупные чувствительные данные
- Отсутствует шифрование на уровне HTTP (требуется HTTPS для защиты)
- GET-запросы легко кэшируются и сохраняются в истории
- Рекомендуется использовать POST с телом запроса для конфиденциальных данных

---

**Итог:** GET небезопасен для передачи секретов из-за открытости URL и логирования; HTTPS + POST лучше для защиты данных.

### Подробный ответ
#### Основной ответ
**GET-запрос** считается менее безопасным для передачи данных, потому что параметры запроса встроены непосредственно в URL — они видимы в адресной строке браузера, логах серверов и промежуточных прокси. Это значительно повышает риск раскрытия чувствительной информации, например, паролей или токенов. Также URL имеет ограничения по длине, что может привести к урезанию данных.

#### Ключевые моменты
- **Видимость данных**: параметры выставлены в URL и могут быть сохранены в истории браузера, кешах и логах веб-серверов, что увеличивает шансы утечки информации.
- **Кэширование**: GET-запросы часто кешируются браузерами и CDN, и если передавать конфиденциальные параметры, они рискуют попасть в чужие руки.
- **Идемпотентность и семантика**: GET ориентирован на получение данных, без изменений на сервере, поэтому передавать через него чувствительные данные — плохая практика с точки зрения HTTP-стандартов.

#### Практический контекст
Для безопасности чаще используют **POST-запросы** с телом, которое не отображается в URL, или применяют шифрование (HTTPS), чтобы уменьшить риск перехвата. В React 18+ и современных SPA, например, для авторизации токены обычно хранятся в httpOnly cookies или в заголовках, а не в URL.
