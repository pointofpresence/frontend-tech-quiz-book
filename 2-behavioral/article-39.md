# Какие паттерны проектирования знаете? Где применяли?

### Категории

Backend, Behavioral, Design patterns, Software architecture

### Краткий ответ

**Какие паттерны проектирования знаете? Где применяли?**

- **Порождающие:** Singleton, Factory, Builder — для управления созданием объектов, обеспечения одного экземпляра, упрощения конфигурации.
- **Структурные:** Adapter, Decorator, Composite — для адаптации интерфейсов, динамического расширения функционала и организации иерархий.
- **Поведенческие:** Observer, Strategy, Command — для разделения обязанностей, изменения поведения во время выполнения, реализации отложенного выполнения.
- Применял в многомодульных системах для улучшения поддержки, масштабируемости, снижения связанных зависимостей.
- В веб-разработке использовал Observer для событий, Factory для инстанциирования сервисов, Decorator для расширения UI-компонентов.
- В backend-проектах — Singleton для логгера, Strategy для выбора алгоритмов, Command для очередей задач.
- Всегда учитываю бизнес-задачи и архитектуру при выборе паттернов, избегая излишней сложности.

### Подробный ответ
#### Основной ответ
**Паттерны проектирования** — это проверенные временем шаблоны решений часто возникающих задач в архитектуре и разработке ПО. Они помогают писать более читаемый, поддерживаемый и расширяемый код. Основные группы паттернов — это **порождающие**, **структурные** и **поведенческие**, и каждый из них решает свою категорию задач.

Лично я активно использовал следующие паттерны:

- **Singleton** — для организации единственного глобального экземпляра, например, менеджера конфигурации в приложении на Java Spring.
- **Factory Method** и **Abstract Factory** — для динамического создания объектов с разными параметрами, применял в системах, где нужно абстрагировать логику создания, например, при реализации генерации разных видов отчетов.
- **Decorator** — для гибкого расширения функциональности объектов без изменения их кода, часто применял в middleware и валидации данных в микросервисах.
- **Observer** — для построения событийных систем и реактивных потоков данных, использовал с помощью RxJava и в асинхронных UI-компонентах в React.
- **Strategy** — для замены алгоритмов на лету, например, разные способы расчёта стоимости доставки в e-commerce приложении.
- **Command** — для реализации undo/redo функционала и очередей задач, применял в десктопных приложениях.

#### Ключевые моменты
- Понимание паттернов помогает создавать **модули с четкими границами ответственности** и улучшает коммуникацию в команде (каждый говорит на одном языке).
- Правильное применение паттерна — это компромисс между гибкостью и сложностью. Например, **Decorator** расширяет функциональность без наследования, но увеличивает количество объектов.
- В современных фреймворках и платформах (React 18, Spring Boot 3) многие паттерны интегрированы или адаптированы под новые парадигмы, например, Dependency Injection — разновидность фабрики.

#### Практический контекст
В одном из последних проектов на **Spring Boot 3** мы внедряли паттерн **Factory** для создания разных адаптеров к внешним API, что позволило просто добавлять новые интеграции без изменений в бизнес-логике. Также активно применял **Observer/Publish-Subscribe** с Kafka для организации событийной архитектуры с гарантией доставки и масштабируемостью.

Такое системное использование паттернов повышает качество кода и ускоряет разработку благодаря повторному использованию и проверенной архитектуре.
