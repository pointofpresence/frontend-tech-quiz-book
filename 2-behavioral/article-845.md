# Как частичные/покрывающие индексы (partial indexes) оптимизируют производительность?

### Категории

Behavioral, Performance, Indexes, Database, Optimization

### Краткий ответ

**Как частичные/покрывающие индексы оптимизируют производительность?**

- частичный индекс — индекс на **часть строк** по условию
- уменьшает **размер** индекса и **объем** сканирования
- ускоряет запросы с фильтром, соответствующим условию индекса
- покрывающий индекс содержит все **нужные поля** запроса
- позволяет выполнить запрос без обратного доступа к таблице (**Index Only Scan**)
- снижает **затраты ввода-вывода** и CPU
- улучшает скорость чтения за счет меньшего **количества данных для обработки**

### Подробный ответ
#### Основной ответ
**Частичные (покрывающие) индексы** — это индексы, которые создаются не на всю таблицу, а только на подмножество строк, удовлетворяющих определённому условию (например, WHERE status = 'active'). Они оптимизируют производительность запросов, снижая размер индекса и количество данных, которые нужно обрабатывать, что ускоряет поиск и обновления.

#### Ключевые моменты
- **Уменьшение размера индекса**: поскольку индекс покрывает только часть данных, он занимает меньше места в памяти и на диске, что снижает I/O и улучшает кеширование.
- **Быстрые запросы с фильтрацией**: запросы, использующие условие выбора, могут пользоваться этим узконаправленным индексом, избегая полного сканирования таблицы или затратного сканирования полноразмерных индексов.
- **Оптимизация записи**: при обновлении или вставке данных индекс пересчитывается только для релевантных строк, что снижает нагрузку на систему по сравнению с полноразмерным индексом, особенно если часть таблицы часто меняется, а другая — нет.

#### Практический контекст
В PostgreSQL 14+, например, частичные индексы часто применяют для больших логов или таблиц со статусами — когда запросы касаются только активных записей. Это позволяет добиться порядка **30-50% сокращения времени выполнения** и значительно уменьшить размер индекса по сравнению с полноразмерным, что критично для OLTP систем с высокими требованиями к latency.
