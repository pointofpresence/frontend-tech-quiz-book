# Что такое ThreadLocal? Практический пример использования.

### Категории

Multithreading, Behavioral, Java, Concurrency, ThreadLocal, Backend

### Краткий ответ

**Что такое ThreadLocal? Практический пример использования**

- **ThreadLocal** — класс в Java для хранения данных, привязанных к текущему потоку
- Позволяет каждому потоку иметь свою **локальную копию** переменной
- Значения потоков **изолированы**, нет совместного доступа
- Упрощает работу с **небезопасными для потоков объектами** без синхронизации
- Пример: хранение **пользовательских сессий** или **конфигурации** на уровне потока в веб-приложениях  
- Часто используется в реализациях **connection pool**, **логгирования**, **transaction context**
- Применение улучшает **параллелизм** без блокировок

---

**Пример кода:**

```java
public class UserContext {
    private static final ThreadLocal<String> userThreadLocal = ThreadLocal.withInitial(() -> null);

    public static void setCurrentUser(String user) {
        userThreadLocal.set(user);
    }

    public static String getCurrentUser() {
        return userThreadLocal.get();
    }

    public static void clear() {
        userThreadLocal.remove();
    }
}
```

В многопоточной среде каждый поток хранит свою «текущую сессию» пользователя, изолированную от других. Это значительно упрощает передачу данных контекста без явного параметрирования.

### Подробный ответ
#### Основной ответ
**ThreadLocal** — это специальный класс в Java, который позволяет связать данные с конкретным потоком выполнения, обеспечивая каждому потоку свой отдельный экземпляр переменной. Это значит, что разные потоки смогут работать с одной и той же переменной без риска конфликтов и необходимости синхронизации, так как каждый поток видит свою копию.

#### Ключевые моменты
- **Изоляция данных:** ThreadLocal хранит данные отдельно для каждого потока, предотвращая гонки данных без использования explicit locks.
- **Используется для контекстов:** Часто используют для хранения контекста запроса (например, пользователя, перехваченного в сервисном слое) или database session, чтобы передавать данные по вызовам без загрязнения сигнатуры методов.
- **Внимание к утечкам памяти:** Объекты в ThreadLocal должны очищаться (вызывать `remove()`), иначе есть риск утечки памяти, особенно в пулах потоков.

#### Практический контекст
Например, в веб-приложении на Spring можно хранить объект `SimpleDateFormat` или сессию пользователя в ThreadLocal, чтобы избежать многопоточного использования одного и того же экземпляра и необходимости синхронизировать доступ. В React 18 и выше, для серверных приложений, похожий паттерн помогает сохранить перезапускаемые контексты на уровне потока.

```java
private static final ThreadLocal<SimpleDateFormat> threadLocalDateFormat =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("dd-MM-yyyy"));

public String formatDate(Date date) {
    return threadLocalDateFormat.get().format(date);
}
```

Здесь каждый поток будет иметь свою копию `SimpleDateFormat`, что предотвращает `ConcurrentModificationException`, которая возникает при совместном использовании одного экземпляра без синхронизации.
