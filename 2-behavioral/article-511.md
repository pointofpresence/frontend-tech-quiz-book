# Как Android обрабатывает смену конфигурации?

### Категории

Android, Mobile, Behavioral, Lifecycle, Configuration, State management

### Краткий ответ

**Как Android обрабатывает смену конфигурации?**

- система перезапускает Activity при смене конфигурации (экран, ориентация, язык)
- происходит вызов onSaveInstanceState() для сохранения состояния
- затем уничтожение Activity (onPause(), onStop(), onDestroy())
- создание нового экземпляра Activity с новой конфигурацией
- восстановление состояния в onCreate() или onRestoreInstanceState()
- можно переопределить поведение через android:configChanges в манифесте
- обработка конфигурации важна для плавного UX и предотвращения данных потерь

### Подробный ответ
#### Основной ответ
В Android смена конфигурации (например, поворот экрана, смена языка или изменение размера экрана) приводит к уничтожению текущей активности и её пересозданию с новой конфигурацией. Это делается, чтобы система могла полноценно адаптировать UI и ресурсы под новые параметры устройства. При смене конфигурации вызываются жизненные циклы методов `onPause()`, `onStop()`, `onDestroy()`, а затем `onCreate()` с обновлёнными ресурсами. Если разработчик не хочет полного пересоздания, можно обработать изменения самостоятельно через атрибут `android:configChanges` в манифесте, но это требует вручную управлять изменениями.

#### Ключевые моменты
- **Полное пересоздание Activity** — стандартное поведение для гарантированной загрузки подходящих ресурсов (layout, strings, drawable и т.д.). В React Native и Jetpack Compose подходы схожи, но реализуются в рамках платформы.
- **Сохранение состояния:** Для восстановления состояния UI используют `onSaveInstanceState()` + `Bundle`, также часто применяется ViewModel из архитектуры Jetpack, которая переживает конфигурационные изменения без уничтожения.
- **Оптимизации:** В Android 13+ появились новые API для оптимизированной обработки конфигураций, например `OnConfigurationChangedCallback`, позволяющие более гибко реагировать, не пересоздавая активность.

#### Практический контекст
На практике при разработке приложений под Android важно корректно сохранять и восстанавливать данные при смене конфигурации, чтобы не терять состояние пользователя. Часто это реализуется с помощью `ViewModel` и `SavedStateHandle` из Jetpack, а кастомная обработка через `configChanges` применяется только для узкоспециализированных случаев, чтобы избежать проблем с багами и техническим долгом.
