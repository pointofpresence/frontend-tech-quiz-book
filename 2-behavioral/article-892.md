# Как спроектировать систему учёта просмотров видео?

### Категории

### Краткий ответ

**Как спроектировать систему учёта просмотров видео?**

- система для подсчёта просмотров видео онлайн  
- ключевые требования: высокая нагрузка, точность подсчёта, масштабируемость  
- сбор событий просмотров через API или SDK клиентов  
- использование **асинхронной** обработки (очереди, брокеры сообщений) для нагрузочного баланса  
- агрегация просмотров в **хранилище с быстрым доступом** (например, Redis, Cassandra)  
- периодическая долговременная запись в **распределённую БД** (SQL/NoSQL) для аналитики  
- защита от фрода: фильтрация дублирующих просмотров, лимиты по IP/пользователю  
- масштабируемость через **шардирование** и **кэширование**  
- аналитика и вывод статистики в реальном времени и по историческим данным

### Подробный ответ
#### Основной ответ
Проектирование системы учёта просмотров видео — это задача сбора, хранения и аналитики большого объёма событий с высокой скоростью. Основная цель — точно и эффективно считать просмотры, избегая накруток и обеспечивая масштабируемость при росте пользователей. Важно продумать архитектуру для реального времени и возможность агрегаций для аналитики.

#### Ключевые моменты
- **Сбор данных**: просмотры — это события, которые можно собирать через систему событий (event tracking) с помощью очередей (Kafka, AWS Kinesis). Важно учитывать уникальность просмотра (например, уникальный пользователь, IP) и минимальную длительность просмотра, чтобы фильтровать спам и накрутки.
- **Хранение и агрегация**: для масштабируемого подсчёта часто используют распределённые NoSQL базы типа **Cassandra** или **ClickHouse** для аналитики, а для быстрого доступа — кеширование в **Redis**. Можно хранить сырые события и аггрегировать их в батчах (например, каждые 5-10 минут) для уменьшения нагрузки.
- **Обеспечение точности и производительности**: применяют алгоритмы типа HyperLogLog или Counting Bloom filters, чтобы оценивать уникальных зрителей без высокой затратности по ресурсам. Также важна idempotency — просмотр не должен считаться дважды при повторном получении события.
- **Обработка накруток и мошенничества**: интеграция с антифрод системами, анализ поведения (частота, длительность) и блеклист IP/устройств.
- **Интерфейс и API**: быстрый доступ к счетчикам через REST или GraphQL, с возможностью обновления и получения статистики в реальном времени.

#### Практический контекст
В современных видеосервисах (Netflix, YouTube) используется event-driven архитектура с Kafka для сбора событий просмотра, промежуточные слои агрегации и хранение в ClickHouse для анализа и отчётности. Redis применяют для realtime счётчиков (например, текущих просмотров в данный момент). Постоянно ведут мониторинг data pipeline через Prometheus и Grafana, чтобы гарантировать 99.9% uptime и точность подсчёта.
