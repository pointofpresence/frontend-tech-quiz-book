# В чём отличие сигнатур методов расширения LINQ (Where, OrderBy и т. д.) для IEnumerable<T> и IQueryable<T>?

### Категории

Math, Behavioral, API, Performance, Backend, Query, Design patterns, Collections

### Краткий ответ

**Отличие сигнатур методов расширения LINQ для IEnumerable<T> и IQueryable<T>**  
- **IEnumerable<T>** — методы принимают **Func<T, bool> (делегаты)**, работают с коллекцией в памяти (отложенное выполнение, но на клиенте)  
- **IQueryable<T>** — методы принимают **Expression<Func<T, bool>> (выражения деревьев)**, переводятся в запрос к источнику данных (например, SQL)  
- **Ключевое отличие:** IQueryable строит **выражение запроса**, которое обрабатывается провайдером (например, EF), IEnumerable — **выполняет работу локально**  
- IEnumerable работает с реальными объектами, IQueryable — с **отложенными выражениями**  
- Выбор зависит от источника данных и необходимости оптимизации запросов (IQueryable для БД, IEnumerable — для коллекций памяти)  
- Пример: `Where(Func<T, bool>)` — фильтрация после загрузки, `Where(Expression<Func<T, bool>>)` — фильтрация в запросе  
- Это различие критично для производительности и переноса логики обработки данных на сервер  

Всегда учитывать, что IQueryable эффективнее для больших и удалённых данных, IEnumerable — удобнее для локальной обработки.

### Подробный ответ
#### Основной ответ
Методы расширения LINQ для **IEnumerable<T>** и **IQueryable<T>** имеют схожие имена и назначения (например, `Where`, `OrderBy`), но существенно различаются по реализации и поведению. Эти методы для **IEnumerable<T>** работают с коллекциями в памяти и выполняют операции немедленно (с помощью делегатов типа `Func<T, bool>`), тогда как методы для **IQueryable<T>** строят **выражения** (expression trees), которые могут быть оптимизированы и транслированы в запросы, например, SQL для удалённого источника данных.

#### Ключевые моменты
- Для **IEnumerable<T>** сигнатуры методов принимают **делегаты** `Func<T, bool>`, `Func<T, TResult>`, что означает выполнение операций непосредственно в памяти, через перебор коллекции.
- Для **IQueryable<T>** методы принимают **деревья выражений** `Expression<Func<T, bool>>`, позволяющие провайдеру запросов (например, Entity Framework) анализировать, преобразовывать и транслировать запрос в целевой язык (SQL, NoSQL и др.).
- Это даёт возможность **отложенного выполнения запроса** и значительного улучшения производительности при работе с удалёнными или большими данными — фильтрация и сортировка происходят на сервере данных, а не в приложении.
- Кроме того, `IQueryable<T>` методы возвращают `IQueryable<T>`, сохраняя возможность добавлять новые выражения к запросу, тогда как `IEnumerable<T>` возвращают `IEnumerable<T>` и сразу начинают перебирать элементы.

#### Практический контекст
В современных приложениях с Entity Framework Core (версии 5, 6+) и другими ORM методы с `IQueryable<T>` используются для построения эффективных SQL-запросов с ленивыми вычислениями. Методы для `IEnumerable<T>` удобны для работы с уже загруженными в память структурами данных, например, коллекциями в памяти на клиенте или тестовыми данными. Понимание этих различий критично для оптимизации производительности и правильного построения бизнес-логики на уровне выборки данных.
