# Есть четыре ViewModel, и нужно каждые 5 секунд подписываться на запрос из сети. Как это лучше организовать?

### Категории

Android, Behavioral, ViewModel, Network, Async, Architecture, Subscription

### Краткий ответ

**Как организовать периодическую подписку на сеть для 4 ViewModel каждые 5 секунд?**

- Контекст: Android/MVVM, многопоточное программирование, RxJava/Coroutines
- Использовать один **общий источник данных** с таймером, а не по отдельности для каждой VM
- Применить **Flowable.interval** (RxJava) или **flow + delay** (Coroutines) для периодики
- Подписка в репозитории, кэширование результата, распределение данных через **LiveData/StateFlow**
- ViewModel подписываются на единый стрим, уменьшается нагрузка и дублирование запросов
- Обработка ошибок и отмена подписки при уничтожении ViewModel (lifecycle-aware)
- Такой подход улучшает **производительность** и упрощает **тестирование**

---

**Детали:**

1. В репозитории создать поток (например, RxJava Flowable.interval(5s)), делать запросы каждые 5 секунд.
2. Запросы выполняются в едином месте — репозитории, результаты помещаются в BehaviorSubject или StateFlow.
3. Каждая ViewModel подписывается на этот поток, получает актуальные данные.
4. При остановке VM подписка отменяется, предотвращая утечки.
5. Если использовать Coroutines, для запуска периодической задачи подойдет `while(isActive) { delay(5000); запрос; }`.

---

**Резюме:** центральная периодическая подписка с единым управлением, подписка ViewModel на общий поток — оптимальное, масштабируемое решение.

### Подробный ответ
#### Основной ответ
Для организации периодического обновления данных с сервера каждые 5 секунд для четырёх ViewModel оптимально использовать единый источник данных с общим таймером и реактивным потоком (например, через RxJava или Kotlin Flow). Каждая ViewModel подписывается на этот поток, получая обновления, что предотвращает дублирование сетевых запросов и повышает эффективность.

#### Ключевые моменты
- **Единый таймер**: создаём один Flow/Observable с интервалом 5 секунд, который выполняет сетевой запрос и эмитит результат.
- **Общий кэш/репозиторий**: запросы идут в репозиторий, который кеширует данные, чтобы все ViewModel получали синхронизированные и обновлённые результаты, снижая количество сетевых вызовов.
- **Подписки во ViewModel**: каждая ViewModel подписывается на поток из репозитория, и получает новые данные по мере поступления.
- **Обработка ошибок**: реализуем retry, backoff, и состояние загрузки для устойчивости.
- **Жизненный цикл**: важно учитывать отписку (dispose, cancel), чтобы избежать утечек памяти.

#### Практический контекст
В Android разработке, например, используют Kotlin Coroutines с Flow и функцию `flow { while (isActive) { emit(request()); delay(5000) } }`, которую репозиторий предоставляет вьюмоделям. Такой подход позволяет единожды выполнять запрос и передавать данные сразу четырём ViewModel, обеспечивая производительность и консистентность данных.
