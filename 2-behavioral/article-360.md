# Как правильно подключить http.Server.Shutdown с воркерами и ресурсами?

### Категории

Behavioral, API, Microservices, Resource management, Message brokers, Integration, Concurrency, Workers, Graceful shutdown

### Краткий ответ

**Как правильно подключить http.Server.Shutdown с воркерами и ресурсами**

- Контекст: graceful shutdown HTTP сервера в Go  
- http.Server.Shutdown — плавное завершение: закрывает слушатель, ждёт завершения текущих запросов  
- Воркеры (горутины) нужно корректно завершать через контекст с таймаутом  
- При Shutdown передача **context.Context** с таймаутом для ограничения времени ожидания  
- Использовать sync.WaitGroup для ожидания завершения всех воркеров  
- Закрывать дополнительные ресурсы (БД, кэш, файлы) после Shutdown сервера  
- Практика: обработка сигнала ОС (SIGINT/SIGTERM) для запуска Shutdown  
- Максимум ожидания — для предотвращения зависаний и утечек ресурсов  

---

**Примерный алгоритм:**

1. Создаёте контекст с таймаутом для Shutdown  
2. Вызов http.Server.Shutdown(ctx)  
3. Воркеры отслеживают ctx.Done() и завершаются  
4. sync.WaitGroup фиксирует завершение всех горутин  
5. После завершения всех — закрываете внешние подключения и ресурсы  
6. Выход из приложения  

---

Такой подход гарантирует безопасное освождение ресурсов и предотвращает обрыв запросов на клиентов, обеспечивая стабильность и корректность завершения HTTP-сервера и фоновых процессов.

### Подробный ответ
#### Основной ответ
Правильное подключение `http.Server.Shutdown` в Go с воркерами и управлением ресурсами — это классический кейс graceful shutdown, когда сервер корректно завершает все активные соединения и освобождает ресурсы при получении сигнала остановки (например, SIGINT или SIGTERM). Основная идея — использовать контекст с таймаутом или дедлайном для `Shutdown`, чтобы дать воркерам время завершить текущие задачи, а затем закрыть сервер.

#### Ключевые моменты
- Для отслеживания фоновых воркеров обычно используют `sync.WaitGroup`. После получения сигнала, перед вызовом `Shutdown`, важно остановить приём новых задач и ждать завершения воркеров.
- `http.Server.Shutdown(ctx)` закрывает HTTP сервер на принятие новых соединений и ждёт завершения всех текущих обработчиков, прерывая их по таймауту контекста.
- Правильный паттерн включает горутину, слушающую сигналы OS (через `os.Signal` + `signal.Notify`), с последующим вызовом `Shutdown` и ожиданием завершения работы всей программы.

#### Практический контекст
Типичный код выглядит так:

```go
srv := &http.Server{Addr: ":8080", Handler: myHandler}

var wg sync.WaitGroup
ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
defer cancel()

// Запуск сервера в отдельной горутине
go func() {
    if err := srv.ListenAndServe(); err != http.ErrServerClosed {
        log.Fatalf("ListenAndServe error: %v", err)
    }
}()

// Захват сигнала остановки
stop := make(chan os.Signal, 1)
signal.Notify(stop, syscall.SIGINT, syscall.SIGTERM)
<-stop

// Остановка приёма новых задач и ожидание воркеров
// wg.Wait() - завершаем рабочие горутины

// Выключение HTTP сервера с таймаутом
if err := srv.Shutdown(ctx); err != nil {
    log.Printf("Server Shutdown error: %v", err)
}
```

Такое решение гарантирует, что сервер не примет новых запросов, текущие обработчики успеют завершиться в разумное время, а параллельные воркеры корректно закончат работу до полного завершения программы.

---

Если воркеры используют свои каналы или очереди, важно правильно закрывать их и синхронизироваться через `WaitGroup` перед вызовом `Shutdown` или после — чтобы избежать потерянных горутин или дедлоков.

Более сложные системы могут использовать дополнительные сигналы состояния и схемы управления задачами, но описанный паттерн — базовый и промышленный стандарт.
