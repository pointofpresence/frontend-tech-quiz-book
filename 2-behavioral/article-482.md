# Чем статическая типизация в Go отличается от утиной?

### Категории

Behavioral, Duck typing, Technical interview, Concurrency, Context management, Backend, Static typing, Data structures, Memory

### Краткий ответ

**Различия статической типизации Go и утиной типизации**

- **Go:** строгая статическая типизация  
  - тип переменной задаётся явно при объявлении  
  - ошибки типов выявляются на этапе компиляции  
  - операции позволяют только совместимые типы  
- **Утиная типизация:** проверка по поведению (методам)  
  - неявное согласование типа через реализованные методы  
  - позволяет работать с разными типами, если они имеют нужный интерфейс (поддерживают методы)  
  - типовая совместимость определяется динамически  
- Go сочетает **жёсткую** статическую типизацию с **утиной** в интерфейсах: интерфейсы реализуются неявно  
- Такой подход даёт баланс **безопасности типов** и **гибкости полиморфизма**  
- Пример: в Go можно передать в функцию значение любого типа, реализующего интерфейс, без явного указания соответствия  
- Отличие: статическая типизация ограничивает операции на уровне компиляции, утиная — ориентирована на поведение объектов  

Таким образом, в Go статическая типизация формальная и строгая, но интерфейсы реализуются утиной типизацией, обеспечивая гибкий полиморфизм без явного наследования.

### Подробный ответ
#### Основной ответ
В языке Go используется **статическая типизация с утиной типизацией** (duck typing), но важно понимать, как именно они сочетаются. **Статическая типизация** в Go означает, что типы переменных и выражений проверяются компилятором на этапе компиляции, что исключает множество ошибок во время выполнения. При этом **утиная типизация** реализована через интерфейсы: объект считается подходящим для интерфейса, если он реализует нужные методы, без явного указания соответствия интерфейсу.

#### Ключевые моменты
- **Статическая типизация** в Go гарантирует, что все типовые проверки происходят до запуска программы, исключая runtime-type ошибки, как в динамически типизированных языках.
- **Утиная типизация через интерфейсы Go** не требует явного объявления того, что тип реализует интерфейс — если у типа есть все методы интерфейса, он автоматически считается его реализацией. Это отличается от классической системы с явно заявленными связями.
- Такой подход обеспечивает гибкость, позволяя писать более абстрактный и расширяемый код без потери безопасности типизации. Это ключевой trade-off: Go сочетает строгую проверку типов с удобством динамической подстановки интерфейсов.

#### Практический контекст
В реальных проектах в Go часто используют интерфейсы для написания тестируемого и расширяемого кода, где конкретная реализация интерфейса может меняться без рефакторинга связей. Например, стандартная библиотека заставляет реализовать интерфейс `io.Reader` — любой тип с методом `Read([]byte) (int, error)` автоматически его реализует. Это показывает мощь утиной типизации на базе строгой статической системы.
