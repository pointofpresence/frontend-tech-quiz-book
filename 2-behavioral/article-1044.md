# Почему unowned считается быстрее weak? Как это реализовано на уровне runtime?

### Категории

Reference counting, Behavioral, Memory management, Runtime, Optimization

### Краткий ответ

**Почему unowned считается быстрее weak? Как это реализовано на уровне runtime?**

- Swift, ARC, управление памятью
- **unowned** — неконролируемая ссылка, не увеличивает счетчик
- **weak** — контролируемая, увеличивает и уменьшает счетчик с синхронизацией
- weak требует **атомарного доступа** к хранимому объекту (runtime проверка nil)
- unowned не требует проверки, поэтому быстрее
- runtime unowned хранит только указатель без дополнительных блоков
- weak реализован через таблицу слабых ссылок и наблюдателей деалокации
- практическое значение: unowned быстрее, но unsafe при неверном использовании (crash), weak безопаснее, но медленнее из-за runtime overhead

### Подробный ответ
#### Основной ответ
**Unowned** и **weak** ссылки — две разные формы слабых ссылок в Swift и других языках с ARC (automatic reference counting). Unowned считается быстрее, потому что он не использует механизм отслеживания изменений объекта и не требует проверки на nil при каждом доступе, тогда как weak ссылки хранятся как optional и runtime должен проверять их актуальность, чтобы избежать dangling pointers. 

#### Ключевые моменты
- **Unowned ссылка** — это не-optional, она предполагает, что объект всегда существует в момент обращения. runtime не вставляет дополнительные проверки, и при попытке обращения к удалённому объекту происходит аварийный сбой (crash), что снижает накладные расходы.
- **Weak ссылка** хранит объект опционально (`Optional<T>`), и runtime использует встроенный механизм отслеживания жизненного цикла объекта, чтобы автоматически занулять weak ссылки при деалокации, что требует дополнительной синхронизации и atomic операций с памятью.
- В плане реализации, **weak** ссылки хранятся в специальной таблице слабых ссылок runtime (weak table) для каждого объекта, где управляется их зануление. В отличие от этого unowned не регистрируются в этой таблице, а просто хранят указатель на объект. Это снижает стоимость чтения и записи.

#### Практический контекст
Unowned ссылки обычно применяются в сценариях, где циклические зависимости невозможны и гарантируется срок жизни объекта-климбающий, например в delegate-паттерне между строгими владельцами. Weak используют там, где жизненный цикл не гарантирован и требуется безопасная обработка нуля, например в реактивных цепочках с потенциально отсутствующими объектами.

Таким образом, выбор между unowned и weak — компромисс между безопасностью (weak — nil-сейф) и производительностью (unowned — прямой указатель без проверок). В Swift 5+ реализация weak ссылок оптимизирована, но накладные расходы остаются заметными при интенсивном использовании.
