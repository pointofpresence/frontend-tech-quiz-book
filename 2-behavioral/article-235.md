# Как устроены индексы с точки зрения структур данных?

### Категории

Indexing, Behavioral, Microservices, Network, Business logic, Performance, Backend, Database, Index, Optimization, Data structures, Queues, Caching

### Краткий ответ

**Как устроены индексы с точки зрения структур данных?**

- Индекс — структура для быстрого поиска в БД  
- Часто реализуется как **B-дерево** или **B+-дерево**  
- B-дерево поддерживает **балансировку** и логарифмическое время поиска (O(log n))  
- Альтернатива — **хеш-таблица** для точного соответствия (O(1) поиск)  
- В B+-дереве все данные хранятся в листах, упрощая **диапазонные запросы**  
- Индексы уменьшают количество операций чтения с диска  
- Используются для оптимизации выборок и сортировок в SQL  

Итог: индекс — это **специализированная структура данных** (обычно B-дерево), обеспечивающая быстрое доступ и фильтрацию данных в БД.

### Подробный ответ
#### Основной ответ
Индексы в базах данных реализованы с помощью специализированных структур данных, которые обеспечивают быстрый доступ к записям без полного сканирования таблицы. Наиболее распространённым вариантом является **B-дерево** (B-Tree) и его вариации (например, B+Tree). Этот тип дерева сбалансирован по высоте, что гарантирует логарифмическую сложность поиска, вставки и удаления. 

#### Ключевые моменты
- **B-дерево / B+дерево** — состоит из узлов с упорядоченными ключами и ссылками на дочерние узлы; в B+дереве данные хранятся только в листах, а внутренние узлы служат только для навигации. Это оптимально для дисковых операций, так как минимизирует число чтений блоков.
- **Хеш-индексы** используются для точного соответствия ключам (например, в HashMap), обеспечивая очень быстрый поиск за O(1) в среднем, но не поддерживают диапазонные запросы и сортировку.
- **Inverted indexes** — структура для полнотекстового поиска, где индекс хранит маппинг слов к документам, реализуемый часто через хеш-таблицы или деревья.
- Trade-off между структурами: B-деревья универсальны для диапазонных запросов и сортировки, тогда как хеш-индексы быстрее для равенств.

#### Практический контекст
В PostgreSQL и MySQL InnoDB по умолчанию используется B+Tree для большинства индексов, что обеспечивает стабильную производительность при различных типах запросов. В Redis для ускорения поиска часто применяются skip lists, а в полнотекстовых поисках — inverted index. Понимание структуры данных индекса помогает оптимизировать запросы и выбирать подходящий тип индекса под конкретные задачи.
