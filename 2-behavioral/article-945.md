# Какие ограничения и недостатки есть у REST API?

### Категории

Behavioral, API, Limitations, QA, Architecture, Testing

### Краткий ответ

**Ограничения и недостатки REST API**

- архитектурный стиль, основанный на HTTP, **ограничен протоколом**
- отсутствие стандарта для **ошибок и форматирования** данных
- **сильно зависит** от состояния сервера (REST обычно stateless, но сложно сессии хранить)
- нет поддержки **реального времени** (веб-сокеты лучше)
- избыточность запросов для сложных операций (REST не хорошо с вложенными ресурсами)
- отсутствие строгой типизации (часто JSON, нет контрактов как в gRPC)
- проблемы с версионированием API и **совместимостью** изменений

Итог: REST прост и универсален, но при сложных сценариях может быть **медленнее** и менее гибок, чем альтернативы (GraphQL, gRPC).

### Подробный ответ
#### Основной ответ
**REST API** — это широко используемый архитектурный стиль для построения веб-сервисов, но у него есть ряд ограничений и недостатков, которые важно учитывать при проектировании систем. Основные из них связаны с избыточностью, ограниченной гибкостью и проблемами с версионированием.

#### Ключевые моменты
- **Избыточные данные и чатовое взаимодействие**: REST часто требует нескольких запросов для получения связанных данных (over-fetching и under-fetching), что увеличивает задержки. Например, чтобы получить объект с вложенными связями, приходится делать несколько отдельных вызовов.
- **Отсутствие строгого контракта**: в отличие от gRPC или GraphQL, REST базируется на использовании HTTP и JSON с менее строгими схемами, что усложняет поддержку и эволюцию API, особенно при масштабировании команд.
- **Проблемы с версионированием**: REST не стандартизирует версионирование, и разработчикам приходится либо внедрять его через URL, заголовки или параметры, что может приводить к сложностям в поддержке нескольких версий одновременно.
- **Невозможность полноценно управлять состоянием сессий**: REST следует принципу stateless, что упрощает масштабирование, но может усложнять работу с длинными транзакциями или состояниями, требующими сложной логики.
- **Ограничения по эффективности**: REST API может быть менее эффективен по сравнению с бинарными протоколами (например, gRPC) в плане производительности и нагрузки сети, особенно в системах с высокой частотой вызовов или мобильных приложениях с ограниченной пропускной способностью.

#### Практический контекст
Во многих современных системах REST API дополняют или заменяют на **GraphQL** для решения проблем с выборочной загрузкой данных и более гибким запросом. REST же остаётся отличным выбором для простых CRUD-сервисов и микросервисов с устоявшимися контрактами, где важна понятность и широкая совместимость с HTTP-инструментами.
