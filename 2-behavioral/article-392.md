# Блокирует ли тяжелый код в Promise рендер страницы?

### Категории

Performance, OOP, SOLID, Async, Rendering, Behavioral

### Краткий ответ

**Блокирует ли тяжелый код в Promise рендер страницы?**

- JS в браузере — **однопоточный**  
- Promise — обёртка для **асинхронных** операций, но сам код внутри — **синхронный**  
- если внутри Promise выполняется **тяжёлый синхронный код**, он блокирует главный поток  
- отложенное выполнение (через then/catch) — ставит колбэки в **микротаски**, но код в них тоже синхронный  
- блокировка рендера происходит из-за загрузки главного потока, а не из-за Promise как такового  
- для разгрузки используют **Web Workers** или разбивают работу на части  
- вывод: Promise не снимает синхронную блокировку тяжелого кода, рендер блокируется, если код тяжелый и выполняется в главном потоке  

Итог: Promise не предотвращает блокировку рендера, если в нём есть тяжёлый синхронный код.

### Подробный ответ
#### Основной ответ
Да, тяжелый код внутри **Promise** может блокировать рендеринг страницы, если он выполняется синхронно в основном потоке JavaScript. Хотя Promise сам по себе реализует асинхронный интерфейс, код, который находится внутри функции конструктора Promise или в её синхронном блоке, выполняется немедленно и блокирует главный поток. Таким образом, если внутри Promise выполняются дорогие вычисления без выхода из основного потока, браузер не сможет обновить интерфейс, и рендеринг будет заблокирован.

#### Ключевые моменты
- **Promise не освобождает главный поток сам по себе**: создание Promise и синхронный код внутри его конструктора или вызовов `.then()` выполняются мгновенно; отложенное выполнение начинается только после завершения текущего стэка вызовов.
- **Асинхронность Promise — это механизм очереди задач (microtasks), а не многопоточность**: тяжелые вычисления все равно выполняются в основном потоке.
- Для выполнения тяжелого кода асинхронно без блокировки UI рекомендуется использовать **Web Workers** или разбивать работу на меньшие части с помощью `setTimeout`/`requestIdleCallback`.
- В React 18+ и в некоторых современных браузерах для рендеринга применяется параллелизм и отложенные обновления, но тяжелый синхронный JS все равно блокирует.

#### Практический контекст
В реальных проектах при необходимости выполнить тяжелую обработку данных стоит избегать запуска таких операций в Promise напрямую. Вместо этого используют **Web Workers** для переноса вычислений в отдельный поток или используют техники сплиттинга задач. Например, при обработке больших массивов данных откладывают и дробят операции, чтобы не тормозить отрисовку UI и сохранить плавность взаимодействия, что особенно важно для SPA на React, Vue или Angular.
