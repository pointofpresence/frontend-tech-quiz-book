# Как исследовать проблему, если созданный платеж не отображается в списке после рефреша, но бэк возвращает 200?

### Категории

Behavioral, API, Microservices, Debugging, Backend, Payment systems, Testing

### Краткий ответ

**Как исследовать проблему с платежом, не отображающимся после рефреша, при ответе 200**

- баг фронтенда или кэша — проверить обновление UI и очистку локальных данных  
- валидность данных — сравнить данные ответа API с отображаемыми  
- синхронизация с БД — проверить, действительно ли платеж сохранён в базе  
- логи сервера — удостовериться, что операция вставки прошла успешно без ошибок  
- возможные скрытые ошибки — проанализировать асинхронные процессы, отклонённые промисы, исключения  
- проверка фронтенд-фильтров и условий отображения платежей  
- мониторинг сетевых запросов — повторный GET после POST, анализ возвращаемых данных

Итог: более глубокий анализ клиент-серверного взаимодействия, соответствия данных и состояния UI для выявления причины рассинхронизации.

### Подробный ответ
#### Основной ответ
Если после создания платежа API возвращает **HTTP 200**, но сам платеж не отображается в списке после обновления страницы, это указывает на несогласованность данных между операциями создания и отображения. Для исследования проблемы важно системно проверить все слои: от API до клиентского отображения.

#### Ключевые моменты
- **Проверка ответа сервера и тела ответа**: Ответ 200 — это статус успешного запроса, но важно убедиться, что тело ответа содержит корректные данные о созданном платеже (ID, статус, timestamp). Возможна ситуация, когда сервер подтверждает запрос, но данные фактически не сохраняются или возвращается некорректный результат.
- **Проверка сохранения данных в базе**: На серверной стороне нужно убедиться, что транзакция с созданием платежа действительно проходит и данные сохраняются (например, смотреть логи, консистентность базы, наличие новых записей). Иногда может быть async-очередь или баг с транзакционной логикой.
- **Исследование запроса списка платежей**: Возможно, запрос, который формирует список, фильтрует платежи или не учитывает свежие данные. Стоит проверить фильтры, кэширование на уровне API или клиентского кэша. Часто используется Redis или in-memory кэш, который не обновляется сразу после создания.
- **Кэширование и задержки репликации БД**: В распределённых системах может быть временная задержка репликации или stale cache. Проверить TTL кэша и политики invalidation.
- **Логирование и трассировка запроса**: Использовать полное логирование и трассировку (например, через OpenTelemetry), чтобы отследить путь платежа от создания до выборки.
- **Проверка фронтенда**: Иногда проблема именно в неподходящем обновлении стейта, рендере или работе с данными (например, список не обновляется или использует старые данные из локального состояния).

#### Практический контекст
Типичный подход — использовать интеграционные логи (backend + database), а также network tab в браузере для подтверждения реальных запросов и ответов. В React 18 частый паттерн — после успешного создания явно перезагружать данные (рефетчить данные с сервера) или использовать state management (Redux, React Query) с invalidation. Также на уровне backend проверяют idempotency и atomicity, чтобы исключить ложные 200 ответы.

Такой шаг за шагом анализ позволяет выявить, где именно происходит рассогласование и исправить проблему.
