# Что такое идемпотентность метода? Являются ли DELETE и PATCH идемпотентными?

### Категории

Testing, Architecture, API, HTTP methods, Idempotency, Behavioral

### Краткий ответ

**Что такое идемпотентность метода? Являются ли DELETE и PATCH идемпотентными?**

- идемпотентность — свойство, при котором повторный вызов метода с теми же параметрами не изменяет результат системы после первого вызова
- DELETE — идемпотентен, так как удаление одного ресурса повторно не изменит состояние после первого удаления (ресурс уже отсутствует)
- PATCH — не всегда идемпотентен, зависит от операции; если обновление приводит к одному и тому же конечному состоянию при повторных вызовах, то идемпотентен, иначе — нет
- PATCH используется для частичных изменений, что может влиять на идемпотентность
- практика: DELETE считают стандартно идемпотентным, PATCH требуют анализа конкретных реализаций
- идемпотентность важна для надёжности и обработки повторных запросов без побочных эффектов
- помогает безопасности и масштабируемости API в распределённых системах

### Подробный ответ
#### Основной ответ
**Идемпотентность метода** — это свойство операции при многократном её повторном вызове с одинаковыми входными данными приводить к одному и тому же результату и состоянию системы, как если бы она была вызвана единожды. В контексте HTTP-методов идемпотентность означает, что повторный запрос не изменит дополнительным образом ресурс и не вызовет побочных эффектов после первого вызова.

**DELETE** и **PATCH** — методы с разной идемпотентностью. **DELETE** обычно идемпотентен: если первый вызов удалил ресурс, повторные вызовы будут возвращать тот же результат (например, 404 Not Found), но не изменят состояние сервера дальше. **PATCH** — обычно не является идемпотентным, поскольку применяется частичное обновление, и повторные вызовы с одним и тем же патчем могут привести к накоплению изменений (например, инкремент значений).

#### Ключевые моменты
- **Идемпотентность в HTTP** критична для устойчивости сетевых приложений: клиент может повторять запросы без опасения дублирующих побочных эффектов (например, при таймаутах).
- **DELETE** считается идемпотентным, потому что после удаления ресурса дальнейшие вызовы не меняют состояние; однако важно корректно обрабатывать отсутствующий ресурс.
- **PATCH** реализован для частичных изменений и не гарантирует идемпотентность без дополнительной логики (например, разумного применения патча или хранения его результата).
- Для гарантированной идемпотентности можно использовать методы **PUT**, который всегда заменяет ресурс целиком.

#### Практический контекст
В реальных API часто требуется продумывать идемпотентность операций для обеспечения корректной обработки повторных запросов в условиях ненадёжных сетей. Например, в RESTful API методы DELETE применяют с условием, что ресурс может уже не существовать, что не повод для ошибки. PATCH используют аккуратно, либо реализуют их с идемпотентностью через специальные схемы обновления.
