# Какие шаблоны проектирования используете?

### Категории

Backend, Message broker, Design patterns, Database, Architecture, Behavioral

### Краткий ответ

**Какие шаблоны проектирования используете?**

- Шаблоны — проверенные архитектурные решения
- **Singleton** — ограничение одного экземпляра, для конфигураций и логов
- **Factory Method** — создание объектов без указания конкретного класса
- **Observer** — подписка на события, реактивное обновление
- **Strategy** — выбор алгоритма во время выполнения, повышает гибкость
- **Decorator** — динамическое расширение функционала без изменения классов
- **MVC/MVVM** — разделение ответственности UI и логики
- Использую в зависимости от задачи: для повышения поддержки, читаемости, масштабируемости кода

### Подробный ответ
#### Основной ответ
В своей практике я использую несколько ключевых **шаблонов проектирования**, которые помогают создавать поддерживаемый, масштабируемый и читаемый код. Среди наиболее часто применяемых — это **Singleton**, **Factory**, **Observer**, **Strategy** и **Decorator**. Эти шаблоны позволяют решать типовые архитектурные задачи, структурировать систему и минимизировать дублирование кода.

#### Ключевые моменты
- **Singleton** часто применяю для классов, представляющих глобальные сервисы или конфигурации, где важно гарантировать единственный экземпляр, например, в логировании или работе с подключением к базе данных.
- **Factory** помогает абстрагировать создание объектов, делая код более гибким и тестируемым. Особенно полезен при разнообразии типов объектов, создаваемых на основе входных данных.
- **Observer** — для реализации событийно-ориентированных архитектур или паттерна подписка/уведомление, например, в UI или системах обработки асинхронных событий.
- **Strategy** используется, когда нужно легко менять алгоритмы на лету, не меняя клиентский код. В реальных проектах применял для выбора разных способов валидации или работы с кешем.
- **Decorator** помогает динамически добавлять поведение объектам, что удобно для расширения функционала без изменения исходного класса — типично используется для middleware или добавления кэширования.

#### Практический контекст
Например, в проекте на **React 18** с TypeScript использовал **Observer** вместе с Redux для подписки на изменения состояния, а **Factory** для упрощения создания разных типов UI-компонентов. В backend на **Spring Boot 3** используемый Singleton для класса конфигурации datasource, а Strategy — для выбора разных алгоритмов шифрования данных. Такой подход обеспечивает гибкость разработки и упрощает поддержку системы.
