# Какие паттерны проектирования вы применяли в реальных проектах?

### Категории

Behavioral, Mathematics, Software architecture, OOP, Architecture, Backend, Interview, Design patterns

### Краткий ответ

**Паттерны проектирования в реальных проектах**

- **Singleton**: гарантирует один экземпляр класса, удобно для конфигураций и логирования  
- **Factory Method**: создаёт объекты без прямой привязки к классам, улучшает расширяемость  
- **Observer**: реализует подписку на события, актуально для UI и асинхронных уведомлений  
- **Strategy**: инкапсулирует алгоритмы, позволяет менять поведение динамически  
- **Decorator**: добавляет поведение объектам без изменения их структуры, полезен для расширения функционала  
- **Repository**: абстрагирует доступ к данным, упрощает работу с БД и тестирование  
- **Dependency Injection**: повышает модульность и упрощает тестирование, внедряя зависимости через конструктор или сеттер  
- Применял для **улучшения поддержки**, повышения **гибкости кода** и упрощения **масштабирования** проектов

### Подробный ответ
#### Основной ответ
В реальных проектах я регулярно применял несколько **классических паттернов проектирования**, которые помогают сделать код более устойчивым и масштабируемым. Среди них наиболее часто использовал **Singleton** для управления глобальным состоянием или конфигурациями, **Factory** и **Abstract Factory** для создания объектов при сильной зависимости от контекста, а также **Observer** для реализации событийных систем и реактивного поведения компонентов.

#### Ключевые моменты
- **Singleton** использовал в сервисах логирования и конфигурации, где нужно было гарантировать единственный глобальный экземпляр с низкой задержкой доступа. Особенно актуально в backend-сервисах на Node.js и Java.
- **Factory и Abstract Factory** позволяют централизовать создание сложных объектов с разными вариантами параметров без привязки к конкретным классам, что особенно полезно в проектах с изменяемыми требованиями (например, платежные шлюзы, разные протоколы связи).
- **Observer** применял в frontend-разработке с React (например, подписка на состояние через EventEmitter или сторонние state management библиотеки) и в backend для обработки событий и асинхронных тасков.
- Также часто использовал **Decorator** для динамического расширения функционала без изменения исходного кода, например, при добавлении логирования или кэширования поверх существующих сервисов.
  
#### Практический контекст
В проектах на React 18 и TypeScript Observer интегрирован с Flux-подобными архитектурами, а паттерн Factory применялся для абстракции разных API-интеграций. В микросервисах на Spring Boot часто используется Singleton для бинов, а Decorator помогает дополнять REST-контроллеры кросс-каттинговой логикой, например аутентификацией или мониторингом с помощью Prometheus. Это позволяет экономить время на поддержку и расширение системы без существенных изменений в базовом коде.
