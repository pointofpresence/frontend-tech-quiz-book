# Почему недостаточно лочить только запись? (check-then-act под замком)

### Категории

Locking, Behavioral, API, Microservices, Message brokers, Concurrency, Database

### Краткий ответ

**Почему недостаточно лочить только запись? (check-then-act под замком)**  
- Контекст: проблемы конкуррентности и синхронизации  
- Только запись под замком не защищает проверку (check) от изменений других потоков  
- Между проверкой и действием (act) возможна гонка данных (race condition)  
- Несплошная блокировка приводит к чтению устаревших или неконсистентных данных  
- Требуется совместный (атомарный) блок для check+act → предотвращение TOCTOU (time-of-check-to-time-of-use)  
- Использование расширенных примитивов синхронизации (mutex, rw-lock) или транзакций  
- Практическая польза: корректность, консистентность данных, предотвращение логических ошибок и багов  

Итог: блокировка только записи недостаточна для защиты всей критической секции, нужна атомарность проверки и действия.

### Подробный ответ
#### Основной ответ
Лочить только запись в паттерне **check-then-act** (проверить — затем выполнить действие) недостаточно, потому что между моментом проверки и выполнением действия может произойти **состояние гонки (race condition)**. Если мы не захватываем замок на весь процесс (и на проверку, и на действие), другой поток или процесс может изменить данные после проверки, но до выполнения действия, что приведёт к недетерминированному поведению и повреждению данных.

#### Ключевые моменты
- Проверка и действие должны быть атомарными: захват замка только на действие не защищает от изменений после проверки.
- Между **check** и **act** без лока возможен **race condition**: например, два процесса оба видят одно состояние и пытаются сделать одно и то же действие, что приводит к конфликтам.
- Более универсальное решение — либо блокировать весь блок "check-then-act", либо использовать атомарные операции/транзакции (например, внутри БД с **Serializable Isolation Level**).

#### Практический контекст
Типичная ситуация в многопоточных системах — обновление счётчиков, проверка доступности ресурса, работа с кастомными кешами. В React 18 или Java Concurrent API применяются полные блокировки или атомарные классы (например, AtomicBoolean), чтобы избежать подобных race condition. Также в распределённых системах с Redis используют Lua-скрипты, где весь check-then-act происходит атомарно внутри Redis, а не частично под замком.
