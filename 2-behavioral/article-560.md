# Как обеспечить, чтобы письма, которые были отложено отправлены, не дублировались при горизонтальном масштабировании сервиса?

### Категории

Scaling, Behavioral, Idempotency, Concurrency, Message queue, Email, Backend, Distributed systems

### Краткий ответ

**Как обеспечить отсутствие дублирования отложенных писем при горизонтальном масштабировании?**

- область: распределённые системы, messaging  
- ключевая проблема: **конкуренция при отправке** из нескольких инстансов  
- решение 1 — **идемпотентность**: каждый email с уникальным ID, отправка только один раз  
- решение 2 — **распределённая блокировка** (например, Redis Redlock, Zookeeper), чтобы только один инстанс отправлял конкретное письмо  
- решение 3 — гарантированное **одноразовое извлечение из очереди** (например, использование брокера с поддержкой точной доставки — Kafka, RabbitMQ с manual ack)  
- решение 4 — хранение статуса письма в БД с **атомарным флагом отправки**, чтобы разные инстансы проверяли и обновляли статус транзакционно  
- практическое применение: выбор в зависимости от архитектуры — брокер сообщений + идемпотентность + распределённая блокировка обеспечивают надежность и масштабируемость  
- итог: комбинировать **уникальные идентификаторы, распределённые лocks и атомарные обновления состояния** для исключения дублирования  
 
Так обеспечивают согласованность при одновременной работе множества сервисов и предотвращают повторную отправку писем.

### Подробный ответ
#### Основной ответ
Чтобы избежать дублирования отложенных писем при горизонтальном масштабировании сервиса, необходимо обеспечить **идемпотентность** отправки и корректную координацию между несколькими экземплярами сервиса. Ключевым подходом является централизованное управление состоянием очереди отложенных писем и блокировка задач на уровне хранилища, чтобы каждый email отправлялся ровно один раз.

#### Ключевые моменты
- **Использование атомарных операций и механизма блокировок** на уровне БД или распределённого кеша (например, Redis с RedLock), чтобы сервисы конкурировали за выполнение задачи, но только один мог начать отправку письма.
- **Хранение статуса отправки** (pending, sending, sent, failed) для каждого письма в БД с уникальным идентификатором. Перед отправкой экземпляр сервиса должен "захватить" письмо, обновив статус atomically, предотвращая параллельную отправку.
- **Идемпотентность** на уровне самого SMTP-сервера или API отправки — если сервис случайно попробует повторно отправить письмо, внешний сервис не будет создавать дубликат (например, передавая уникальный Message-ID).
- Опционально, использование **distributed task queue** с поддержкой гарантии однократного выполнения, например RabbitMQ с подтверждениями или Apache Kafka с трансакционностью, где отложенные задачи хранятся централизованно и обрабатываются одним воркером.

#### Практический контекст
В реальных системах на базе PostgreSQL 14+ часто реализуют поведение через поле статуса и SELECT FOR UPDATE для блокировки записи с письмом. Для масштабирования используют Redis с RedLock, чтобы гарантировать, что только один инстанс “захватит” задачу. В больших системах акцент делают на событиях: очередь сообщений (Kafka) и гарантированная обработка с помощью idempotent consumers помогает избежать рассинхронизации и дублирования.
