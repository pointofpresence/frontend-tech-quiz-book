# Какие правила включает REST-архитектурный стиль?

### Категории

Behavioral, API, Promises, Async, Web, Frontend, Templates

### Краткий ответ

**Какие правила включает REST-архитектурный стиль?**

- REST — архитектурный стиль для **веб-сервисов**
- **Клиент-серверная** архитектура: разделение UI и данных
- **Бесподсостояние** (stateless): каждый запрос независим
- Использование **кеширования** для ускорения ответов
- Единый интерфейс: 
  - идентификация ресурсов через **URI**
  - операции через стандартные HTTP-методы (GET, POST, PUT, DELETE)
- Многоуровневая система: промежуточные серверы для масштабирования
- Самоописываемые сообщения: данные + метаданные для обработки

Эти правила обеспечивают **масштабируемость**, **управляемость** и **простоту интеграции** REST API, широко применяемых в современных распределённых системах.

### Подробный ответ
#### Основной ответ
**REST (Representational State Transfer)** — это архитектурный стиль для построения распределённых систем, в частности веб-сервисов, ориентированных на использование стандартных методов HTTP. Он описывает набор правил и ограничений, которые обеспечивают простоту, масштабируемость и независимость компонентов.

#### Ключевые правила REST-архитектуры
- **Клиент-серверная архитектура**: разделение на клиента и сервера позволяет улучшить независимость и масштабируемость системы.
- **Отсутствие состояния (statelessness)**: каждый запрос от клиента к серверу должен содержать всю необходимую информацию для обработки — сервер не хранит состояние между запросами. Это упрощает масштабирование и отказоустойчивость.
- **Кэширование**: ответы сервера должны явно указывать, можно ли их кэшировать и на какое время, чтобы уменьшить нагрузку и увеличить производительность.
- **Единый интерфейс (Uniform Interface)** — ключевое ограничение, состоящее из четырёх частей:
  - Идентификация ресурсов через URI
  - Манипуляция ресурсами через представления (представление ресурса — обычно в JSON, XML, HTML)
  - Самоописывающие сообщения (включая HTTP методы — GET, POST, PUT, DELETE)
  - Гипермедиа как средство навигации (HATEOAS) — отвечает за переход по состояниям приложения
- **Многоуровневая система**: архитектура может иметь промежуточные серверы (прокси, балансировщики), скрывающие детали нижних уровней.
- **Код по требованию (опционально)**: сервер может отправлять исполняемый код клиенту (например, JavaScript) для расширения функционала.

#### Практический контекст
REST широко применяется в создании API для веб-приложений и мобильных сервисов (например, API в React 18+ проектах). Соблюдение указанных правил обеспечивает простоту масштабирования, кэширования (через HTTP headers) и улучшает интеграцию между разными системами. Важно помнить, что многие RESTful API не всегда полностью реализуют все ограничения, но ключевыми остаются безсостояние и единый интерфейс.
