# Если у тебя опыт в дебаге утечки памяти? Прилетает задача, что кончилось место в оперативной памяти, твои действия?

### Категории

Scope, Behavioral, HitTest, Performance, Debugging, Memory leak, Live coding, iOS

### Краткий ответ

**Опыт дебага утечек памяти и действия при критическом использовании RAM**

- профиль памяти: применяю **инструменты профилирования** (Valgrind, perf, VisualVM, Memory Profiler)
- анализ дампов: исследую **heap snapshot**, ищу объекты без освобождения
- проверка циклических ссылок и ресурсов без удаления (**listeners**, **сокеты**, **файлы**)
- использую **логирование** аллокаций и освобождений памяти
- рефакторинг: минимизация удержания больших объектов, **оптимизация кода**
- автоматизация тестов на выявление утечек (stress-tests, load-tests)
- при критической ситуации: временное ограничение функций, рестарт служб, **измельчение задач** на мелкие процессы  
- в итоге — предотвращение роста потребления, повышение стабильности приложения и контроль через мониторинг  
- применяю цикличные проверки и CI-интеграцию профайлеров для раннего обнаружения утечек  

Это последовательный процесс диагностики, локализации и устранения проблем с памятью.

### Подробный ответ
#### Основной ответ  
Да, у меня есть опыт в диагностике и устранении утечек памяти. В ситуации, когда оперативная память завершается, я действую системно: сначала пытаюсь воспроизвести проблему, затем собираю и анализирую метрики и дампы памяти, чтобы определить источник утечки. Обычно это бывает вызвано накапливающимися объектами, которые не освобождаются из-за ошибок в коде, замыканий, глобальных ссылок или неправильного управления ресурсами.

#### Ключевые моменты  
- **Сбор метрик и логов**: использую профилировщики (например, VisualVM, YourKit, Valgrind для нативного кода) или встроенные инструменты (jmap, jstack для Java) для анализа потребления памяти, heap dump и tracking allocations. Важно понять, растет ли GC, фрагментируется ли память.  
- **Выделение подозрительных областей**: смотрю на объекты с большим retention size и цепочки ссылок, которые не дают сборщику мусора освободить память. Анализирую, где и почему ссылки сохраняются и кто их удерживает.  
- **Устранение корня утечки**: исправляю код: убираю ненужные глобальные ссылки, закрываю правильно ресурсы, пересматриваю cache/pool, чтобы не было бесконечного накопления. В React или frontend — слежу, чтобы компоненты корректно размонтировались и очищали слушателей.

#### Практический контекст  
На практике часто использую Prometheus и Grafana для мониторинга памяти в проде — чтобы быстро заметить аномалии. В крупных сервисах с Java 11+ при проблемах с OutOfMemoryError сначала снимаю heap dump, затем анализирую с помощью Eclipse MAT или jvisualvm. Если сервис на C++ — применяю AddressSanitizer или Valgrind для поиска подвешенного выделения памяти. Благодаря системному подходу удалось повысить стабильность и снизить инциденты с MEMORY_LIMIT_EXCEEDED.
