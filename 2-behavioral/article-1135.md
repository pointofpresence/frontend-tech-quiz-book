# Как повлияет тип получателя (по значению или по ссылке) на реализацию интерфейса двумя структурами?

### Категории

Behavioral, Methods, Interfaces, Value receiver, Pointer receiver, Backend, Programming concepts

### Краткий ответ

**Влияние типа получателя на реализацию интерфейса двумя структурами**  
- контекст: реализация интерфейсов в Go  
- метод с **значением** принимает копию объекта  
- метод с **ссылкой** (указателем) может изменять состояние объекта  
- интерфейс реализует тип, если все методы совпадают по сигнатурам  
- значение реализует интерфейс, если методы с получателем-значением  
- ссылка реализует интерфейс, если методы с приемником-ссылкой или значением  
- разница в том, что ссылка может вызвать методы с указателем, значение — нет  
- применение: выбор зависит от потребности изменять состояние или экономить копии объектов

### Подробный ответ
#### Основной ответ
Тип получателя (receiver) в методах Go — по значению или по ссылке — существенно влияет на реализацию интерфейсов двумя структурами. В Go интерфейс реализуется, если у типа есть методы с нужной сигнатурой, причем тип и тип указателя считаются разными типами с разными наборами методов. Если метод реализован с получателем по значению, и структура реализует интерфейс, то значение структуры и её указатель оба считаются реализующими интерфейс. Если же метод реализован с получателем по ссылке (указателю), только указатель на структуру реализует интерфейс, а значение — нет.

#### Ключевые моменты
- **Полиморфизм и метод-сеты:** значение структуры содержит методы с получателем по значению, указатель — и методы с получателем по значению, и по ссылке (с учетом встроенности).
- **Интерфейс реализуется типом или указателем:** если метод с receiver *T (указатель), то значение T не реализует интерфейс, только *T; если receiver T (значение) — и T, и *T реализуют.
- **Практическое влияние:** если у вас две структуры с методами на получателях разного типа (одна по значению, другая по ссылке), придётся создавать переменные разных типов — значения или указатели, чтобы они удовлетворяли одному интерфейсу. Это важно при хранении в срезах интерфейсов или передаче в функции.

#### Практический контекст
В реальных проектах часто для оптимизации (избежать копирования, модификация состояния) делают методы с pointer receiver — тогда интерфейс реализует только указатель на структуру. В случаях, когда структуры небольшие и immutable, практичнее использовать value receiver, тогда обе формы — и значение, и указатель реализуют интерфейс, что упрощает использование. Типично в крупных системах предпочтение отдают pointer receiver для структур с изменяемым состоянием, чтобы интерфейсы были консистентны.
