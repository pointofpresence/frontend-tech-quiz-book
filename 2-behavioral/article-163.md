# Как синхронизировать данные между командной и Query базой в паттерне CQRS?

### Категории

Behavioral, Architecture, Data synchronization, Pattern, Database, Design

### Краткий ответ

**Как синхронизировать данные между командной и Query базой в паттерне CQRS?**

- CQRS разделяет **командную (write)** и **запросную (read)** базы для оптимизации
- Синхронизация через **асинхронную репликацию** изменений
- Используют **event sourcing** или **событийную шину (event bus)**
- Командная основа генерирует и публикует **события изменений**
- Query-система **подписывается** на события и обновляет проекцию БД
- Обеспечивается **консистентность eventual consistency**, а не жёсткая синхронизация
- Выгода: отделение нагрузки и масштабируемость, минимизация блокировок  

Детали: важна надежная доставка событий и обработка ошибок для корректного sync.

### Подробный ответ
#### Основной ответ
В паттерне **CQRS (Command Query Responsibility Segregation)** данные разделены на две части: командную (write model) и query (read model) базы. Синхронизация между ними — ключевая задача, поскольку команды изменяют состояние, а запросы читают оптимизированные проекции. Обычно для этого применяется **асинхронная репликация через события**. При изменении командной базы генерируются события (Event Sourcing или просто domain events), которые потребляются сервисом проекций для обновления query базы. Это гарантирует eventual consistency между моделями.

#### Ключевые моменты
- **Event-driven синхронизация:** команды отправляют domain events в шину сообщений (например, Kafka, RabbitMQ, или даже Azure Event Grid) сразу после успешных транзакций.
- **Идемпотентность проекций:** обработчики событий для query базы должны быть идемпотентными, чтобы избежать проблем из-за дублирования или повторной обработки сообщений.
- **Eventual consistency:** между двумя базами может быть небольшая задержка (обычно десятки мс — 1-2 секунды), что важно учитывать в архитектуре и UX.
- **Обработка ошибок и согласованность:** для надежности используют механизмы повторной попытки, dead-letter очереди, мониторинг состояния обработки событий.

#### Практический контекст
Например, в микросервисной архитектуре write-сервис публикует события изменений в Kafka топик. Read-сервис подписывается на этот топик, трансформирует события и обновляет свою оптимизированную базу (ElasticSearch, MongoDB, Redis). Такое разделение позволяет масштабировать запросы и обрабатывать их с минимальной задержкой, при этом сохраняя консистентность бизнес-логики в write модели.
