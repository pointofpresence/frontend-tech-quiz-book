# Как Postgres хранит данные внутри себя?

### Категории

Behavioral, Architecture, Storage, Database, Backend

### Краткий ответ

**Как Postgres хранит данные внутри себя?**

- Реляционная СУБД с **табличной** структурой  
- Основной блок — **страница (page)**, обычно 8 KB  
- Данные хранятся в **heap-таблицах** (несортированные страницы)  
- В каждой странице — **записи (tuples)** с системным заголовком  
- Использует **MVCC (Multi-Version Concurrency Control)**: хранит версии строк через **xmin/xmax** для параллелизма  
- Индексы (например, **B-tree**) позволяют быстрый доступ, не затрагивая heap  
- Страницы организованы в **файлы по объектам БД**, хранятся на диске в каталоге базы  
- WAL (Write-Ahead Logging) гарантирует целостность и восстановление после сбоев

Это обеспечивает **эффективное хранение**, параллельность и отказоустойчивость.

### Подробный ответ
#### Основной ответ
**PostgreSQL** хранит данные в виде страниц (pages) фиксированного размера по 8 КБ, упакованных в файлы, которые составляют таблицы и индексы. Каждая таблица — это набор физических файлов на диске, организованных в сегменты по 1 ГБ. Данные внутри страниц структурированы в строки (tuples), которые содержат служебную информацию (like xmin, xmax) для поддержки MVCC (многоверсионности). Для хранения транзакционной целостности PostgreSQL использует систему хранения WAL (Write-Ahead Logging), обеспечивающую восстановление и консистентность базы.

#### Ключевые моменты
- **Формат страниц**: страницы — минимальная единица чтения/записи (8 КБ), в них хранятся строки и служебные заголовки. Структура страницы оптимизирована под быстрый поиск и обновление.
- **MVCC**: с помощью полей xmin/xmax и информации о транзакциях реализуется многоверсионность — актуальные версии строк видны, а старые — игнорируются без блокировок.
- **Структура таблиц**: данные хранятся в файловой системе как набор сегментов в каталоге базы (PGDATA/base/), где каждый сегмент — файл размером до 1 ГБ.
- **Индексы**: отдельные файлы хранятся в своей файловой структуре, привычный индекс — B-tree с плотным размещением данных, что обеспечивает быстрый поиск.
- **WAL**: журналы для обеспечения durability и восстановления данных в случае сбоев.

#### Практический контекст
Такой подход позволяет PostgreSQL быть надежным и продуктивным в OLTP и OLAP сценариях. Например, при обновлении записи PostgreSQL не перезаписывает данные напрямую, а создает новую версию строки, благодаря чему запросы не блокируются, а обеспечивается высокая параллелизация и консистентность на уровне snapshot isolation. В реальных проектах для анализа структуры хранения удобно использовать утилиту `pg_filedump` или расширения вроде `pgstattuple` для диагностики состояния таблиц и страниц.
