# Какие классовые паттерны проектирования ты знаешь и используешь?

### Категории

Behavioral, Mobile, OOP, Architecture, Design patterns, iOS

### Краткий ответ

**Классовые паттерны проектирования**

- Порождающие (создание объектов): **Singleton**, **Factory Method**, **Abstract Factory**, **Builder**, **Prototype**
- Структурные (организация классов): **Adapter**, **Bridge**, **Decorator**, **Composite**, **Facade**
- Поведенческие (взаимодействие объектов): **Strategy**, **Template Method**, **Observer**, **State**, **Chain of Responsibility**
- Использую, исходя из задачи:
  - **Singleton** — для ограничения экземпляра
  - **Factory Method** — создание подклассов объектов
  - **Decorator** — добавление поведения без наследования
  - **Strategy** — смена алгоритмов на лету
  - **Observer** — реакция на события
- Ключевое — паттерны улучшают **расширяемость**, **поддержку** и **понимание** кода  
- Выбираю паттерны с учётом **простоты** и **конкретных требований** проекта

### Подробный ответ
#### Основной ответ
Классовые паттерны проектирования — это шаблоны, которые помогают правильно организовать отношения между классами и объектами для решения типовых задач архитектуры и дизайна ПО. Они направлены на эффективное управление наследованием, инкапсуляцией и взаимодействием классов. Основные классовые паттерны делятся на три группы: порождение объектов, структурные и поведенческие. В своей практике я часто использую паттерны из каждой группы, чтобы повысить модульность, расширяемость и читаемость кода.

#### Ключевые паттерны и их применение
- **Singleton** — гарантирует существование только одного экземпляра класса, полезен для управляемых ресурсов (например, конфигурация, логгер). В React 18 и современных приложениях избегаю чрезмерного использования из-за сложности с тестированием, но для управления глобальным состоянием применяю аналоги.
- **Factory Method и Abstract Factory** — для создания объектов без жесткой привязки к конкретным классам, улучшает расширяемость системы. Например, в микросервисной архитектуре для создания специфичных клиентов API.
- **Adapter** — класс, который преобразует интерфейс одного класса к интерфейсу, который ожидают клиенты. Часто применяю при интеграции с legacy-кодом или сторонними библиотеками.
- **Decorator** — добавляет функциональность объекту динамически, избегая создания большого количества подклассов. Использую при расширении возможностей без изменения базового кода (например, в middleware).
- **Strategy** — определяет семейство алгоритмов, инкапсулирует каждый из них и делает их взаимозаменяемыми. Удобно для конфигурируемой логики обработки данных.
- **Observer** — позволяет объектам подписываться на события другого объекта, широко применяется для event-driven систем, UI-обновления, в том числе с использованием RxJS или Pub/Sub моделей.

#### Практический контекст
В коммерческих проектах я часто комбинирую классовые паттерны с другими типами (структурными, поведенческими) для построения гибких и расширяемых архитектур. Например, совместно с паттерном Dependency Injection (часто в Spring Boot или NestJS) упрощаю тестирование и поддержку кода. Также применяю паттерны с учётом особенностей конкретных языков и фреймворков, чтобы максимум использовать их возможности.

#### Ключевые моменты
- Классовые паттерны направлены на организацию отношений между классами и объектами
- Часто комбинируются с другими паттернами для решения комплексных задач
- Выбор паттерна зависит от требований проекта, поддерживаемости и производительности

Такой системный подход помогает писать чистый, поддерживаемый код и легко адаптироваться к изменениям.
