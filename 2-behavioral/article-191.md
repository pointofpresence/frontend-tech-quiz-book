# Какие стратегии инвалидации кэша использовали?

### Категории

Behavioral, API, Microservices, Backend, Optimization, Cache

### Краткий ответ

**Стратегии инвалидации кэша**

- **TTL (Time To Live)**: автоматическая очистка после заданного времени
- **LRU (Least Recently Used)**: удаление наименее используемых записей
- **Write-through**: обновление кэша параллельно с базой
- **Write-back**: запись в кэш с отложенным обновлением базы
- **Explicit Invalidation**: ручное удаление по событию или триггеру
- Использовать комбинации для баланса скорости и актуальности
- Выбор стратегии зависит от требований к **свежести данных** и **производительности**

### Подробный ответ
#### Основной ответ
Стратегии инвалидации кэша — это методы обновления или удаления устаревших данных из кэша для обеспечения актуальности и согласованности с основным источником данных. В практике я применял несколько основных подходов, которые помогают сбалансировать производительность и консистентность.

#### Ключевые моменты
- **Time-To-Live (TTL)** — самый простой и распространённый способ, когда у каждого кэш-ключа задаётся время жизни, по истечении которого данные автоматически удаляются. Этот метод эффективен для данных, которые не требуют немедленного обновления, например, горячие, но не критично быстровозобновляемые.
- **Cache-aside (ленивая инвалидация)** — приложение при чтении сначала проверяет кэш, и если данные устарели или отсутствуют, загружает из БД и обновляет кэш. При изменениях данных приложение вручную инвалидаирует соответствующий ключ. Этот подход широко применяется с Redis и Memcached.
- **Write-through и Write-back** — при записи данных в базу кэш либо обновляется синхронно (write-through), либо асинхронно (write-back). Write-through обеспечивает сильную консистентность, но может повлиять на задержки.
- **Event-driven инвалидация** — в микросервисных архитектурах я использую события (через Kafka, RabbitMQ), которые при изменениях данных посылают сигнал обновления или очистки конкретных кэш-ключей.
- **Полная сброска (cache purge)** — применяется в ситуациях с большими изменениями, когда проще сбросить весь кэш, чем управлять множеством ключей. Используется редко, из-за нагрузки на БД.

#### Практический контекст
Например, в проекте на React с backend на Node.js и Redis я использовал cache-aside с TTL 5 минут для страниц с динамическим контентом, и event-driven инвалидацию через WebSocket для real-time обновлений. Для API с высокой нагрузкой применял write-through стратегию, чтобы обеспечить консистентность без задержек. Такой комплексный подход обеспечивает баланс между скоростью отклика и актуальностью данных.
