# Чем отличается функционально GET от POST? Можем ли писать запрос в body GET?

### Категории

HTTP, Behavioral, API, Request methods, Backend integration, Web, Networking

### Краткий ответ

**GET vs POST: функциональные отличия и body в GET**

- HTTP методы для передачи данных, часть REST
- **GET** — запрашивает данные с сервера, параметры передаются в URL (query string)
- **POST** — отправляет данные на сервер, параметры в теле запроса (body)
- GET должен быть **идемпотентным** и не изменять состояние сервера
- POST используется для операций, изменяющих состояние (создание, обновление)
- По спецификации HTTP GET **не поддерживает** тело запроса (body), хотя некоторые клиенты позволяют, серверы часто игнорируют или запрещают
- Практически **записывать body в GET не рекомендуется и не стандартно**

Итог: GET — безопасный метод запроса с параметрами в URL, POST — метод отправки данных с параметрами в теле. Body для GET технически нестандартен и использовать его нельзя.

### Подробный ответ
#### Основной ответ
**GET** и **POST** — это два основных HTTP метода, которые выполняют разные функции. **GET** предназначен для получения ресурсов с сервера и должен быть идемпотентным и безопасным — то есть не изменять состояние сервера. Запросы GET передают параметры через URL (query string). В отличие от него, **POST** используется для отправки данных на сервер, часто для создания или изменения ресурсов, и может изменять состояние сервера. Данные POST обычно передаются в теле запроса (body).

По поводу тела в GET — согласно спецификациям HTTP/1.1 и практике, GET-запросы обычно не содержат body, и даже если его отправить, большинство серверов и библиотек игнорируют или не ожидают тело в GET. Это связано с тем, что семантика GET основывается на передаче данных через URL, и body для GET не стандартизирован и плохо поддерживается.

#### Ключевые моменты
- **Семантика:** GET — запрос на чтение, POST — запрос на создание/изменение (не идемпотентный).
- **Передача данных:** GET — параметры в URL, POST — данные в body (например, JSON, form-data).
- **Поддержка тела:** GET обычно не имеет body; хотя технически возможен, это не рекомендуется, и многие инструменты такое не обработают.
- **Кэширование:** GET-запросы кэшируются браузерами и прокси по умолчанию, POST — нет.
  
#### Практический контекст
В реальных проектах GET используют для получения данных/страниц с query-параметрами, а POST — для форм, API вызовов с отправкой больших объёмов данных (JSON, файлы). Если надо передать много информации, которая не подходит в URL (ограничения длины и безопасность), используют POST. Тело в GET — нестандартная практика и часто вызывает проблемы с совместимостью.
