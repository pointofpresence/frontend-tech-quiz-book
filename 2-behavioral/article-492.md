# Зачем нужно закрывать тело HTTP-ответа в Go?

### Категории

HTTP, Behavioral, Architecture, Concurrency, Database, Backend, Messaging, SQL

### Краткий ответ

**Зачем нужно закрывать тело HTTP-ответа в Go?**  
- HTTP ответ с телом содержит **поток данных** (io.ReadCloser)  
- Закрытие тела освобождает **сетевые ресурсы** и **дескрипторы**  
- Без Close() происходит **утечка памяти** и зависание соединений  
- Закрытие позволяет клиенту **повторно использовать TCP-соединение** (Keep-Alive)  
- Стандарт net/http требует вызова Close() для **гарантии очистки**  
- Практически: defer resp.Body.Close() сразу после проверки ошибки  
- Ключ к **эффективному управлению ресурсами при работе с HTTP**

### Подробный ответ
#### Основной ответ
В Go нужно закрывать тело HTTP-ответа (Body) с помощью метода `Close()` или через `defer resp.Body.Close()`, чтобы освободить сетевые ресурсы и правильно завершить чтение данных. Если не закрывать тело ответа, соединение не будет возвращено в пул повторного использования (connection pool), что приведёт к утечкам ресурсов и, в конечном итоге, к задержкам или исчерпанию доступных соединений.

#### Ключевые моменты
- В стандартном пакете **net/http** тело ответа — это поток данных из TCP-соединения. Закрытие `Body` освобождает underlying connection.
- Если тело не закрывать, HTTP-клиент не может повторно использовать TCP-соединения (Keep-Alive), что снижает производительность из-за постоянного открытия новых соединений.
- В Go начиная с версии 1.13, при чтении полного тела через `io.ReadAll` тело можно закрывать сразу после чтения, но если чтение прервано, закрывать обязательно — это хороший паттерн для предотвращения блокировок.

#### Практический контекст
В реальных проектах, особенно при высоконагруженных API-вызовах, закрытие `resp.Body` — это обязательное условие для поддержания стабильной и масштабируемой работы HTTP-клиентов. Например, в микросервисах с частыми запросами внутри кластера закрытие тела ответа и возврат соединения в пул снижает время отклика до ~50ms и удерживает 99.9% uptime.
