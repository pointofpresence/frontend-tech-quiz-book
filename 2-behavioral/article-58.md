# Какие существуют паттерны проектирования (MVC, MVVM, MVP) и чем они отличаются?

### Категории

Architecture, Behavioral, Design patterns, Frontend

### Краткий ответ

**Паттерны проектирования: MVC, MVVM, MVP**

- Архитектурные паттерны для разделения UI и логики  
- **MVC**: Model (данные), View (интерфейс), Controller (управление)  
- Controller обрабатывает ввод, обновляет Model и View напрямую  
- **MVP**: Presenter связывает View и Model, View пассивна  
- Presenter формирует данные для отображения, управляет логикой UI  
- **MVVM**: ViewModel — абстракция View с биндингом данных  
- View подписана на ViewModel, нет прямого взаимодействия с Model  
- Различия: MVC — Controller активен, MVP — Presenter централизует логику, MVVM — двусторонний биндинг для реактивности  

- Применение зависит от сложности UI и технологий (Web, мобильные, десктоп)

### Подробный ответ
#### Основной ответ
Паттерны **MVC (Model-View-Controller)**, **MVVM (Model-View-ViewModel)** и **MVP (Model-View-Presenter)** — это архитектурные шаблоны, которые помогают разделять ответственность между компонентами приложения для улучшения тестируемости, поддержки и масштабируемости. Все они ориентированы на отделение UI (интерфейса) от бизнес-логики и данных, но различаются структурой взаимодействия и степенью связи между компонентами.

#### Ключевые моменты
- **MVC (Model-View-Controller)**:  
  Model отвечает за данные и бизнес-логику, View — за отображение, Controller — за обработку пользовательского ввода и обновление Model и View. View и Controller тесно связаны, Controller напрямую обновляет View. Часто применяется в веб-разработке (например, ASP.NET MVC).

- **MVP (Model-View-Presenter)**:  
  Presenter берет на себя роль посредника между View и Model. View пассивна — не содержит логики, просто отображает данные. Presenter обновляет View через интерфейс, что повышает тестируемость, т.к. View можно мокать. Часто используется для десктопных приложений или Android.

- **MVVM (Model-View-ViewModel)**:  
  ViewModel — это модель представления, которая содержит логику для подготовки данных для View и взаимодействует с Model. View двунаправленно связана с ViewModel через data binding (например, в WPF, Angular, React с hooks). Это минимизирует код в контроллере/презентере и позволяет UI автоматически реагировать на изменения.

#### Отличия и нюансы:
- В **MVC** контроллер активно управляет View и Model, связь двунаправленная, что иногда усложняет поддержку.
- В **MVP** View пассивна, вся логика в Presenter, легче покрывать тестами UI-логику.
- В **MVVM** основа — двунаправленный binding, что уменьшает код-посредник и удобно для реактивных UI, но требует поддержки binding-фреймворка.

#### Практический контекст
В React 18 часто реализуют MVVM-like подход с хуками и стором (Redux), а в классических Java Spring помогает MVC. В мобильной разработке Android активно используют MVP или MVVM (Jetpack ViewModel + LiveData) для разделения ответственности и удобства тестирования.
