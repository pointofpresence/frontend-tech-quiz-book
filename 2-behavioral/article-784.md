# Каково назначение папок в Go-проекте (cmd, adapters, delivery)?

### Категории

Architecture, Behavioral, Project structure, Backend

### Краткий ответ

**Назначение папок в Go-проекте (cmd, adapters, delivery)**  
- **Организация кода** по ответственности  
- **cmd**: точка входа, главный исполняемый файл  
- **adapters**: интеграция с внешним миром (БД, API, сервисы)  
- **delivery**: слой передачи данных, обработка запросов (HTTP, GRPC)  
- Четкое разделение слоев повышает **поддерживаемость** и тестируемость  
- Следует паттернам **Clean Architecture** / Hexagonal Architecture  
- Помогает масштабировать проект и изолировать зависимости

### Подробный ответ
#### Основной ответ
В структуре **Go-проекта** папки **cmd**, **adapters**, и **delivery** часто используются для организации кода по принципам чистой архитектуры или hexagonal architecture, чтобы чётко разделить бизнес-логику и инфраструктуру.  
- Папка **cmd** содержит точки входа в приложение — исполняемые программы, обычно по одному подкаталогу на каждую команду/сервис.
- Папка **adapters** — слой адаптеров, который преобразует данные и интерфейсы между внешними системами и внутренним ядром приложения.
- Папка **delivery** отвечает за реализацию способов доставки данных, то есть конкретные механизмы взаимодействия с пользователем или другими системами (например, HTTP handlers, gRPC, CLI).

#### Ключевые моменты
- **cmd** — ключевой старт программы, каждое поддиректория там создает отдельный бинарник. Например, `cmd/api` содержит основной HTTP-сервер.
- **adapters** инкапсулирует логику интеграции — адаптеры к базам данных, внешним API, брокерам сообщений. Позволяет не привязываться к конкретным технологиям в ядре.
- **delivery** реализует коммуникационный слой (например, REST-контроллеры, WebSocket, CLI-интерфейс) и переводит входящие запросы в вызовы бизнес-логики.

#### Практический контекст
Такое разделение помогает поддерживать **чистоту архитектуры**: ядро (core) остаётся независимым от технологий, а внешние детали (инфраструктура, интерфейсы) сгруппированы в понятных слоях. В проектах на Go 1.18+ это часто сочетается с использованием интерфейсов и пакетов для повышения тестируемости и масштабируемости.
