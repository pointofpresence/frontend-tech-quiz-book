# Что такое цикл сильных ссылок и как его избежать?

### Категории

Behavioral, Reference counting, Retain cycle, Memory management, Strong reference cycle, ARC, Optimization, Refactoring

### Краткий ответ

**Что такое цикл сильных ссылок и как его избежать?**  
- управление памятью, сборщик мусора  
- **цикл сильных ссылок** — объекты ссылаются друг на друга напрямую  
- из-за цикла GC не освобождает память — **утечка памяти**  
- в языках с подсчетом ссылок (например, Swift, C++) цикл мешает очистке  
- избегают через слабые ссылки (**weak references**) или разбивают циклы  
- используют **weak** или **unowned** ссылки (Swift) для разрыва циклов  
- предотвращает утечки, повышает производительность  
- практика — анализ зависимостей и архитектурный контроль циклов

### Подробный ответ
#### Основной ответ
**Цикл сильных ссылок** — ситуация, когда два или более объекта ссылаются друг на друга через сильные (обычные) ссылки, из-за чего сборщик мусора не может освободить память, даже если эти объекты больше не используются извне. Это приводит к утечке памяти, так как объекты остаются "живыми" из-за взаимных ссылок.

#### Ключевые моменты
- В языках с автоматическим управлением памятью (например, Java, C#, Swift) объект считается доступным для сборки мусора, если до него есть сильные ссылки из корневых объектов. Циклы сильных ссылок создают "островки" объектов, не достижимых извне, но живущих из-за взаимных ссылок.
- Для предотвращения таких циклов используют **слабые ссылки** (weak references), которые не удерживают объект в памяти. Например, в Swift применяют `weak` или `unowned` свойства для разрыва цикла между объектами.
- В ручном управлении памятью (например, в C++ с shared_ptr) циклы сильных ссылок возникают при использовании взаимных shared_ptr без слабых ссылок; для решения применяют `weak_ptr` для одной из сторон.

#### Практический контекст
Типично проблема возникает в системах с observer-паттернами, delegate, или взаимосвязанными объектами UI. Например, в iOS при использовании **closures** и захвате `self` можно легко создать циклы сильных ссылок, которых избегают, используя `[weak self]` или `[unowned self]`. Аналогично, при построении графов объектов или кэшах важно проектировать ссылки так, чтобы не образовывать циклы, либо применять слабые ссылки там, где ссылка не должна владеть объектом.

Таким образом, понимание циклов сильных ссылок и навыки работы со слабыми ссылками — ключевой элемент эффективного управления памятью и предотвращения утечек.
