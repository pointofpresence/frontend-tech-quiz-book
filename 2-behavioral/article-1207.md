# При повороте экрана нужно сохранить и изменить определённые данные в Presenter без ViewModel. На каком этапе жизненного цикла и как это сделать?

### Категории

Android, Behavioral, Architecture, Lifecycle, Senior, State management, Kotlin

### Краткий ответ

**Сохранение данных при повороте экрана без ViewModel**

- контекст: Android, MVP, жизненный цикл Activity  
- задача: сохранить и восстановить данные Presenter при recreate  
- используем: метод **onSaveInstanceState(Bundle)** Activity  
- сохраняем данные в Bundle через ключи в onSaveInstanceState  
- восстанавливаем данные в методе **onCreate(Bundle savedInstanceState)** или **onRestoreInstanceState(Bundle)**  
- передаем восстановленные данные в Presenter (через конструктор, сеттер или метод)  
- таким образом сохраняется состояние без ViewModel, при повороте данные не теряются  
- применяется для небольшого объёма примитивов или сериализуемых объектов  

Основной ответ:  
Сохраняю данные в Bundle в onSaveInstanceState, восстанавливаю в onCreate и передаю в Presenter.

### Подробный ответ
#### Основной ответ
При повороте экрана в Android происходит пересоздание Activity и связанного с ней Presenter, если не предпринимать дополнительных мер. Без использования **ViewModel** для сохранения состояния логично использовать механизм сохранения состояния в методах жизненного цикла **onSaveInstanceState(Bundle)** и **onRestoreInstanceState(Bundle)** или в параметре Bundle метода **onCreate(Bundle)**. В Presenter можно передать сохранённые данные из Bundle для восстановления состояния после пересоздания.

#### Ключевые моменты
- **onSaveInstanceState(Bundle)** вызывается до уничтожения Activity (например, при повороте), здесь можно сериализовать необходимые данные Presenter в Bundle.
- В методе **onCreate(Bundle savedInstanceState)** или **onRestoreInstanceState(Bundle)** получаем Bundle и восстанавливаем данные Presenter.
- Такой подход требует, чтобы данные были сериализуемыми или Parcelable, чтобы поместиться в Bundle.
- Важно, что Presenter должен иметь API для передачи и восстановления данных из Bundle — например, методы `saveState(Bundle)` и `restoreState(Bundle)` вызываемые из Activity.
- Альтернативой может быть хранение состояния в отдельном объекте, например, через Retain Fragment, но это менее популярно в современных приложениях.

#### Практический контекст
В реальных проектах без ViewModel часто используют именно Bundle для сохранения минимального объёма состояния, особенно когда Presenter не управляет сложным состоянием. Для сложных сценариев стоит подумать о внедрении ViewModel или других паттернов сохранения состояния, чтобы избежать утечек памяти и сложной сериализации.
