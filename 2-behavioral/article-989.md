# Расскажи про принципы Solid

### Категории

SOLID, Behavioral, Live coding, OOP, Event loop, Design principles, Software design, Development, Principles, Code review, Interview, Code quality, Testing

### Краткий ответ

**Принципы SOLID**

- SOLID — пять базовых принципов объектно-ориентированного дизайна
- Single Responsibility: класс должен иметь одну ответственность  
- Open/Closed: открыт для расширения, закрыт для изменений  
- Liskov Substitution: подклассы заменяют базовые без нарушений  
- Interface Segregation: интерфейсы специфичны, не заставляют реализовывать лишнее  
- Dependency Inversion: зависимости от абстракций, а не конкретик  

Эти принципы улучшают **поддержку**, **расширяемость** и **тестируемость** кода, минимизируют жёсткую связность и повышают качество архитектуры.

### Подробный ответ
#### Основной ответ
**SOLID** — это пять основных принципов объектно-ориентированного проектирования, которые помогают создавать гибкие, поддерживаемые и расширяемые системы. Эти принципы разработал Роберт Мартин (Uncle Bob) для повышения качества архитектуры ПО и снижения технического долга. Их часто применяют в современных языках вроде Java, C#, а также в TypeScript и других.

#### Ключевые принципы SOLID
- **S — Single Responsibility Principle (Принцип единственной ответственности)**: класс должен иметь только одну причину для изменения, то есть отвечать за одну конкретную задачу. Это упрощает сопровождение и тестирование.
- **O — Open/Closed Principle (Принцип открытости/закрытости)**: программные сущности (классы, модули) должны быть открыты для расширения, но закрыты для модификации. Это означает, что можно добавлять новую функциональность, не меняя существующий код, что снижает риск регрессий.
- **L — Liskov Substitution Principle (Принцип подстановки Барбары Лисков)**: объекты подкласса должны быть взаимозаменяемы с объектами базового класса без нарушения корректности программы. Гарантирует, что наследование не ломает логику.
- **I — Interface Segregation Principle (Принцип разделения интерфейса)**: клиенты не должны зависеть от интерфейсов, которые они не используют. Лучше сделать несколько специфичных интерфейсов, чем один универсальный и громоздкий.
- **D — Dependency Inversion Principle (Принцип инверсии зависимостей)**: модули верхнего уровня не должны зависеть от модулей низкого уровня напрямую, оба должны зависеть от абстракций. Это улучшает декуплинг и облегчает тестирование через mock-объекты.

#### Практический контекст
В крупных проектах с микросервисной архитектурой и сложной бизнес-логикой соблюдение SOLID позволяет делать код более модульным: новые требования внедряются без полного переписывания, а покрытие юнит-тестами становится проще. Например, в React 18 проработка компонентов и хук-композиций часто следует принципу единственной ответственности, а внедрение зависимостей через Context API или Injection помогает применять Dependency Inversion.
