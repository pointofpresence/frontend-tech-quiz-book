# Чем отличается PUT от PATCH?

### Категории

HTTP, Behavioral, API, HTTP methods, Methodologies, Methods, Architecture, Integration, Put, Onboarding, Patch, Testing

### Краткий ответ

**Чем отличается PUT от PATCH?**

- HTTP методы для обновления ресурсов (REST)
- **PUT**: полное замещение ресурса  
- требует отправить **весь** ресурс целиком  
- идемпотентен: повтор PUT не меняет результат
- **PATCH**: частичное обновление ресурса  
- отправляет только изменённые поля  
- не обязательно идемпотентен (зависит от реализации)
- PUT подходит для замены, PATCH — для частичных изменений  
- использование PATCH уменьшает трафик и повышает эффективность  

Итог: PUT — полное обновление, PATCH — частичное, с разной семантикой и областью применения.

### Подробный ответ
#### Основной ответ
**PUT** и **PATCH** — это два HTTP-метода для обновления ресурсов, но они различаются по семантике и области применения. **PUT** используется для полной замены ресурса, то есть клиент отправляет полное представление объекта, и сервер заменяет им существующий ресурс. **PATCH**, наоборот, предназначен для частичного обновления — клиент отправляет только те поля, которые нужно изменить, а остальные остаются без изменений.

#### Ключевые моменты
- **PUT** — идемпотентный метод: повторный вызов с одним и тем же телом всегда дает одинаковый результат, так как происходит полная замена ресурса.
- **PATCH** — может быть неидемпотентным, зависит от конкретного патча (например, JSON Patch или Merge Patch), и обычно используется, чтобы минимизировать трафик и риск случайного перезаписывания данных.
- В REST API **PUT** часто требует отправлять весь объект, включая поля без изменений, а **PATCH** — только изменённые поля, что делает PATCH более удобным для частичных обновлений.

#### Практический контекст
В современных системах сервисами и API, например на основе OpenAPI или GraphQL, **PATCH** часто применяется для оптимизации обновлений и уменьшения объёма передаваемых данных. **PUT** же полезен там, где важно четко контролировать целостность объекта и его полное состояние. К примеру, в React 18 с REST API при редактировании пользовательских профилей логично использовать PATCH, чтобы изменить только одно поле — email или имя, не пересылая весь профиль.
