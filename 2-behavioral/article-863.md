# Как обрабатывать ошибки в RESTful API (форматы ответов, коды статусов)?

### Категории

HTTP status codes, Behavioral, API, Error handling, Backend, Database, REST

### Краткий ответ

**Обработка ошибок в RESTful API (форматы и коды статусов)**  
- HTTP коды — основа передачи статуса ошибки (4xx, 5xx)  
- 4xx — ошибки клиента (например, 400 Bad Request, 401 Unauthorized, 404 Not Found)  
- 5xx — ошибки сервера (например, 500 Internal Server Error)  
- Формат ответа — обычно JSON с полями: code, message, details (если нужно)  
- Сообщение должно быть читабельным и конкретным для отладки  
- Использование стандарта RFC 7807 (Problem Details) для единого формата  
- Логирование ошибок для мониторинга и поддержки  
- Пример:  
  ```json  
  {  
    "status": 404,  
    "error": "Not Found",  
    "message": "Пользователь не найден",  
    "path": "/api/users/123"  
  }  
  ```  
- Цель — понятное, стандартизированное информирование клиента о причине сбоя  
- При необходимости возвращать заголовки Retry-After, WWW-Authenticate и т.п.  
- Для валидации часто 422 Unprocessable Entity с деталями ошибок полей  
- Обрабатывать неожиданные ошибки через универсальный middleware или фильтр  

Этот подход повышает удобство использования API, упрощает интеграцию и ускоряет устранение проблем.

### Подробный ответ
#### Основной ответ
Обработка ошибок в RESTful API — ключевой аспект для обеспечения предсказуемого и понятного взаимодействия с клиентом. Правильное использование HTTP статусов и четких форматов ответов помогает быстро диагностировать проблему и адекватно реагировать на нее. Обычно при ошибках возвращают статус-код из диапазона 4xx (ошибки клиента) или 5xx (ошибки сервера) и тело с описанием ошибки в стандартизированном формате, например JSON.

#### Ключевые моменты
- **Коды статусов:**  
  - 400 Bad Request — некорректный запрос (валидация, неверный синтаксис).  
  - 401 Unauthorized — отсутствие авторизации или токен невалиден.  
  - 403 Forbidden — пользователь не имеет прав на действие.  
  - 404 Not Found — запрашиваемый ресурс не найден.  
  - 409 Conflict — конфликт, например, дублирование уникального поля.  
  - 422 Unprocessable Entity — семантические ошибки в данных (например, невалидные поля).  
  - 500 Internal Server Error — неожиданные ошибки на сервере.
- **Формат ответа:** Стандартный формат ответа на ошибку обычно включает:  
  - поле `error` или `message` с читабельным описанием ошибки;  
  - поле `code` — внутренний код ошибки для удобства клиента (опционально);  
  - может включать `details` с дополнительной информацией, например, список ошибок валидации.  
  Часто используют JSON-формат с четкой структурой, соответствующей OpenAPI или RFC7807 (Problem Details).
- **Единообразие и детализация:** Очень важно поддерживать единообразие во всех эндпоинтах API. Ошибки должны быть понятными, не раскрывать лишнюю техническую информацию (избегать утечек), но при этом быть информативными для разработчиков.

#### Практический контекст
В проектах на Spring Boot обычно используют @ControllerAdvice с кастомными классами ошибок, которые возвращают ResponseEntity с нужным статусом и телом. В Node.js с Express — middleware для централизованной обработки ошибок, возвращающий стандартизированный JSON. Это упрощает поддержку, интеграцию с фронтендом и автоматизацию тестирования API, а также улучшает UX клиентов API.
