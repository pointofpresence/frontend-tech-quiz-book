# Как обрабатываются ошибки и паники в Go?

**Как обрабатываются ошибки и паники в Go?**  
- ошибки — возвращаются как **значения** (error интерфейс)  
- ручная проверка после вызовов функций (**if err != nil**)  
- паника — при **критических ошибках** (например, выход за пределы среза)  
- паника вызывает **откат** стека вызовов (stack unwinding)  
- восстановление через **defer + recover()** для обработки паники  
- defer выполняются в порядке **LIFO** перед возвратом из функции  
- практика: ошибки для контролируемых ситуаций, паника — для неожиданных багов  

Детально: Go лозунг — ошибки не исключения, а значения для явной обработки, panic/recover — крайние средства для устойчивости программы.

# Подробный ответ
## Основной ответ
В **Go** ошибки и паники обрабатываются различными способами, чтобы обеспечить надежность и управление неожиданными ситуациями. Основной механизм для обычных ошибок — это возвращаемое значение типа **error**, которое вызывающий код обязан проверить и обработать. Для критических сбоев, которые нельзя или неудобно обрабатывать обычными ошибками, используется механизм **panic/recover**.

## Ключевые моменты
- **Обработка ошибок через error**: В Go принято явно возвращать ошибку как второй или последний результат функции. Такой подход позволяет явно контролировать ошибки без использования исключений. Пример: `value, err := doSomething()`. Ошибки обрабатывают через условные конструкции `if err != nil`.
- **Паника (panic)**: Используется для неожиданных или фатальных ошибок, например, нарушение инвариантов или неустранимые сбои. Вызов `panic()` прерывает нормальное выполнение и начинает unwind stack.
- **Восстановление (recover)**: Вызов внутри отложенной функции `defer` позволяет «поймать» панику и восстановить управление, предотвращая падение всей программы. Это полезно для изоляции критических компонентов и обеспечения устойчивости сервиса.
- **Разделение ответственности**: Использовать `panic` только в действительно непредвиденных сценариях или когда ошибка свидетельствует о программной ошибке. В бизнес-логике ошибка должна возвращаться как value.
- **Примеры и best practices**: В стандартной библиотеке Go, например в `net/http`, используется `recover` для предотвращения краха сервера от одной плохой обработки запроса, при этом логируя информацию об ошибке.

## Практический контекст
В реальных проектах подход с возвращаемыми ошибками обеспечивает прозрачную логику обработки ошибок и простоту тестирования. Механизм `panic/recover` часто применяется в высоконагруженных сервисах (например, в gRPC-серверах или HTTP middleware), чтобы избежать полного краха из-за форс-мажорных ситуаций и передать информацию об ошибке в лог или метрики (например, Prometheus).
