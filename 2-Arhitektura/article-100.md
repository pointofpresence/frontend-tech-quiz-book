# Почему Hashtable считается устаревшим в Java?

Категория: Архитектура

### Краткий ответ

**Почему Hashtable считается устаревшим в Java?**

- Hashtable — **синхронизированная** реализация словаря  
- Использует **coarse-grained** блокировки на весь объект  
- **Низкая производительность** при конкурентном доступе  
- Заменён **ConcurrentHashMap** с более эффективной блокировкой  
- Современный код предпочитает **несинхронизированные** Map + внешняя синхронизация  
- Hashtable не поддерживает **null** ключи и значения, в отличие от HashMap  
- Рекомендуется использовать **HashMap** или **ConcurrentHashMap** для гибкости и производительности

### Подробный ответ
#### Основной ответ
**Hashtable** в Java считается устаревшим, поскольку он был введён ещё в Java 1.0 и обладает рядом ограничений, которые стали критичными с появлением современных коллекций, таких как **HashMap** и **ConcurrentHashMap**. Основная причина — устаревшая модель синхронизации и менее гибкий API. В современных проектах обычно используют **HashMap** для несинхронного доступа и **ConcurrentHashMap** для конкурентного, что более эффективно и удобно.

#### Ключевые моменты
- **Полная синхронизация всех методов**: все методы Hashtable объявлены как synchronized, что приводит к чрезмерной блокировке и ухудшению производительности в многопоточных средах. В то время, как **ConcurrentHashMap** применяет более изощрённые механизмы блокировки, позволяя большей степени параллелизма.
- **Отсутствие поддержки null ключей и значений**: Hashtable не допускает null, а HashMap позволяет хранить один null-ключ и несколько null-значений, что важнее для гибкости.
- **Наследование от устаревшего класса Dictionary**: Hashtable наследуется от Dictionary, который тоже считается устаревшим интерфейсом, тогда как современные коллекции реализуют интерфейс Map.
- **Менее гибкий и функциональный API**: HashMap и ConcurrentHashMap имеют улучшенный интерфейс, поддерживающий современные методы из Java 8+, например для потоковой обработки данных.

#### Практический контекст
Если нужна коллекция для однопоточных сценариев, предпочтительна **HashMap** из-за отсутствия синхронизации и, соответственно, лучшей производительности. Для многопоточных приложений лучше применять **ConcurrentHashMap**, обеспечивающий высокую пропускную способность без полной блокировки, что критично в высоконагруженных системах (например, в серверах с большим количеством параллельных запросов).  
Hashtable стоит рассматривать только при поддержке наследия старого кода, но для новых проектов избегать.
