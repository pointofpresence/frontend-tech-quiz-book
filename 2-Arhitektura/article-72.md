# Опиши проектирование БД для системы расписания (сущности: уроки, записи, связи, обработка замен занятий)

Категория: Архитектура

### Краткий ответ

**Проектирование БД для системы расписания**

- модель предметной области: уроки, записи, учителя, классы, аудитории  
- урок — базовая сущность с временем, предметом, учителем, аудиторией  
- запись — конкретное занятие по уроку с датой, статусом (запланировано, проведено, отменено)  
- связи через внешние ключи (учитель→урок, урок→класс, урок→аудитория)  
- обработка замен через отдельную таблицу замен с указанием причины, нового учителя/аудитории и даты  
- поддержка истории изменений (лог замен) для аудита  
- индексация по дате, классу и учителю для быстрого поиска и обновления  
- применение транзакций для согласованности при замене занятия и обновлении расписания

Итог: реляционная модель с четкой нормализацией, гибкой логикой обработок замен и высокой производительностью запросов.

### Подробный ответ
#### Основной ответ  
Проектирование базы данных для системы расписания — это задача, которая требует внимательного моделирования сущностей и их связей, чтобы обеспечить гибкость и простоту обработки замен занятий. Основные сущности — **Уроки**, **Записи (занятия в расписании)**, **Связи (учителя, классы, аудитории)** и механизм обработки **замен**.

Стандартный подход — использование реляционной СУБД, например PostgreSQL 14+. Основные таблицы:  
- **Lessons (Уроки)** — описывает предмет, продолжительность, тип (лекция, практ.)  
- **Schedules (Записи)** — конкретные занятия: привязка урока, время, аудитория, класс и преподаватель  
- **Teachers, Classes, Rooms** — справочники для структурирования связей  
- **Substitutions (Замены)** — отдельная таблица, которая содержит замену конкретной записи расписания (ссылку на исходный урок, заменителя и время)

Обработка замен реализуется по принципу наложения: замена создаёт отдельный запись, которая имеет ссылку на оригинальную и при выборке данных отображается вместо неё. Такой подход поддерживает аудит и легко интегрируется с бизнес-логикой.

#### Ключевые моменты  
- **Нормализация**: разделение на справочники и отдельные таблицы для уроков, аудиторий, классов и преподавателей снижает дублирование и улучшает поддержку данных  
- **Связи**: многие-ко-многим между классами, уроками и преподавателями решается через промежуточные таблицы (например, ScheduleLessons или Assignment)  
- **Обработка замен**: выделение отдельной таблицы с ссылкой на исходную запись позволяет сохранять историю и минимальным вмешательством отображать актуальное расписание  
- **Индексы** и **констрейнты** на время и пересечения записей помогут избежать конфликтов в расписании и ускорить выборку  
- В продвинутых решениях можно добавить логику повторяющихся событий (Recurring events) и поддержки календаря

#### Практический контекст  
В реальных проектах, например в учебных системах (LMS), часто используют именно такой подход для максимальной прозрачности и гибкости. При этом фронтенд запрашивает расписание на день, где замены уже применены на уровне SQL-запроса или бизнес-логики, обеспечивая пользователю актуальный статус без сложных вычислений. Для производительности нередко применяют кэширование в Redis и для изменений расписания — event-driven архитектуру с RabbitMQ или Kafka.
