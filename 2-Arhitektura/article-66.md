# Как работает Jest под капотом (разница между jsdom и браузером)?

**Как работает Jest под капотом (jsdom vs браузер)**  
- Jest — тест-раннер для JavaScript с поддержкой моков и изоляции  
- Для тестов DOM использует **jsdom** — эмулятор браузерного окружения на Node.js  
- jsdom парсит HTML/CSS, имитирует DOM API без реального рендеринга  
- В отличии от браузера с движком (Blink, Gecko), jsdom не рендерит визуально и не выполняет CSS полностью  
- jsdom быстрее и легковеснее, подходит для unit-тестов компонентов с доступом к DOM  
- Браузерное окружение даёт полный API и поведение реального рендера, важно для e2e или интеграционных тестов  
- Jest может интегрироваться с Puppeteer/Playwright для браузерных тестов с реальным рендером  
- Использование jsdom снижает зависимость от окружения, улучшает скорость CI/CD и упрощает настройку  

Итог: Jest через jsdom имитирует браузер для быстрого и изолированного тестирования, но не воспроизводит всё поведение браузера с рендерингом.

# Подробный ответ
## Основной ответ
**Jest** — это популярный JavaScript тест-раннер, который из коробки предоставляет удобную среду для unit и integration тестов. Под капотом он запускает тесты в Node.js, а для эмуляции браузерного окружения использует **jsdom** — легковесный JavaScript-реализация DOM API. В отличие от настоящего браузера, jsdom лишь симулирует DOM и часть браузерного поведения, позволяя тестам запускаться быстро и изолированно без запуска реального браузера.

## Ключевые моменты
- **jsdom** реализует браузерный DOM в Node.js, но без полноценного рендеринга, CSS, или сложного JS движка — это просто набор API для работы с элементами, событиями и навигацией.
- В отличие от **настоящего браузера** (например, Chrome), jsdom не поддерживает многие современные веб-API (Canvas, WebGL, WebRTC, сложные события) и не выполняет реального layout/rendering, что влияет на некоторые UI-тесты.
- Jest упрощает настройку, предоставляя виртуальную среду `jsdom` по умолчанию. Для тестирования в настоящем браузере используют инструменты как **Puppeteer** или **Playwright**, если нужна максимальная Fidelity.

## Практический контекст
jsdom хорошо подходит для unit-тестов React-компонентов, где важна работа с DOM-структурой, но не критичен визуальный рендеринг. Он позволяет быстро запускать сотни тестов с низкой задержкой (~10-50ms). Если проект требует тестов, завязанных на реальное поведение браузера (например, сложные CSS-анимации или Web API), стоит интегрировать браузерные среды через end-to-end фреймворки.
