# Как реализовать worker pool с ограничением по concurrency в Go?

**Как реализовать worker pool с ограничением по concurrency в Go?**

- concurrency: управление количеством **одновременных goroutine**
- создаём **канал-буфер** (buffered channel) с размером равным concurrency
- перед запуском worker'а **отправляем "токен"** в канал (если заполнен, ждём)
- по окончании работы worker'а **забираем токен** из канала, освобождая слот
- запускаем фиксированное количество worker'ов, которые берут задачи из очереди (канала)
- канал задач обеспечивает **управление распределением** работы между worker'ами
- применимо для контроля нагрузки и предотвращения исчерпания ресурсов (CPU, память)

**Суть:** канал токенов реализует лимит одновременно работающих горутин, контролируя concurrency и защищая систему от перегрузки.

# Подробный ответ
## Основной ответ
**Worker pool с ограничением по concurrency в Go** реализуется путем создания фиксированного количества воркеров (goroutine), которые получают задачи из общей очереди (канала). Основная идея — контролировать число одновременно активных воркеров, что позволяет эффективно использовать ресурсы и не перегружать систему.

## Ключевые моменты
- Для организации очереди задач обычно используют **буферизированный или небуферизированный канал**: продюсер отправляет задачи в канал, а воркеры извлекают их для обработки.
- Количество воркеров ограничивается количеством запущенных горутин, например, создаём ровно N воркеров в цикле, где N — максимальный уровень параллелизма.
- Для контроля завершения работы и ожидания всех воркеров используется **sync.WaitGroup** — это гарантирует, что основная горутина дождётся выполнения всех задач.
- Важно учитывать **обработку ошибок и отмену**: часто добавляют контекст (context.Context) для возможности graceful shutdown или таймаутов.
- Такой подход используют в продакшене для ограничений на внешние сервисы, CPU-intensive задачи и т.д.

## Практический контекст
В реальных проектах (например, в Go 1.18+) worker pool помогают реализовать ограничение запросов к API, параллельную обработку данных или фоновые задачи. Пример: сбор метрик с множества источников, где concurrency ставится по лимитам внешних систем.

---

**Кодовый пример для иллюстрации:**
```go
type Task func()

func worker(id int, tasks <-chan Task, wg *sync.WaitGroup) {
    defer wg.Done()
    for task := range tasks {
        task() // Выполнение задачи
    }
}

func main() {
    const maxWorkers = 5
    tasks := make(chan Task)
    var wg sync.WaitGroup

    for i := 0; i < maxWorkers; i++ {
        wg.Add(1)
        go worker(i, tasks, &wg)
    }

    for i := 0; i < 20; i++ {
        taskNum := i
        tasks <- func() {
            fmt.Printf("Выполняется задача %d\n", taskNum)
            time.Sleep(time.Second)
        }
    }
    close(tasks)
    wg.Wait()
}
```

Этот пример создаёт 5 воркеров, которые последовательно выполняют 20 задач, не превышая concurrency 5.
