# Как протестировать сервис, который обращается к базе и внешним системам?

### Категории

Live coding, System design, Backend, Integration, External systems, Database, Service testing, API, Testing

### Краткий ответ

**Как тестировать сервис с БД и внешними системами?**

- **Типы тестов:** юнит, интеграционные, e2e
- **Мокауты/стабы** для внешних API и БД-запросов на юните
- Интеграционные тесты с реальной или тестовой БД и внешними системами
- Использование изолированного тестового окружения (in-memory БД, тестовые сервисы)
- Тесты на обработку ошибок и таймаутов внешних систем
- Автоматизация с CI/CD для стабильности и повторяемости
- Валидация бизнес-логики отдельно от инфраструктуры

### Подробный ответ
#### Основной ответ
Для тестирования сервиса, который взаимодействует с базой данных и внешними системами, используют комбинацию уровней тестирования и техник изоляции. В первую очередь важно отделить логику сервиса от внешних зависимостей, чтобы обеспечить быстрые, стабильные и воспроизводимые тесты. Обычно применяют **юнит-тесты** с моками (mock) для имитации БД и внешних API, **интеграционные тесты** с настоящей базой (например, отдельным тестовым инстансом) и **энд-то-энд тесты** для проверки полного сценария работы.

#### Ключевые моменты
- **Юнит-тесты с моками**: позволяют изолировать бизнес-логику, имитируя ответы от базы и внешних систем. Особенно удобно использовать библиотеки типа Mockito (Java), unittest.mock (Python). Поддерживают быстрый feedback и catch ошибок в логике.
- **Интеграционные тесты**: запускают сервис с настоящей тестовой базой (PostgreSQL в Docker) и, если возможно, с тестовыми эмуляторами внешних API (например, WireMock). Это проверяет корректность работы с реальными интерфейсами и схемами данных.
- **Контроль за окружением**: для внешних сервисов — использовать sandbox / staging окружения, или тестовые версии API с изолированными данными. Для БД — применять миграции и проливать тестовые данные, чтобы обеспечить консистентность.
- **CI/CD и тестовые сроки**: балансировать между обширностью интеграционных тестов и временем их выполнения, чтобы не замедлять деплой.

#### Практический контекст
В сложных сервисах часто сочетают подходы: например, для логики с высокой бизнес-комплексностью — юнит-тесты с моками, а критичные пути с БД и внешними системами покрывают интеграционными или контрактными тестами. Используют Prometheus + Grafana для мониторинга в реальном времени. При разработке микросервисов типично применяют consumer-driven контрактное тестирование (Pact) для внешних API, что снижает риск регрессий в интеграциях.
