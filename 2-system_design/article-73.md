# Что такое FSD-архитектура? Какие слои использовали?

### Категории

Layers, Templates, Async, Coding, Frontend, Live coding, System design, TypeScript, Promises, Architecture

### Краткий ответ

**Что такое FSD-архитектура? Какие слои использовали?**  
- FSD (Feature-Sliced Design) — архитектурный паттерн фронтенда  
- Делит приложение на **фичи**, выделяет их слои по ответственности  
- Основные слои:  
  - **App** — инициализация и маршрутизация  
  - **Process** — сложная логика, бизнес-процессы  
  - **Feature** — конкретный функционал (ui+логика)  
  - **Entity** — сущности данных, типы  
  - **Shared** — общие утилиты, компоненты и стили  
- Повышает **масштабируемость** и поддерживаемость  
- Упрощает навигацию и разделение командной работы  
- Применяется в сложных SPA для модульности и изоляции кода

### Подробный ответ
#### Основной ответ
**FSD-архитектура (Feature-Sliced Design)** — это подход к организации фронтенд-кода, ориентированный на разбивку приложения по функциональным срезам (фичам) вместо технической структуры (например, папки по типу — компоненты, сервисы и т.д.). Это помогает поддерживать масштабируемость, читаемость и изоляцию кода в больших проектах.

#### Ключевые слои в FSD  
В классической FSD-архитектуре выделяют несколько слоев, каждый из которых выполняет строго определённую роль:

- **App** — корневой слой, где инициализируются глобальные провайдеры, роутинг и основные настройки приложения.
- **Pages** — слой страниц приложения, каждую страницу можно считать самостоятельной фичей, которая объединяет компоненты, логику и стили, необходимые для конкретного экрана.
- **Features** — функциональные возможности, которые служат для организации бизнес-логики, например, авторизация, работа с корзиной, поиск.
- **Entities** — доменные объекты и модели, отражают реальные бизнес-сущности (пользователь, продукт), содержат типы, модели и бизнес-логику.
- **Shared** — переиспользуемые абстракции и утилиты, которые не зависят от конкретных фич, например, UI-библиотеки, утилитарные функции, константы.

#### Ключевые моменты
- **Изоляция и инкапсуляция фич**: каждый функциональный срез может разрабатываться и поддерживаться независимо.
- **Ясная градация ответственности**: слои отделяют абстракции разного уровня — от специфики UI до бизнес-логики и общих утилит.
- **Удобство масштабирования**: при росте приложения проще добавлять новые фичи, не ломая существующую структуру.

#### Практический контекст
FSD часто применяют в больших React-приложениях (в React 18+) с TypeScript, где архитектура помогает командам работать параллельно и сокращает количество конфликтов при слиянии кода. Она особенно полезна, когда нужно поддерживать сотни компонентов и сложные бизнес-процессы в одном репозитории.
