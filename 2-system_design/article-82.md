# Что такое паттерн Strategy и зачем он нужен?

### Категории

SQL, Database, Kafka, Strategy pattern, HTTP, Design patterns, System design, Concurrency, Microservices, Outbox, Architecture

### Краткий ответ

**Что такое паттерн Strategy и зачем он нужен?**

- поведенческий паттерн проектирования  
- позволяет менять алгоритмы **динамически**  
- инкапсулирует разные стратегии в отдельные классы  
- клиент работает с ними через общий интерфейс  
- облегчает расширение и замену алгоритмов без изменения клиента  
- уменьшает количество условных операторов (if/switch)  
- применяется для повышения **гибкости** и поддержки Open/Closed принципа

### Подробный ответ
#### Основной ответ
**Паттерн Strategy** — это поведенческий паттерн проектирования, который позволяет определить семейство алгоритмов, инкапсулировать каждый из них и делать их взаимозаменяемыми. Он позволяет менять поведение объекта во время выполнения, отделяя алгоритмы от клиента, который их использует. Это упрощает расширяемость и улучшает поддержку кода, потому что новые стратегии можно добавлять без изменения существующего кода.

#### Ключевые моменты
- **Инкапсуляция алгоритмов**: каждая реализация стратегии оформляется в отдельный класс с единым интерфейсом, что облегчает тестирование и модификацию.
- **Взаимозаменяемость**: объект использует ссылку на интерфейс стратегии, что позволяет легко менять алгоритм без изменения кода клиента.
- **Улучшение поддержки Open/Closed Principle**: добавление новых алгоритмов не требует модификации существующих классов, только расширения через добавление новых реализаций.

#### Практический контекст
Часто используется в случае, когда разные способы выполнения задачи должны динамически применяться в зависимости от окружения, пользовательского выбора или состояния. Например, в платежных системах можно реализовать разные стратегии оплаты (кредитная карта, PayPal, криптовалюта) — каждая инкапсулирована и подключается по необходимости. В React 18+ паттерн Strategy можно применять для управления поведением компонентов, например, выбор разных алгоритмов виртуального рендеринга в зависимости от нагрузки.
