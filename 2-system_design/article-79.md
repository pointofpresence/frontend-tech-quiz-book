# Разница между POST, PUT и PATCH?

### Категории

HTTP, System design, API, Methods, Backend

### Краткий ответ

**Разница между POST, PUT и PATCH**  
- HTTP методы для управления ресурсами (REST)  
- **POST:** создание нового ресурса, неидемпотентный  
- **PUT:** полное обновление ресурса, идемпотентный  
- **PATCH:** частичное обновление ресурса, идемпотентный или не всегда  
- POST не требует полного тела, PUT заменяет весь ресурс  
- PATCH меняет только указанные поля, легче по трафику  
- Выбор зависит от задачи: создать, заменить или обновить частично  
- Правильное использование повышает семантику API и стабильность

### Подробный ответ
#### Основной ответ
**POST, PUT и PATCH** — это методы HTTP, используемые для отправки данных на сервер, но они имеют разные семантические значения и предназначены для различных сценариев обновления ресурсов.  
- **POST** обычно создаёт новый ресурс или выполняет действие, не обязательно идемпотентное.  
- **PUT** служит для полного обновления существующего ресурса или создания его по определённому URI, при этом операция идемпотентна.  
- **PATCH** применяется для частичного обновления ресурса, меняет только указанные поля, тоже обычно идемпотентен, но зависит от реализации.

#### Ключевые моменты
- **POST** используется, когда клиент отправляет данные для создания нового объекта, например, отправка формы или создание заявки. Результат неоднозначен: повторные запросы могут привести к созданию нескольких копий.  
- **PUT** заменяет весь целевой ресурс новым представлением: если ресурс существует, происходит полное замещение; если нет — создаётся новый по этому URI. Это идемпотентная операция, повторный запрос не изменит ситуацию после первого.  
- **PATCH** оптимален для ситуаций, когда нужно обновить лишь некоторые поля без замены всего объекта. Стандарты (RFC 5789) предполагают передачу списка изменений (например, в формате JSON Patch). PATCH может быть сложнее реализовать, но снижает нагрузку на передачу и обработку данных.

#### Практический контекст
В RESTful API в React 18 и современных backend-фреймворках (например, FastAPI, Spring Boot) обычно:  
- Для создания новых сущностей используют **POST**, например, `/api/users` с телом нового пользователя.  
- Для полной замены ресурса применяют **PUT**, например, `/api/users/123` с полностью новым профилем.  
- Для частичного обновления данных, например, изменить только email, без пересылки всего профиля — используют **PATCH** на тот же URI.  

Понимание этих различий помогает создавать API с правильной семантикой и улучшает прогнозируемость работы клиентов и серверов.
