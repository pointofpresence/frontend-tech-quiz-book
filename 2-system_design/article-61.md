# Как управлять версионностью зависимостей (например, React) между микрофронтами?

### Категории

System design, Microfrontends, Versioning, Live coding, Dependency management, Frontend architecture

### Краткий ответ

**Как управлять версионностью зависимостей между микрофронтами?**  
архитектура микрофронтов, зависимости  

- **Управление независимое**: каждая микрофронт-часть содержит свою версию React  
- Использование **peerDependencies** для согласования версий в npm  
- **Singleton pattern** в module federation (Webpack 5) обеспечивает единый React в рантайме  
- Контроль версий через общий **пакет-билд или shared library**  
- Версионный конфликт решается через **aliasing и резолвинг модулей** в сборщике  
- Автоматизация через **CI/CD и lock-файлы** (yarn.lock, package-lock.json)  
- Практика: тщательно контролировать и согласовывать версии React при релизах  
- Почему важно: снижает ошибки, уменьшает размер бандла, гарантирует стабильность UI  

Этот подход гарантирует согласованность и минимизирует конфликты версий React в разных микрофронтах, что критично для поддержки единой UX.

### Подробный ответ
#### Основной ответ
Управление версионностью зависимостей между **микрофронтами** — это одна из ключевых задач для обеспечения совместимости и стабильности системы, так как каждый микрофронт может использовать свою версию библиотеки, например **React**. Основная цель — избежать конфликтов и дублирования, сохранить изоляцию и при этом обеспечить максимально возможное переиспользование общих библиотек.

#### Ключевые моменты
- **Singleton-подход и shared dependencies:** В системах на **Module Federation (Webpack 5)** обычно объявляют React как `singleton` и `shared` dependency с заданной минимальной версией. Это позволяет всем микрофронтам автоматически использовать одну и ту же версию React при загрузке, управляя версиями через semver rules. Подход снижает размер бандлов и предотвращает появление одновременно двух версий React на странице, что крайне важно из-за внутреннего состояния React.
- **Версионное выравнивание и согласованность:** На уровне команды и репозиториев желательно поддерживать согласованность версий ключевых библиотек, например через monorepo с **Yarn Workspaces** или **PNPM** с фиксированными зависимостями, либо с помощью **Renovate/bot** для автоматического обновления package.json. Это минимизирует несовместимости и ошибки рантайма.
- **Изоляция через iframes или shadow DOM:** Для крайнего случая, когда разные версии реально конфликтуют (например, major-версии React), можно использовать полную изоляцию микрофронтов через **iframe** или **shadow DOM**. Это снижает влияние версий друг на друга, но усложняет коммуникацию между ними.

#### Практический контекст
В реальных проектах часто комбинируют **Webpack 5 Module Federation** с четким договором о версиях, что позволяет всем микрофронтам использовать React 18+ версии без дублирования. При более сложных сценариях выбирают изоляцию через shadow DOM или iframe. В крупных компаниях отдельно выделяют сервисы обновления общих библиотек и контролируют обновления через CI/CD, чтобы не сломать интеграцию.
