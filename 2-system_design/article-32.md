# Как реализовать параллельную обработку с ограничением количества горутин?

### Категории

Goroutines, System design, Concurrency, Performance, Backend, Database, Scalability, Optimization, Parallelism

### Краткий ответ

**Параллельная обработка с лимитом горутин**

- контекст: **конкурентность в Go**
- проблема: ограничить число одновременно работающих **горутин** для контроля ресурсов
- решение: использовать **буферизированный канал** как семафор
- принцип: перед запуском горутины отправлять "токен" в канал, после завершения — вытягивать, блокируя лишние горутины
- альтернатива: **worker pool** — фиксированное количество воркеров читает задачи из канала
- пример:
  ```go
  sem := make(chan struct{}, limit)
  for _, task := range tasks {
      sem <- struct{}{}
      go func(t Task) {
          defer func() { <-sem }()
          process(t)
      }(task)
  }
  ```
- ценность: контролирует **параллелизм**, предотвращает **перегрузку**, управляя памятью и CPU

Такое решение устойчиво и широко используется для балансировки нагрузки в Go-проектах.

### Подробный ответ
#### Основной ответ
Для реализации параллельной обработки с ограничением количества **горутин** в Go часто используют паттерн **пул воркеров** (worker pool). Идея в том, что создаётся фиксированное число горутин, которые параллельно обрабатывают задачи из общей очереди (канала), при этом строго контролируется максимальное количество одновременно работающих горутин. Это позволяет избежать чрезмерного потребления ресурсов и перегрузки системы.

#### Ключевые моменты
- Используйте **буферизированный канал** для очереди задач, в который основная горутина отправляет задачи, а фиксированное число воркеров их читает и выполняет.
- Контролируйте количество воркеров с помощью **запуска строго N горутин**, где N — желаемый максимум параллелизма (например, `runtime.NumCPU()` или заданное число).
- Для завершения работы используйте **sync.WaitGroup**, чтобы дождаться окончания обработки всех задач, корректно закрывая воркеры и канал.
- Альтернативно можно использовать семафор на базе канала (например, с capacity = N) для динамического ограничения количества одновременно запускаемых горутин, если горутины запускаются ad-hoc.

#### Практический контекст
В реальных сервисах с высокой нагрузкой такой подход помогает контролировать concurrency, сохраняя стабильность и предсказуемость. Например, при обращениях к базе данных или внешним API, где слишком большой параллелизм приведет к таймаутам или ограничению ресурсов. В Go 1.18+ можно дополнительно использовать контексты (context.Context) для отмены и таймаутов при обработке задач.

Пример кода простого пула воркеров:

```go
tasks := make(chan Task, 100)
var wg sync.WaitGroup

workerCount := 5
for i := 0; i < workerCount; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        for task := range tasks {
            task.Process()
        }
    }()
}

// Добавляем задачи
for _, t := range taskList {
    tasks <- t
}
close(tasks)
wg.Wait()
```

Такой паттерн гарантирует, что одновременно работает ровно 5 горутин, а остальные задачи ждут своей очереди.
