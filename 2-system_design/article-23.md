# Как в Go реализовать graceful shutdown для сервиса, у которого есть HTTP-сервер и фоновые воркеры?

### Категории

System design, High load, HTTP server, Concurrency, Message brokers, Network, Graceful shutdown, Kubernetes, Transactions, Background workers, DevOps

### Краткий ответ

**Как в Go реализовать graceful shutdown для HTTP-сервиса с воркерами?**  
- Контекст: управление завершением сервиса, работающего с HTTP и воркерами  
- Использовать встроенный контекст **context.Context** для координации завершения  
- Запускать HTTP-сервер через **http.Server** с методом **Shutdown(ctx)** для плавного закрытия соединений  
- Реализовать фоновым воркерам слушание сигнала из контекста для планового завершения  
- Использовать **sync.WaitGroup** для ожидания завершения всех горутин воркеров  
- Перехватывать сигналы ОС (**SIGINT, SIGTERM**) с помощью **os/signal** для триггера shutdown  
- Итог: гарантировать закрытие HTTP и завершение фоновых задач без потери данных и новых запросов  

---

**Примерный алгоритм:**  
1. Слушать сигналы ОС в горутине  
2. При сигнале контекст отменяется  
3. HTTP сервер вызывается `Shutdown(ctx)` — не принимает новые запросы, ждёт текущие  
4. Фоновые воркеры проверяют контекст, завершают работу и вызывают Done в WaitGroup  
5. Основная горутина ждёт завершения всех воркеров через WaitGroup  
6. После этого процесс безопасно завершается  

Таким образом достигается **плавное отключение сервиса** без резких прерываний и утечки ресурсов.

### Подробный ответ
#### Основной ответ
В Go для реализации **graceful shutdown** сервиса с HTTP-сервером и фоновыми воркерами обычно используют комбинацию контекста (`context.Context`), сигналов ОС (`os.Signal`) и синхронизации через `sync.WaitGroup`. Идея в том, чтобы в момент получения сигнала завершения (например, SIGINT или SIGTERM) корректно остановить HTTP-сервер (завершить текущие запросы), а также дать фоновым воркерам время завершить свою работу.

Основные шаги:
1. Создать канал для перехвата системных сигналов.
2. Запустить HTTP-сервер в отдельной горутине, используя `http.Server` с методом `Shutdown(ctx)`, который дождётся обработки текущих запросов.
3. Фоновые воркеры выполняются в своих горутинах, а их работу контролируют через `context.Context` и `WaitGroup`, чтобы можно было корректно прервать и дождаться завершения.
4. При получении сигнала передать cancel в контекст, инициирующий остановку всех задач и вызвать `Shutdown` у HTTP-сервера.
5. Ждать завершения всех воркеров и окончательного закрытия сервиса.

#### Ключевые моменты
- **http.Server.Shutdown(ctx)** — в Go 1.8+ встроена нативная поддержка graceful shutdown HTTP-сервера, что позволяет дождаться завершения активных соединений с таймаутом.
- **context.Context** — используется для простого распространения сигнала отмены среди всех компонентов, включая фоновые воркеры, чтобы они могли остановиться корректно.
- **sync.WaitGroup** — полезен для синхронизации и ожидания завершения всех фоновых процессов перед завершением программы.
- Обработка системных сигналов через пакет `os/signal` — стандартный способ реагировать на завершение, например SIGINT (Ctrl+C) или SIGTERM.

#### Практический контекст
В реальных проектах `graceful shutdown` важен для предотвращения потери данных и неконсистентного состояния при перезапуске/выключении. Например, в микросервисах на Go с REST API и воркерами, слушающими сообщения из очередей, прием сигнала и корректное ожидание завершения позволяют избежать прерывания долгих запросов и ошибок из-за неполного выполнения фоновых задач. Например, в моём проекте на Go 1.18 с использованием `gin-gonic` и воркерами на `go-redis` такой подход дал надёжность при деплое без потерь трафика и с минимальным downtime (~1-2 сек).
