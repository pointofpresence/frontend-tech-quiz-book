# Какие существуют способы текстового документирования требований?

### Категории

Analysis, Documentation, Workflow, Requirements

### Краткий ответ

**Способы текстового документирования требований**

- структурированные спецификации (SRS) — формальный документ с описанием требований
- user stories — краткие описания функционала с ролью, действием и целью
- use cases — сценарии взаимодействия пользователя с системой, описанные шаг за шагом
- acceptance criteria — чёткие условия приёмки каждой user story или требования
- требования в виде списков — простое перечисление функциональных и нефункциональных требований
- mind maps или схемы (текстовые с описаниями) — визуализация с текстовым сопровождением
- требования в формате Gherkin — сценарии поведения системы (Given-When-Then) для BDD
- каждый способ помогает обеспечить понятность, проверяемость и согласованность требований для команды и заказчика

### Подробный ответ
#### Основной ответ
Существует несколько основных способов текстового документирования требований, которые используются для эффективной коммуникации между заказчиками и командами разработки. Основные форматы включают **use case descriptions (описание прецедентов)**, **user stories (пользовательские истории)** и **requirement specifications (спецификации требований)**. Выбор метода зависит от масштаба проекта, заинтересованных сторон и гибкости процесса разработки.

#### Ключевые моменты
- **Use cases** — подробные сценарии взаимодействия пользователей с системой, описывающие последовательные шаги и альтернативные потоки. Подход подходит для сложных систем, где важно чётко понимать логику взаимодействия и условия.
- **User stories** — короткие, лаконичные формулировки требований с акцентом на пользу пользователю (например: "Как [роль], я хочу [цель], чтобы [причина]"). Этот формат обычно применяется в Agile-проектах, позволяя быстро и гибко адаптировать требования.
- **Specification documents** — формальные текстовые документы с детализированными требованиями, которые могут включать функциональные, нефункциональные требования, ограничения и критерии приёмки. В больших корпоративных проектах или при контрактном развитии это стандарт.
- Также стоит упомянуть **Acceptance criteria (критерии приёмки)**, которые часто сопровождают user stories и обеспечивают чёткие условия для проверки выполнения требования.
  
#### Практический контекст
В проектах с использованием Scrum или Kanban чаще применяются user stories с краткой текстовой формулировкой и дополнительными критериями приёмки, а в традиционных Waterfall-проектах — формальные требования в отдельном спецификационном документе. При интеграции с Jira зачастую создаются отдельные тикеты с user story, а к ним добавляются подробные описания и критерии в markdown формате.
